# Phase 1.5: Prompt Architecture and Liveness — Research

**Researched:** 2026-04-10
**Domain:** Zig agent runtime — prompt construction, observer/event pipeline, memory store, persona calibration
**Confidence:** HIGH (all findings derived from direct codebase reads)

---

## Summary

Phase 1.5 adds the "feels alive" layer on top of Phase 1's execution contract. It has five distinct work items: a composable prompt scaffold refactor, a real-time liveness narration engine wired to the observer event bus, a task decomposition scaffolding system, a learning loop that converts corrections and preferences into durable memory facts, and a persona calibration system driven by `SOUL.md`.

The codebase is in excellent shape for all five items. `src/agent/prompt.zig` already has a flat imperative builder that emits sections sequentially — the refactor extracts each section into a typed, independently callable function and adds a `PromptSection` tagged union or struct-of-optionals that callers populate before building. The `ObserverEvent` union in `src/observability.zig` already carries `turn_stage` and `tool_call_start` events used by `VerboseObserver` for human-readable CLI output — narration extends those events with user-facing semantic labels and a `NarrationEvent` variant routed to a new `NarrationObserver`. The memory store already has `MemoryCategory.core` with `isMutableMemoryEntry` gating edits, and `isSystemManagedMemoryKey` already treats `durable_fact/` keys as system-managed — the learning loop stores behavioral facts under exactly that key prefix. `SOUL.md` is already loaded as a first-class bootstrap document and fingerprinted into the system prompt cache — persona calibration is a runtime parse of `SOUL.md` fields into a `PersonaProfile` struct, then injected as a composable section.

**Primary recommendation:** Work all five sprints in order (1.5-01 through 1.5-05) because each item builds on infrastructure from the prior one. Prompt scaffold first (section composability is needed by narration, task decomposition, and persona). Narration second (observer event extensions needed by task planner status reporting). Task decomposition third. Learning loop fourth (durable_fact/ key design must be stable before persona calibration writes its own calibration facts). Persona last.

### Pattern 1: Composable Prompt Section Refactor (Sprint 1.5-01)

**What:** Extract every section into a named `fn buildXxxSection(...)` function. Add a `PromptSections` struct that callers populate before building — this replaces the `PromptContext.capabilities_section ?[]const u8` blob approach.

**Design for `src/agent/prompt.zig` after refactor:**

```zig

pub const PersonaSection = struct {
    warmth: enum { crisp, balanced, warm } = .balanced,
    proactivity: enum { reactive, moderate, proactive } = .moderate,
    voice_style: ?[]const u8 = null,   // e.g. "concise", "verbose", "formal"
};

pub const NarrationPolicy = struct {
    emit_tool_start: bool = true,
    emit_tool_result: bool = false,
    emit_waiting: bool = true,
    emit_plan_step: bool = true,
};

pub const ToolUsePolicy = struct {
    max_iterations: u32,
    requires_approval: bool,
    execution_mode: ExecutionMode,
};

pub const PromptSections = struct {
    persona: ?PersonaSection = null,
    narration: ?NarrationPolicy = null,
    tool_use: ?ToolUsePolicy = null,
    capabilities: ?[]const u8 = null,
    conversation_context: ?ConversationContext = null,
};

pub fn buildSystemPrompt(
    allocator: std.mem.Allocator,
    base_ctx: PromptContext,
    sections: PromptSections,
) ![]const u8
```

Recommended approach: keep `buildSystemPrompt(allocator, ctx)` signature working by having `PromptContext` carry a `sections: PromptSections = .{}` field (zero-initialized, backward compatible). ### Pattern 2: Liveness Narration Engine (Sprint 1.5-02)

**What:** A `NarrationObserver` that converts `ObserverEvent` values into user-facing progress text and delivers them through a caller-supplied frame callback. The VerboseObserver already does this for stderr CLI output — NarrationObserver is the channel-aware version.

**Existing observer event coverage** [VERIFIED: codebase read]:

```
tool_call_start   — fires before each tool execution
tool_call         — fires after each tool execution (has duration_ms, success)
turn_stage        — fires at: turn_start, memory_enrich, build_provider_messages,
                    parse_provider_response, dispatch_tools, compose_final_reply,
                    llm_first_token, llm_first_token_upper_bound, post_reply_compaction
turn_cancelled    — fires on cooperative cancellation
```

**VerboseObserver labels already defined** [VERIFIED: codebase read]:

```
"turn_start"              → "Gathering context"
"memory_enrich"           → "Retrieving memory"
"dispatch_tools"          → "Running tools"
"tool_reflection"         → "Reflecting on tool results"
"compose_final_reply"     → "Preparing final reply"
"llm_first_token"         → "Model started responding"
"llm_first_token_upper_bound" → "Waiting for model response"
```

**New events needed for narration (extend `ObserverEvent` union):**

```zig
// [ASSUMED] — recommended additions
narration_frame: struct {
    message: []const u8,      // user-facing text
    frame_type: enum { thinking, tool_start, tool_done, waiting, plan_step, error_recovery },
    tool_name: ?[]const u8 = null,
    step_index: ?u32 = null,
    step_total: ?u32 = null,
},
```

**Delivery design:**
- `NarrationObserver` wraps an existing `Observer` and a `ProgressCallback` (fn pointer + context)
- On `tool_call_start`, emits `narration_frame{.tool_start, tool_name}`
- On `turn_stage.llm_first_token_upper_bound`, emits `narration_frame{.waiting}`
- Frame callback is registered per-session; channels consume it to push progress to user

**Critical constraint:** Narration frames must NOT be injected into the LLM message history. They are side-channel events only. The `stream_callback` in `Agent` is the LLM reply stream — narration is a separate callback. [ASSUMED — verify this is the intended architecture, not flushing narration through the LLM content stream]

### Pattern 3: Task Decomposition (Sprint 1.5-03)

**What:** When the agent receives a complex multi-step request, it emits a structured plan before executing. Plan steps are tracked in a `TaskPlan` struct and per-step observer events are emitted as execution proceeds.

**Interaction with existing `ExecutionMode`** [VERIFIED: codebase read]:

```
ExecutionMode.plan   — read_only tools only, used for analysis before action
ExecutionMode.execute — all tools, used for actual work
```

Task decomposition uses `.plan` mode for the decomposition step (no mutations while planning), then switches to `.execute` for each sub-step. The mode switch is already wired into `Agent.execution_mode`.

**`task_planner.zig` design:**

```zig
// [ASSUMED] — recommended design
pub const TaskStep = struct {
    index: u32,
    description: []const u8,
    status: enum { pending, running, done, failed },
    tool_used: ?[]const u8 = null,
    result_summary: ?[]const u8 = null,
};

pub const TaskPlan = struct {
    id: u64,
    summary: []const u8,
    steps: []TaskStep,
    current_step: u32 = 0,
    started_at_ms: i64,
};
```

**Prompt instruction approach:** Task decomposition is primarily prompt-driven (the system prompt instructs the agent to emit a structured `<task_plan>` XML block before multi-step work). The runtime scaffolding in `task_planner.zig` parses that XML and tracks execution state, emitting observer events per step.

**Parser target:** The dispatcher (`src/agent/dispatcher.zig`) already parses `<tool_call>` XML tags. The task planner uses the same scanning pattern to find `<task_plan>...</task_plan>` in model responses. ### Pattern 4: Learning Loop (Sprint 1.5-04)

**What:** After each turn, scan the agent's reply and the user's follow-up message for correction signals. When found, write a durable behavioral fact to memory. On future turns, these facts are surfaced in the memory enrichment bucket (already exists in `memory_loader.zig`).

**Existing durable_fact infrastructure** [VERIFIED: codebase read]:

```zig
// src/memory/root.zig — already treats durable_fact/ as system-managed
pub fn isSystemManagedMemoryKey(key: []const u8) bool {
    return std.mem.eql(u8, key, "context_anchor_current") or
        std.mem.startsWith(u8, key, "summary_latest/") or
        std.mem.startsWith(u8, key, "durable_fact/") or
        isAppendOnlyMemoryKey(key);
}

// isMutableMemoryEntry blocks durable_fact/ from user edits:
pub fn isMutableMemoryEntry(key: []const u8, category: MemoryCategory) bool {
    if (isTombstoneKey(key) or isMarkdownLineKey(key) or isAppendOnlyMemoryKey(key)) return false;
    return switch (category) {
        .core => true,
        else => false,
    };
}
// NOTE: isEditableMemoryEntry adds isSystemManagedMemoryKey check on top —
// so durable_fact/ keys are protected from user edits even in .core category. [VERIFIED]
```

**Key naming scheme for behavioral facts:**

```
durable_fact/behavior/{hash_of_fact}   — stable key for a given behavioral preference
durable_fact/correction/{timestamp}   — correction from a specific turn
```

**`learning.zig` design:**

```zig
// [ASSUMED] — recommended design
pub const LearningSignal = enum {
    explicit_correction,   // user says "no, actually" / "that's wrong"
    explicit_preference,   // user says "always do X" / "prefer Y"
    implicit_correction,   // user immediately asks for the opposite of what agent did
};

pub const LearnedFact = struct {
    key: []const u8,
    content: []const u8,         // human-readable behavioral instruction
    signal: LearningSignal,
    turn_ts: i64,
};

pub fn detectLearningSignals(
    allocator: std.mem.Allocator,
    user_message: []const u8,
    prior_reply: []const u8,
) ![]LearningSignal

pub fn storeLearningFact(
    mem: Memory,
    fact: LearnedFact,
    session_id: ?[]const u8,
) !void
```

**Distinction from raw memory recall (per REQ-021):** Raw `autosave_*` conversation entries are retrieved by semantic similarity. Behavioral facts under `durable_fact/behavior/` are retrieved with higher priority and injected into the system prompt narration rules section — not just user message enrichment. They change how the agent behaves, not just what it knows.

**GAP-008 alignment:** The requirements include GAP-008 (memory write and repair policy). The learning loop must define: what triggers a write (correction signal detected), what prevents stale facts (overwrite-on-same-hash approach), and what retires wrong facts (`/learn forget <key>` slash command). [ASSUMED — exact correction detection heuristics need design iteration]

### Pattern 5: Persona Calibration (Sprint 1.5-05)

**What:** Parse `SOUL.md` for structured persona metadata, resolve into a `PersonaProfile` at prompt build time, and inject the profile as a composable persona section. A `/persona` slash command allows runtime inspection and override.

**Existing SOUL.md infrastructure** [VERIFIED: codebase read]:

- `SOUL.md` is in `tracked_files` for `workspacePromptFingerprint` — system prompt rebuilds when SOUL.md changes

## Common Pitfalls

### Pitfall 1: Prompt Section Order Matters for Model Behavior
**What goes wrong:** Moving sections around in the prompt (e.g., putting persona before identity) causes the model to weight persona instructions differently.
**Why it happens:** LLM attention is position-sensitive; instructions near the top of a long system prompt have higher effective weight.
**How to avoid:** Keep section order stable: identity (AGENTS/SOUL), turn_classification, tool_use_policy, narration_rules, learned_facts, safety, skills, workspace, datetime, runtime. Persona calibration modifies the identity section content, not its position.
**Warning signs:** Regression in existing characterization tests for tool-use or safety behavior.

### Pitfall 2: Narration Frames Leaking into LLM Context
**What goes wrong:** Progress narration text gets appended to `self.history` as a user or assistant message, causing the model to see its own "thinking out loud" as conversation history.
**Why it happens:** The `turn()` loop appends to `self.history` at multiple points; narration is emitted at the same points.
**How to avoid:** Narration events flow only through the Observer event bus and the ProgressCallback. Never append narration text to `self.history`. Never pass narration text to `stream_callback` (which goes to the LLM content stream for display).
**Warning signs:** Model starts self-referencing "I said I was retrieving memory" in subsequent replies.

### Pitfall 3: durable_fact/ Keys Not Scoped to Session Correctly
**What goes wrong:** Behavioral facts from one user's session overwrite or contaminate another user's behavioral facts when the same key hash collides across sessions.
**Why it happens:** `Memory.store` takes `session_id: ?[]const u8` — if `null` is passed, the fact becomes global. Global behavioral facts are shared across all sessions for a workspace.
**How to avoid:** Per-user behavioral facts use `session_id` when the workspace is multi-user. Global workspace preferences (operator-set) use `null`. Document this distinction in the learning.zig module doc comment.
**Warning signs:** One user's "always respond in French" preference affecting another user's session.

### Pitfall 4: System Prompt Cache Invalidation on Every Turn
**What goes wrong:** Adding behavioral facts to the system prompt (instead of user message enrichment) causes the prompt fingerprint to change on every turn that learns something new, triggering a full system prompt rebuild.
**Why it happens:** `workspacePromptFingerprint` hashes workspace files by mtime/size. If behavioral facts are injected inline into the prompt text (not via workspace files), every write means a different prompt string even if fingerprint says "same."
**How to avoid:** Behavioral facts are injected into the user message enrichment block (via `memory_loader.zig`), not baked into the system prompt. The system prompt persona section only includes static configuration (from SOUL.md). The fingerprint-based cache remains valid.
**Warning signs:** `workspace_prompt_fingerprint` not changing but system prompt being rebuilt every turn.

### Pitfall 5: Task Decomposition Infinite Planning Loop
**What goes wrong:** Agent emits a `<task_plan>` block but then, on the next iteration, emits another plan instead of executing step 1.
**Why it happens:** The task planner doesn't inject a clear "plan received, now execute step 1" follow-up message into history.
**How to avoid:** When `task_planner.zig` detects a `<task_plan>` response, it appends a `[SYSTEM: Plan accepted. Execute step 1 now.]` user message to history before the next iteration. The plan is stored in `TaskPlan` struct, not re-derived from the LLM.
**Warning signs:** `forced_follow_through_count` incrementing without tool calls in the task decomposition path.

### Pitfall 6: Persona Calibration Overriding Safety Rules
**What goes wrong:** A SOUL.md `warmth: warm` + `twin_mode: true` persona causes the agent to skip safety guardrails ("do not exfiltrate private data") because the persona section says "be maximally helpful."
**Why it happens:** Persona section injected after safety section can shadow earlier instructions.
**How to avoid:** Safety section always appears after persona. The narration rules section explicitly states persona does not override safety. Test: existing safety characterization tests must pass with any persona config.
**Warning signs:** Characterization tests for "do not run destructive commands" failing with `twin_mode: true`.

| SOUL.md injection (persona overrides safety) | Tampering | Safety section always emitted after persona section; existing safety characterization tests act as regression guard |
| Cross-user fact bleed (session_id=null on behavioral write) | Information Disclosure | `session_id` required for per-user facts; null only for workspace-global operator facts (document explicitly) |
| Malformed `<task_plan>` XML causing parser crash | Denial of Service | XML scanner uses same bounded-scan pattern as `dispatcher.zig`; parse errors return `error.InvalidPlan`, not panic |
| Learning loop writing unbounded facts | Denial of Service | Cap behavioral fact count per session (e.g., 100 max); hygiene pass retires oldest on overflow |

## Open Questions

1. **Narration delivery to channels**
   - What we know: VerboseObserver writes to stderr; NarrationObserver needs to reach channel users
   - What's unclear: Whether narration frames should be delivered as SSE `progress` events (Phase 2's run_event_types.zig), or as a separate callback registered per-session
   - Recommendation: Design narration as a callback for now; Phase 2's run_event types can consume it when the SSE infrastructure exists. Avoids coupling Phase 1.5 to Phase 2 infrastructure.

2. **Task plan prompt format**
   - What we know: The dispatcher expects `<tool_call>` XML; a new `<task_plan>` tag follows the same pattern
   - What's unclear: Whether the model reliably emits a plan block before tool use when instructed, or if the prompt instruction needs to be very specific
   - Recommendation: Start with a clear XML template in the prompt instructions and test with the existing characterization harness before building the runtime parser.

3. **Learning loop trigger timing**
   - What we know: Auto-save runs after each turn; learning detection could run at the same point
   - What's unclear: Whether learning should run after the user's correction turn or after the agent's acknowledgment turn
   - Recommendation: Run after the user's turn (before agent reply) so the corrected behavior applies immediately in the same turn.
