Create src/tasks/delivery.zig:

The key design: TaskDelivery does NOT add a new ObserverEvent variant. Instead, it emits task_update events via the existing `narration_frame` variant with `frame_type = .plan_step` and a message containing the task status. This avoids modifying observability.zig (which is shared infrastructure). The RunEventObserver in Plan 02 will recognize task-related narration frames by message prefix convention.

**Alternative approach (preferred if simpler):** Add a new `task_update` variant to ObserverEvent. This is cleaner but requires modifying observability.zig and updating all exhaustive switch statements (LogObserver, FileObserver, OtelObserver, VerboseObserver).

Choose the approach that results in the least code churn. If adding a new ObserverEvent variant, follow the exact pattern used when `narration_frame` was added in Phase 1.5 Plan 02:
1. Add variant to ObserverEvent union in observability.zig
2. Add switch arm to LogObserver.logRecordEvent
3. Verify other observers use `else =>` or add arm

pub const TaskDelivery = struct {
    ledger: *TaskLedger,
    observer: Observer,

    pub fn createTask(self: *TaskDelivery, description: []const u8, owner_session: []const u8) ![]const u8 {
        const entry = try self.ledger.createTask(description, owner_session);
        self.emitTaskUpdate(entry.taskIdSlice(), "queued", description);
        return entry.taskIdSlice();
    }

    pub fn markRunning(self: *TaskDelivery, task_id: []const u8) !void {
        try self.ledger.markRunning(task_id);
        self.emitTaskUpdate(task_id, "running", null);
    }

    pub fn markSucceeded(self: *TaskDelivery, task_id: []const u8, result: ?[]const u8) !void {
        try self.ledger.markSucceeded(task_id, result);
        self.emitTaskUpdate(task_id, "succeeded", result);
    }

    pub fn markFailed(self: *TaskDelivery, task_id: []const u8, err_msg: ?[]const u8) !void {
        try self.ledger.markFailed(task_id, err_msg);
        self.emitTaskUpdate(task_id, "failed", err_msg);
    }
