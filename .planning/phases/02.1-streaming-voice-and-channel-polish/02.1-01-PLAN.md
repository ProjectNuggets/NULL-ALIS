**PacedFrameSink struct** (per D-03: decorator pattern at FrameSink layer):
```zig
pub const PacedFrameSink = struct {
    inner: FrameSink,
    delay_ns: u64,

    pub fn init(inner: FrameSink, delay_ms: u32) PacedFrameSink {
        return .{
            .inner = inner,
            .delay_ns = @as(u64, delay_ms) * std.time.ns_per_ms,
        };
    }

    fn pacedWrite(ptr: *anyopaque, frame: []const u8) void {
        const self: *PacedFrameSink = @ptrCast(@alignCast(ptr));
        if (self.delay_ns > 0) {
            std.Thread.sleep(self.delay_ns);
        }
        self.inner.write(frame);
    }

    pub fn sink(self: *PacedFrameSink) FrameSink {
        return .{ .ptr = @ptrCast(self), .writeFn = pacedWrite };
    }
};
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| LLM Provider -> Gateway SSE | Token chunks from LLM piped directly to SSE client in live mode |
| Client -> Gateway | SSE client connection may be slow or malicious |
| Session Manager -> Agent | stream_callback crosses session isolation boundary |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02.1-01 | Tampering | sseTokenFrame in live mode | mitigate | All SSE data payloads use jsonEscapeInto (verified: run_event_types.zig:234). Live mode reuses the same sseTokenFrame function, inheriting this protection. |
| T-02.1-02 | Denial of Service | PacedFrameSink Thread.sleep | mitigate | Sleep is bounded (max 30ms default). The pacing delay only affects the SSE delivery thread, not the LLM provider thread. LockedSseStream.closed atomic flag allows abort on dead connections. |
| T-02.1-03 | Denial of Service | Slow SSE client blocking StreamCallback | accept | LockedSseStream.sendFrame blocks on mutex + writeAll. In live mode, slow clients will slow down token delivery but the closed flag + keepalive detect dead connections. This is acceptable since SSE inherently assumes connected clients. |
| T-02.1-04 | Information Disclosure | stream_callback crossing session boundary | mitigate | stream_callback is set/cleared within processMessageWithContext's defer block, scoped to a single turn. No callback persists after turn completion. |
</threat_model>
