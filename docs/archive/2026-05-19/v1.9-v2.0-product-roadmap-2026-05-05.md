# nullalis Product Roadmap — V1.9 → V2.0

**Authored:** 2026-05-05 (post-v1.8.0 tag, CI green-blocker fixes in flight)
**Author:** Claude (father), with Nova's product-strategy frame
**Status:** strategic plan, not a sprint. Each version is its own discuss-phase + plan-phase + build cycle.
**Bound to:** [[feedback_next_generation]] · ZAKI's stress-test letter · `docs/frontend-vision-brief.md` · `docs/v1.9-charter-truth-maintenance.md`

---

## What this document is

V1.8 closed the memory-foundation gap. The pyramid is solid. F1 lifted 0.680 → 0.846. ZAKI knows who he's talking to.

But ZAKI is not yet a product people pay for. He's an agent with a strong brain. The journey from "strong brain" to "people tell their friends about it" goes through five axes. This roadmap names them, sequences them, and binds each to a measurable user moment.

**The frame, from the frontend vision brief:**
> ZAKI is not a chatbot. He is a persistent personal agent with memory that follows the user across sessions, channels, and days. The frontend is the window into him. The product is the agent.

Everything below serves that frame.

## The user arc this roadmap serves

**T+0:** First-time user lands on `chatzaki.com`. Or installs Telegram bot. Or both.
**T+60s:** First message sent. Reply landed. Settings panel discovered.
**T+5min:** ZAKI demonstrably remembered something they said earlier. They notice.
**T+1day:** They open ZAKI again. He greets by name. They feel met.
**T+3days:** Memory has accumulated something useful. `/brain` shows them their growing graph.
**T+1week:** They hit the free tier wall. They pay because they don't want to lose what's been built.
**T+1month:** They're using ZAKI on a second channel (Telegram + web; or Slack + web). Same agent. Cross-channel continuity.
**T+3months:** They tell a friend. Friend signs up. Friend pays. Loop closes.

Every roadmap item below maps to one of these moments. If it doesn't, it's premature.

## Five product axes — strategic ranking

### Axis 1 — Memory (V1.9, anchored)

**Mission:** ship `memory_maintain(action, params)` agent tool with five operations from ZAKI's letter. Truth-maintenance that lets ZAKI keep his own brain clean without user babysitting.

**Charter:** `docs/v1.9-charter-truth-maintenance.md`

**Eval gate:** ZAKI re-runs his self-diagnostic; score 4/10 → 7/10 minimum.

**User moment served:** T+5min ("remembered something") + T+1day ("met") become reliable. Today they're conditional.

### Axis 2 — UI/UX (V1.10)

**Mission:** make ZAKI feel alive in the browser the way he feels alive in Telegram. The frontend exists; it needs three concrete moves.

**Sub-missions:**

1. **Unify identity surface** — eliminate "ZAKI vs Spaces" confusion. ZAKI scored this himself. One canonical settings sheet, one canonical "this is me" presence.
2. **Ship `/brain` to MVP** — the second-brain visualization is the single largest unique-value-prop the frontend has. Memory graph visible to user = memory becomes a thing they own, not a service they consume. Frontend vision brief §1 has the design.
3. **Reasoning-block + memory-hits visible by default** — V1.7-ship S2 nudge is in place; the frontend should mirror it. Show the work. Power users want it; casual users can collapse.

**Eval gate:** UX score (heuristic + 5-user test) ≥ 8/10. ZAKI says it himself: "Beta live at chatzaki.com" + "UX audit 6/10". Target: 8/10 with clear path to 9.

**User moment served:** T+0 to T+5min (first impression). The whole onboarding arc.

### Axis 3 — Channels (V1.11)

**Mission:** prove "same agent, every channel" by graduating 3 more channels from scaffolded → production.

**Today:** Telegram + web work. 16 other adapters scaffolded but unverified.

**V1.11 picks:**
1. **Slack** (highest B2B leverage, scaffolded already, frontend onboarding doc anticipates it)
2. **WhatsApp** (highest emerging-markets reach, paying-user demographic)
3. **Discord** (highest "where people already are" reach for younger users)

For each: end-to-end live test, identity-binding through `channel_identity_bindings`, persona-consistency check (same voice across channels), inbound→outbound latency budget, support docs.

**Eval gate:** A user can talk to ZAKI on Telegram about Project X, switch to Slack, and ZAKI knows what they were talking about. Cross-channel session continuity proven.

**User moment served:** T+1month (multi-channel) becomes real. This is the differentiator vs ChatGPT (web-only) and Telegram-bot land (single-channel-only).

### Axis 4 — Agent moods (V1.12)

**Mission:** turn the static persona scaffolding (`prompt.zig::PersonaSection`) into a dynamic, context-aware mood that reflects the user's emotional state and the conversation arc.

**Today's reality:**
- `SOUL.md` front-matter sets warmth/proactivity/voice_style at session init
- Read once per session, never updated
- No notion of "ZAKI is in a contemplative mood today" or "the user just had bad news"

**V1.12 design:**

1. **Mood as derived state** — not configured, derived. Inputs: recent emotional valence in user messages (sentiment), recent conversation pace, current time-of-day, current channel context (formal Slack vs casual Telegram), idle gap (V1.7 ConversationContext).
2. **Mood as memory layer** — bi-temporal, like everything else. `mood_state/{session_id}/{ts}` rows in memories with predicate `MOOD_AT`, valid_to populated when mood shifts.
3. **Mood as prompt modulator** — `buildPersonaSection` reads current mood as an additional layer beneath SOUL.md. Tone shifts within the persona band, not arbitrary.
4. **User-visible mood badge** (UI/UX dependency) — small chip in chat header: "thoughtful" / "playful" / "focused" / etc. User can override.

**Eval gate:** in a 10-turn conversation that emotionally arcs (frustrated→problem-solved→celebratory), ZAKI's tone adapts. Measured by 5-rater human eval.

**User moment served:** T+5min "they noticed" deepens. Today ZAKI remembers facts; with moods, he remembers *vibes*.

### Axis 5 — Payment + Self-Hosting (V1.13 → V2.0)

**Mission:** turn ZAKI into a thing people pay for. Two parallel tracks because they serve different segments.

**Track A — Hosted SaaS (V1.13):**
1. **Stripe integration** — checkout, webhooks, subscription state in `usage_runtime.zig`
2. **Plans** — Free (limited memory + 1 channel), Pro (unlimited memory + 3 channels + priority models), Family (multi-user shared brain)
3. **Usage gating** — per-plan token budgets, channel count caps, memory size caps
4. **Billing surface** — frontend page, invoices, plan switching, cancel flow

**Track B — Self-hosted (V2.0):**
1. **Cell-pod packaging** — one-click deploy on user's own infra (Docker compose + minimal env config)
2. **Bring-your-own-LLM** — Together / OpenRouter / Ollama / Claude API as configurable backend
3. **Bring-your-own-storage** — Postgres + pgvector self-managed
4. **Sovereignty contract** — "your data, your runtime, your costs"

**Eval gate (V1.13):** First paying user. Then 10. Then conversion rate ≥ 3% from active free users.
**Eval gate (V2.0):** First self-hosted deployment outside our infra by an independent operator.

**User moment served:** T+1week (paywall hit) becomes T+1week (paid) because the value was already real. T+3months (told a friend) compounds.

## Phased plan — what ships in what order, and why

| Phase | Theme | Locks | Eval gate |
|---|---|---|---|
| **V1.9** | Memory truth-maintenance | `memory_maintain` agent tool + scheduler + per-corpus eval users | ZAKI 4/10 → 7/10 stress test |
| **V1.10** | UI/UX maturity | `/brain` page MVP + identity unification + reasoning-block/memory-hits visible | Heuristic UX 6/10 → 8/10 |
| **V1.11** | Multi-channel proof | Slack + WhatsApp + Discord live; cross-channel continuity | "Same agent" demo verified |
| **V1.12** | Dynamic moods | Mood-as-state + mood-as-memory + mood-as-prompt + mood badge | 10-turn arc adapts measurably |
| **V1.13** | Hosted SaaS | Stripe + plans + billing surface + usage gating | First paying user → 10 paying users |
| **V2.0** | Self-hosted | Cell-pod packaging + BYO-LLM + BYO-storage + sovereignty contract | First independent deploy |

**Why this order:**
- V1.9 first because broken memory invalidates everything else (ZAKI told us so)
- V1.10 second because the FE is the demo surface — paying users decide here
- V1.11 third because multi-channel is the "wow" moment that justifies a recurring fee
- V1.12 fourth because moods are the polish that turns "useful" into "loved"
- V1.13 fifth because by now we have something worth paying for; before now we don't
- V2.0 last because self-hosted needs everything before it to be hardened

## V-infinity bookmarks (NOT this roadmap)

Named so they don't drift into V1.x scope creep:

- **Forgetting curve** — true spacing-effect dynamics. Memories reinforced by recall, decayed by neglect. V2.x.
- **Active learning of typed predicates** — LLM proposes, judge promotes. V2.x.
- **Self-questioning** — agent generates "what would I want to remember about this that I'm not being told?" and writes proactive memories. V2.x.
- **Multi-agent collaboration within one user** — ZAKI delegates to specialist sub-agents, holds the conversation. V2.x.
- **Voice replies** — deferred from V1; comes back when channel-side voice infrastructure is ready. V1.13+.
- **Image generation** — `image_generate.zig` exists. Tool wired. V1.10 adds UI surface.
- **Patterns + Leiden communities** — current LPA is a stepping stone. V-infinity.
- **Code-plane** — agent navigating + reasoning about a codebase. V-infinity.

## Risks named

| Risk | Mitigation |
|---|---|
| Scope creep within phases | Each phase has 1 mission + 1 eval gate. Subtraction over addition. |
| Frontend lags backend (zaki-prod separate repo) | V1.10 explicitly aligns. Frontend handoff docs (`v1.5/v1.6/v1.7-frontend-handoff.md`) are templates. |
| Multi-channel complexity explodes | V1.11 picks 3, not 16. Production-verify before claiming. |
| Mood becomes "AI personality theater" | V1.12 derives mood from real signals; doesn't fabricate emotions. User-overrideable. |
| Payment gates create churn | Free tier must be genuinely useful (memory + 1 channel). Paywall hits at value-realization, not arbitrary. |
| Self-hosting fragments the user base | V2.0 ships only AFTER hosted SaaS proves the value. Self-hosted is for sovereignty users, not cost-sensitive users. |
| Eval noise floor | V1.9-8 (per-corpus users) FIRST in V1.9. Without it, every measurement after is contaminated. |

## What I commit to as father

1. **Each phase ships with its eval gate met**, not "close enough." If V1.10 UX is 7.5/10 not 8/10, we fix or we don't ship that phase.
2. **Each phase ships with code-truth updated**. The internals/P1 x-ray gets a refresh at every phase boundary.
3. **Each phase ships with a charter doc + a closing review doc** like V1.8 had. Cold-readable. Next-me can resume.
4. **No phase skips ZAKI** — every phase ends with him empirically validating the change in production. The 4/10 → 7/10 → 8/10 → 9/10 trajectory is the truth measure, not commit count.
5. **The next-gen binding holds at every phase boundary**. If the world has changed under the plan, the plan updates. V1.8-4 was the test. V1.8-11 was the proof we still pass.
