## Summary

Phase 1.5 delivers five modules: composable prompt sections (Plan 01), liveness narration (Plan 02), task decomposition XML parser (Plan 03), learning loop (Plan 04), and persona calibration (Plan 05). The implementation is generally well-structured -- each module follows the established section-builder and observer-vtable patterns, tests are thorough, and the integration into root.zig is clean.

The most significant finding is a **memory leak** in `handlePersonaCommand` where the `voice` string allocated by `resolvePersonaFromFile` is never freed. There are also several warnings around unenforced rate limits, missing key validation on `/learn forget`, and a silent data-loss edge case in the XML text extractor.

The codebase demonstrates good Zig practices: `defer`/`errdefer` for resource cleanup, bounded reads for file I/O, null-safe return patterns for parsers, and exhaustive switch coverage in observers.

## Critical Issues

### CR-01: Memory Leak in handlePersonaCommand -- voice string never freed

**File:** `src/agent/commands.zig:3604-3638`
**Issue:** `handlePersonaCommand` calls `prompt_mod.resolvePersonaFromFile(self.allocator, workspace_dir)` which allocates and dupes the `voice` string (see `prompt.zig:160-163`). The returned `PersonaProfile.voice` is an owned allocation. However, `handlePersonaCommand` never frees `profile.voice` before returning. Every invocation of `/persona` when SOUL.md has a `voice:` field leaks the duped voice string.

Compare with the correct pattern in `root.zig:1286-1288`:
```zig
defer if (persona_profile_opt) |p| {
    if (p.voice) |v| self.allocator.free(v);
};
```

**Fix:** Add a defer to free the voice allocation:
```zig
const profile = profile_opt.?;
defer if (profile.voice) |v| self.allocator.free(v);
```

Insert this immediately after line 3616 (`const profile = profile_opt.?;`).

## Warnings

### WR-01: MAX_FACTS_PER_SESSION defined but never enforced

**File:** `src/agent/learning.zig:19` (definition), `src/agent/root.zig:1399-1413` (write site)
**Issue:** `MAX_FACTS_PER_SESSION = 100` is exported as the DoS mitigation for threat T-1.5-08, but the actual write call site in `root.zig` (line 1409: `mem.store(k, fc, .core, ...)`) never checks the current fact count against this limit. An adversary who sends 1000 messages containing correction patterns ("no, actually...") would store 1000 facts without bound.

The SUMMARY acknowledges this ("Enforcement at the write call site... is a follow-on task"), but the threat model T-1.5-08 claims it is mitigated. The constant exists but enforcement does not.

**Fix:** Before storing, count existing behavioral facts and reject if at limit:
```zig
if (fact_content) |fc| {
    // T-1.5-08: enforce MAX_FACTS_PER_SESSION
    const existing = mem.list(self.allocator, null, self.memory_session_id) catch null;
    const fact_count = if (existing) |entries| blk: {
        defer memory_mod.freeEntries(self.allocator, entries);
        var c: usize = 0;
        for (entries) |e| {
            if (std.mem.startsWith(u8, e.key, "durable_fact/behavior/")) c += 1;
        }
        break :blk c;
    } else 0;
    if (fact_count >= learning.MAX_FACTS_PER_SESSION) {
        log.warn("learning.max_facts_reached session={?s}", .{self.memory_session_id});
    } else {
        // ... existing store logic ...
    }
}
```

Note: This is listed as Warning rather than Critical because the memory store itself has independent size limits, and the stored facts are session-scoped. But the documented threat mitigation is currently a no-op.

### WR-02: extractTextAndPlan silently discards text_after when text_before is non-empty

**File:** `src/agent/task_planner.zig:159-168`
**Issue:** When an LLM response has text both before AND after the `<task_plan>` block, only `text_before` is returned. The `text_after` portion is silently dropped. The comment on line 162 acknowledges this ("caller can reconstruct from offsets"), but the `ExtractResult` struct provides no offset information -- only the `text` slice and `plan_xml` slice. A caller cannot reconstruct the after-text from these two fields alone.

Example: `"Explanation\n<task_plan>...</task_plan>\nNow executing step 1..."` -- the "Now executing step 1..." portion is lost.

**Fix:** Either return both text portions in ExtractResult:
```zig
pub const ExtractResult = struct {
    text_before: []const u8,
    text_after: []const u8,
    plan_xml: ?[]const u8,
};
```
Or document the limitation explicitly in the function's doc comment and ensure callers are aware. The current doc says "Text before and after the block is trimmed and returned as `text`" which is misleading -- only one portion is returned.

### WR-03: /learn forget does not validate key suffix format

**File:** `src/agent/commands.zig:3571-3591`
**Issue:** When a user provides a key suffix (not the full key), it is concatenated directly: `durable_fact/behavior/{user_input}`. No validation ensures the suffix contains only hex characters. While this is not exploitable (the memory store uses key-value semantics, not filesystem paths), it means a user typing `/learn forget my-note` would attempt to forget `durable_fact/behavior/my-note` which can never match a real key (real keys are 16-char hex). The user gets "No learned fact found" with no indication that their input format was wrong.

**Fix:** Validate the suffix is exactly 16 hex chars when it does not start with the full prefix:
```zig
const full_key = if (std.mem.startsWith(u8, raw_key, "durable_fact/behavior/"))
    try self.allocator.dupe(u8, raw_key)
else blk: {
    if (raw_key.len != 16) {
        return try self.allocator.dupe(u8, "Key must be the full durable_fact/behavior/... key or a 16-char hex suffix.");
    }
    break :blk try std.fmt.allocPrint(self.allocator, "durable_fact/behavior/{s}", .{raw_key});
};
```

### WR-04: NarrationObserver callback fallback uses inner Observer pointer as context

**File:** `src/agent/narration.zig:90`
**Issue:** When `self.callback_ctx` is null but `self.callback` is set, the code falls back to `@ptrCast(&self.inner)` as the context pointer passed to the callback. The `inner` field is an `Observer` (a fat pointer struct with `ptr` and `vtable`), so `&self.inner` is a `*Observer`. But the callback signature is `fn(ctx: *anyopaque, frame: NarrationFrame) void` -- the callback implementation has no way to know the actual type of `ctx`. If someone sets a callback without a context, the callback receives a pointer to the NarrationObserver's inner Observer field, which it cannot safely use without knowing the type.

This is a latent correctness issue -- no current caller exercises this path (the summary says "callback (for direct consumers)" but no code registers a callback yet). When a callback consumer is added, this fallback will likely cause a type confusion bug.

**Fix:** Either require `callback_ctx` when `callback` is set (assert at construction), or pass a null sentinel instead of a pointer to unrelated data:
```zig
fn emitFrame(self: *NarrationObserver, frame: NarrationFrame) void {
    if (self.callback) |cb| {
        if (self.callback_ctx) |ctx| {
            cb(ctx, frame);
        }
        // If no context is provided, skip the callback rather than passing garbage.
    }
    // ...
}
```

## Info

### IN-01: ToolUsePolicy.execution_mode is a string literal with no validation

**File:** `src/agent/prompt.zig:218`
**Issue:** `ToolUsePolicy.execution_mode` defaults to `"execute"` as a `[]const u8`. There is no enum or validation for this field. If a caller sets it to an arbitrary string, there is no compile-time or runtime check. This is an extension-point stub that is currently unused (`_ = ctx.sections.tool_use`), so no immediate risk, but when it is wired it should use an enum.

**Fix:** When the tool_use extension point is wired, convert `execution_mode` from `[]const u8` to an enum:
```zig
pub const ExecutionMode = enum { execute, plan_only, dry_run };
```

### IN-02: Redundant "i meant" check in detectLearningSignals

**File:** `src/agent/learning.zig:105`
**Issue:** The explicit_correction patterns already include `"i meant"` (line 40). The code on line 105 adds a separate check for `"i meant"` under the implicit_correction branch, but this branch only runs when `explicit_correction` was NOT detected. Since `"i meant"` is in CORRECTION_PATTERNS, if "i meant" appears in the message, `explicit_correction` will always be set first, making the line-105 check unreachable.

This is dead code -- the `std.mem.indexOf(u8, lower, "i meant")` on line 105 can never evaluate to true when `explicit_correction` is not already in the found set.

**Fix:** Remove lines 104-107:
```zig
// Remove: redundant check -- "i meant" always triggers explicit_correction first
```

### IN-03: Task decomposition prompt section is unconditional

**File:** `src/agent/prompt.zig:326-327`
**Issue:** `buildTaskDecompositionSection(w)` is called unconditionally in `buildSystemPrompt`, regardless of whether task decomposition is enabled or desired. For simple chat-only deployments, this adds ~15 lines of task decomposition instructions to every system prompt. The Plan 03 SUMMARY notes this is intentional ("task decomposition is always-on"), but it contrasts with the conditional pattern used for persona (`if (ctx.sections.persona) |persona|`).

**Fix:** No immediate change needed. When a feature flag is desired, gate it behind `ctx.sections.tool_use` or a dedicated `task_decomposition_enabled` flag in PromptSections.
