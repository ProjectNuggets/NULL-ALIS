# Phase 02.1: Streaming, Voice, and Channel Polish - Context

**Gathered:** 2026-04-10 (assumptions mode)
**Status:** Ready for planning

<domain>
## Phase Boundary

Make ZAKI narrate itself like Claude Code — progressive streaming showing real-time agent activity, voice-first agent mode across channels, operator channel health dashboard, and structured security audit command. Builds on Phase 1.5 narration infrastructure and Phase 2 SSE run-event stream.

Requirements: REQ-023, REQ-024, REQ-025, REQ-026
</domain>

<decisions>
## Implementation Decisions

### Progressive Streaming (REQ-023)
- **D-01:** Extend existing RunEventObserver/FrameSink pipeline with per-channel delivery mode selection (live SSE vs buffered-replay vs batch) and human-pacing delays between token chunks. Do NOT build a new streaming infrastructure.
- **D-02:** Switch gateway from hardcoded `buffered_replay` (live: false) to configurable delivery modes. The ReplyStartPayload already has `delivery_mode` and `live` fields — wire them to actual behavior.
- **D-03:** Human pacing = configurable inter-chunk delays at the SSE delivery layer (FrameSink decorator pattern with optional delay logic), NOT in the LLM provider callback. Different channels have different latency tolerance.
- **D-04:** Per-channel pacing config: web SSE client gets token-by-token fidelity (~20-50ms), Telegram/Discord tolerate larger batches, CLI gets immediate dump.

### Voice-First Agent Mode (REQ-024)
- **D-05:** Extend existing TtsMode/Transcriber vtable to work across channels, not just Telegram. Remove `ttsAudioChannelSupported` hardcode for "telegram" — replace with channel-capabilities check.
- **D-06:** Add VoiceMode that composes STT (inbound) + agent processing + TTS (outbound) as a unified pipeline, with existing TtsMode and WhisperTranscriber as internal components.
- **D-07:** Voice-specific narration through NarrationObserver (e.g., "Listening...", "Thinking...", "Speaking...") — hooks into the Phase 1.5 narration_frame events.
- **D-08:** Multi-provider TTS resolution already exists (openai/openrouter/together) — reuse, don't rebuild.

### Channel Health Dashboard (REQ-025)
- **D-09:** Aggregate existing health.zig component registry + ChannelManager entry state + SupervisedChannel failure tracking into a single dashboard view.
- **D-10:** Expose as gateway HTTP endpoint `/api/v1/channels/health` returning JSON, reusing ReadinessResult.formatJson pattern. Also add `/health` slash command for chat-interface access.
- **D-11:** Include per-channel: listener type, thread status, failure count, backoff state, last success/error timestamps.

### Security Audit Command (REQ-026)
- **D-12:** Build on doctor.zig DiagItem/Severity framework + security/audit.zig event types. New `/security-review` slash command + gateway API endpoint.
- **D-13:** Structured checks cover: sandbox enablement, API key exposure, auth config, SSRF protection (net_security.zig), workspace isolation (config.workspace_only), audit log config, rate limiting (max_actions_per_hour).
- **D-14:** Output as JSON report with pass/fail/warning per rule and overall security score. Extensible via SecurityCheck vtable pattern for future checks.

### Claude's Discretion
- Exact inter-chunk delay values for pacing (start with 30ms default, tune later)
- Voice-first mode activation UX (slash command vs config vs auto-detect)
- Security score calculation formula
- Health dashboard polling interval recommendations

</decisions>

## Existing Code Insights

### Reusable Assets
- **LockedSseStream** — Thread-safe SSE frame delivery with keepalive heartbeat (gateway.zig)
- **RunEventObserver** — Translates 6 ObserverEvent types into SSE frames via FrameSink (gateway_run_events.zig)
- **NarrationObserver** — Wraps Observer, translates tool_call_start and turn_stage into NarrationFrame (narration.zig)
- **WhisperTranscriber** — STT via Whisper API with vtable dispatch (voice.zig)
- **TTS synthesis pipeline** — Multi-provider endpoint resolution, audio file handling (voice.zig)
- **Health component registry** — Thread-safe global health tracking with JSON serialization (health.zig)
- **DiagItem framework** — Severity-classified diagnostic checks with summary renderer (doctor.zig)
- **SecurityContext** — Policy violation tracking and audit event recording (security/audit.zig)

### Established Patterns
- Observer vtable chain: NarrationObserver -> inner Observer, RunEventObserver -> inner Observer
- SSE chunking: text split into SSE_TOKEN_CHUNK_SIZE (96 byte) frames
- Slash commands: registered in commands.zig with consistent help/execution pattern
- Gateway API endpoints: `/api/v1/...` pattern with JSON responses
- Tool vtable: metadata-based dispatch (used by Phase 2 task tools)

### Integration Points
- gateway.zig `chat_stream` handler — where delivery_mode switch from buffered_replay to live must happen
- ReplyStartPayload — already has delivery_mode/live fields, just not wired
- ObserverEvent enum — add new variants (voice_status, channel_health) for observer bus delivery
- ChannelManager — source of per-channel runtime state for health aggregation
- Provider StreamCallback — existing streaming path that needs to wire through to live SSE delivery

</code_context>

## Specific Ideas

- "ZAKI needs to narrate itself like Claude Code" — the core UX motivation. Progressive streaming should show what the agent is doing in real-time: reading files, calling tools, thinking, generating responses.
- The existing narration_frame events from Phase 1.5 already carry semantic labels ("Reading memory...", "Calling tool X...") — these need to flow through to SSE clients in live mode.
- Voice-first is about multi-channel voice, not just Telegram — WhatsApp, Discord, and web should all support voice interaction.
