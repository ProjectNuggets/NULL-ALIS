# Phase 02.1: Streaming, Voice, and Channel Polish - Research

**Researched:** 2026-04-10
**Domain:** SSE streaming delivery modes, voice pipeline, health aggregation, security audit
**Confidence:** HIGH

## Summary

This phase extends four existing subsystems rather than building new ones. The codebase already has all the primitives needed: `LockedSseStream` for thread-safe SSE delivery, `RunEventObserver`/`FrameSink` for event-to-SSE translation, `NarrationObserver` for semantic progress frames, `WhisperTranscriber`/`synthesizeTextToTempAudio` for voice, `health.zig` component registry for health tracking, and `doctor.zig` DiagItem framework for structured diagnostics.

The primary engineering challenge is wiring existing pieces together, not building new abstractions. For streaming (REQ-023), the `sseChatPayload` function is hardcoded to `buffered_replay` with `live: false` -- the agent collects the full reply, then the gateway chunks it into 96-byte SSE frames post-hoc. Switching to live delivery means the provider's `StreamCallback` must pipe chunks directly to `LockedSseStream.sendFrame` as they arrive from the LLM. For voice (REQ-024), `ttsAudioChannelSupported` is hardcoded to return true only for `"telegram"` -- this needs to become a channel-capabilities check. For health (REQ-025) and security (REQ-026), the primitives exist but no aggregation endpoints exist yet.

**Primary recommendation:** Implement in four independent vertical slices (streaming, voice, health, security-audit) since each requirement touches different files with minimal overlap. Streaming is the riskiest and should come first since it modifies the gateway's hot path.

## Architecture Patterns

### Current SSE Delivery (Buffered Replay -- What Exists Today)

The current chat_stream handler in `gateway.zig` (lines 8575-8707) follows this flow:

```
1. Client connects -> LockedSseStream created
2. SseKeepalive thread spawned (heartbeat every 10s)
3. SseProgressObserver created -> sends progress frames during agent processing
4. Agent processes full message -> returns complete reply text
5. Gateway THEN chunks reply into 96-byte SSE frames (post-hoc)
6. reply_start(delivery_mode="buffered_replay", live=false) -> tokens -> done
```

### Target SSE Delivery (Live Streaming)

```
1. Client connects -> LockedSseStream created
2. SseKeepalive thread spawned
3. reply_start(delivery_mode="live", live=true) sent BEFORE agent starts
4. Agent's StreamCallback pipes chunks directly to LockedSseStream.sendFrame
5. Progress events interleaved with token chunks
6. done frame sent when agent completes
```

### Pattern: FrameSink Decorator for Pacing (D-03)

```zig
// PacedFrameSink wraps a FrameSink and adds inter-chunk delays
pub const PacedFrameSink = struct {
    inner: FrameSink,
    delay_ms: u32,       // inter-chunk delay
    batch_size: u32,     // accumulate N chunks before flush (for batch channels)
    
    pub fn write(self: *PacedFrameSink, frame: []const u8) void {
        if (self.delay_ms > 0) {
            std.Thread.sleep(@as(u64, self.delay_ms) * std.time.ns_per_ms);
        }
        self.inner.write(frame);
    }
    
    pub fn sink(self: *PacedFrameSink) FrameSink {
        return .{ .ptr = @ptrCast(self), .writeFn = pacedWrite };
    }
};
```

This follows the existing observer chain pattern: `NarrationObserver -> inner Observer`, `RunEventObserver -> inner Observer`. [VERIFIED: codebase pattern from narration.zig and gateway_run_events.zig]

## Common Pitfalls

### Pitfall 1: SSE Backpressure When Client is Slow
**What goes wrong:** If the SSE client reads slowly, `LockedSseStream.sendFrame` blocks on `sendChunkedSseFrame` which calls `stream.writeAll`. In live mode, the provider's StreamCallback thread will block.
**Why it happens:** The provider stream is fast; the SSE client may be on a slow connection.
**How to avoid:** The PacedFrameSink decorator's delay already rate-limits. Additionally, the `closed` atomic flag on LockedSseStream lets the keepalive or main thread detect a dead connection and abort.
**Warning signs:** `chat.stream.progress emit failed` log messages (already present in SseProgressObserver).

### Pitfall 2: Concurrent Access Between StreamCallback and Progress Observer
**What goes wrong:** In live mode, both the agent's StreamCallback (emitting token chunks) and the Observer chain (emitting progress events) write to the same LockedSseStream simultaneously.
**Why it happens:** Progress events happen on the agent turn thread; stream chunks arrive on the same thread but interleaved with tool execution.
**How to avoid:** LockedSseStream's mutex already serializes writes. The existing pattern is correct. But ensure the PacedFrameSink also acquires the lock (it delegates to the inner FrameSink which ultimately calls LockedSseStream.sendFrame which is already mutex-protected). [VERIFIED: LockedSseStream.sendFrame at gateway.zig:7990-7994]

### Pitfall 3: ttsAudioChannelSupported Check Scattered
**What goes wrong:** The hardcoded `"telegram"` check at `agent/root.zig:674-677` may not be the only place channel support is checked. Other code paths may also assume voice = telegram.
**Why it happens:** Voice was originally Telegram-only.
**How to avoid:** Search for all `"telegram"` string comparisons near voice/tts/audio paths. The TelegramSttMetrics in voice.zig are also telegram-specific -- these should remain as-is (they track telegram-specific metrics) but new generic metrics should be added for the cross-channel VoiceMode.
**Warning signs:** Voice works on web but not on Discord, or vice versa.

### Pitfall 4: Security Review Check Ordering
**What goes wrong:** Some security checks depend on config being loaded; others depend on runtime state. If checks run before the system is fully initialized, they may report false positives.
**Why it happens:** The doctor.zig pattern loads config then runs checks. But the security-review slash command runs WITHIN a running session where config is already loaded.
**How to avoid:** SecurityCheck functions should take `*const Config` as input (always available in a running session). Runtime-dependent checks (like "is sandbox process actually running") should use fallback values and report "unknown" rather than "fail" when runtime data is unavailable.

### Pitfall 5: Gateway Size Risk
**What goes wrong:** gateway.zig is already 15,599+ lines. Adding live streaming delivery path adds significant complexity to the chat_stream handler.
**Why it happens:** All SSE logic lives in one file.
**How to avoid:** Extract the new live-delivery path into a helper function. The PacedFrameSink and delivery mode logic can live in gateway_run_events.zig (already a sibling file). Keep gateway.zig changes to the minimum switch from buffered_replay to the new mode.

### Pitfall 6: Reply Ordering in Live Mode
**What goes wrong:** In buffered mode, `reply_start` -> all tokens -> `done` is guaranteed sequential. In live mode, progress events (tool_start, narration_frame) can interleave with token chunks.
**Why it happens:** Live mode sends tokens as they arrive while the observer chain also sends events.
**How to avoid:** The SSE protocol handles this naturally -- each frame has an `event:` type field. The frontend must handle interleaved `progress` and `token` events. Document that live mode interleaves event types (this is expected behavior, not a bug).

## Streaming Architecture Deep Dive

### Current Data Flow (Buffered Replay)
```
LLM Provider --StreamCallback--> Agent (accumulates full text)
                                    |
                                    v
                              Agent.turn() returns reply text
                                    |
                                    v
                              Gateway chunks into 96-byte SSE frames
                                    |
                                    v
                              LockedSseStream.sendFrame(each chunk)
```

### Target Data Flow (Live Mode)
```
LLM Provider --StreamCallback--> StreamTimingContext 
                                    |
                                    v (each chunk)
                              PacedFrameSink.write()
                                    |  (optional 30ms delay)
                                    v
                              LockedSseStream.sendFrame(sseTokenFrame)
                                    
In parallel:
NarrationObserver --narration_frame--> RunEventObserver
                                          |
                                          v
                                    FrameSink.write() 
                                          |
                                          v
                                    LockedSseStream.sendFrame(sseProgressFrame)
```

### Key Integration Points

1. **`gateway.zig:8596-8650` (ReplyOutcome block):** Currently calls `processMessage`/`processMessageWithContext` which returns the full reply text. For live mode, the agent needs `stream_callback` and `stream_ctx` set to pipe chunks to the SSE stream. [VERIFIED: agent/root.zig:358-361 shows stream_callback/stream_ctx fields]

2. **`gateway.zig:8584` (SseProgressObserver):** Already sends progress events during processing. In live mode, this observer chain continues to work -- it just coexists with token chunk delivery.

3. **`agent/root.zig:1553` (is_streaming check):** `self.stream_callback != null and self.provider.supportsStreaming()` -- this is where the agent decides to use streaming providers. The gateway must set stream_callback when delivery_mode="live". [VERIFIED: agent/root.zig:1553]

4. **`gateway.zig:8668-8684` (post-hoc chunking):** In buffered mode, this loop splits text into 96-byte chunks. In live mode, this loop is SKIPPED because chunks were already sent via StreamCallback. The done frame still needs to be sent.

### Delivery Mode Resolution

Per D-04, different channels need different modes:
```
"zaki_app" (web SSE)  -> "live"     (token-by-token, 30ms pacing)
"telegram"            -> "batch"    (accumulate, send as message)
"discord"             -> "batch"    (accumulate, send as message)
"cli"                 -> "live"     (immediate dump, 0ms pacing)
```
