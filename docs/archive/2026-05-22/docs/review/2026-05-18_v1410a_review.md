# V1.14.10 A — Async Lifecycle Persist: Code Review

**Reviewed:** 2026-05-18
**Commit:** `9115f501` (already merged to `main`)
**Reviewer:** Claude (gsd-code-reviewer, deep)
**Scope:** `src/agent/commands.zig` (+~75 LOC), `src/agent/root.zig` (~70 LOC modified)

## Summary

The patch is **well-scoped, correctly motivated, and the atomic-guard mechanics are sound**. It cleanly fixes a real bench-blocking timeout. However, the "detached-thread + spin-wait + give-up-and-segfault" lifecycle model leaks several latent correctness traps the patch's own comments candidly acknowledge ("worker may segfault" is logged, not prevented). The most pressing concerns are:

1. **Observer is NOT thread-safe at the vtable boundary** — the patch silently makes it concurrent.
2. **`recycleSessionInPlace` (TTL recycle path) calls `previous_agent.deinit()` while a worker may be mid-flight on the OLD `*Agent`** — the deinit-wait helps, but the worker also touches `agent.mem`, which is then reused by the replacement.
3. **The "ENOMEM / spawn-failed → fall back to sync" branch re-introduces the exact 30-180s blocking the patch was written to prevent** under load — which is precisely when thread spawn is most likely to fail.

No BLOCKERs (none of these will hard-crash a healthy battery), but several HIGH items want decisions before V1.14.10 B/C land.

## HIGH

### H-01: Observer is not documented as thread-safe; async worker calls `recordEvent` concurrently with `agent.turn()`

**File:** `src/agent/commands.zig:1722-1727` (deferred `recordEvent` inside `persistSessionCheckpointDetailed`) + `src/agent/commands.zig:1139` (`emitLifecycleSummarizerStage`)
**Observer vtable:** `src/observability.zig:155-181`

`Observer` is a vtable struct (`ptr: *anyopaque, vtable: *const VTable`). The `recordEvent` dispatch is a single indirect call, but **the implementations behind the vtable hold their own state** — e.g. `LogObserver` at L212, the OTLP-style observer at L535 has its own `mutex: std.Thread.Mutex` + `requests_total: atomic.Value(u64)`, and `recordEvent` callees frequently allocate (formatted log lines, span buffers).

Pre-async, every `recordEvent` on a given `Observer` instance was serialized by `agent.turn()` being the sole writer. Now the async worker fires `recordEvent` for `continuity_refresh` and `memory_lifecycle_summarizer` events on a parallel thread while the next `agent.turn()` may already be emitting events on the same `Observer` instance.

The L535 observer already takes a mutex — so it's safe. But `LogObserver` (L212+) is **not obviously safe** (`std.log` is thread-safe, but if it formats into a shared buffer it's not). And there's no contract anywhere in `observability.zig` that says "Observer implementations MUST be thread-safe."

**Fix recommendation:**
1. Add a doc-comment on `Observer.VTable` stating "Implementations MUST be safe to call concurrently from any thread." (or alternatively "MUST be called from a single thread" — and then the patch needs a queue).
2. Audit every concrete `Observer` impl (`NoopObserver`, `LogObserver`, the prod observer at L535, gateway_run_events, run_trace_store) and confirm each `recordEvent` path is reentrant. The mutex-holding one is fine; `LogObserver` is fine if it never touches shared mutable buffers; `gateway_run_events` and `run_trace_store` need spot-check.
3. If any impl is single-threaded by assumption, either fix it OR have `persistSessionCheckpointAsync` snapshot the observer's records to a thread-local buffer and have the next `agent.turn()` drain them (more code; only do this if audit finds a real un-fixable impl).

Severity rationale: HIGH not BLOCKER because in practice the `LogObserver` path is what battery runs use (no OTLP), and `std.log` IS thread-safe; the prod observer is mutex-protected. But the patch closes the door on any future single-threaded Observer impl without flagging it.

### H-02: `recycleSessionInPlace` deinits a "previous_agent" without coordinating with the just-fired `persistSessionCheckpoint("ttl_recycle")`

**File:** `src/session.zig:501-511`

The TTL recycle path:
```zig
session.agent.persistSessionCheckpoint("ttl_recycle");   // SYNC — fine
var previous_agent = session.agent;
session.agent = replacement_agent;
previous_agent.deinit();                                  // waits 30s for prev async
```

`persistSessionCheckpoint` at the comment-claimed call site is **sync** (line 1637-1639), so the TTL recycle is fine for *its own* call. BUT — if a hot-path `compaction:auto` or `summary_seed:auto` had fired async on `session.agent` minutes earlier and is *still in flight* (e.g. provider hanging), then `previous_agent.deinit()` will block up to 30s on `waitForLifecycleIdle`. That's a TTL-eviction stall, not a correctness bug — but the comment on the new deinit-wait describes it as a "shutdown path." This is *not* shutdown; it's a hot-loop reaping under live traffic. A 30s block on session recycle will cascade into the SessionManager's lock if recycling holds it.

Worse: if the worker DOES timeout-and-proceed (`log.warn "...may segfault"`), the worker on `previous_agent` then writes via `previous_agent.mem` — but `agent.mem` is typically a pointer into the *shared* `MemoryRuntime`, not owned by the Agent. So the worker survives the Agent free... and then dereferences `previous_agent` itself (e.g. `c.agent.history.items` inside `persistSessionCheckpointDetailed:1735`). **Use-after-free on `previous_agent.history`** is the failure mode.

**Fix recommendation (pick one):**
1. **Block TTL recycle on lifecycle drain (preferred for V1.14.10 A safety):** before `previous_agent.deinit()`, call `previous_agent.waitForLifecycleIdle(60_000)` and **never give up** — escalate to log.err and skip the recycle (return early, `session.agent` unchanged, replacement_agent gets deinit'd by errdefer). Recycling a session with a still-running worker is worse than letting a slow lifecycle finish.
2. Alternatively, **track the worker's `std.Thread`** (don't detach), and have `waitForLifecycleIdle` actually `thread.join()` it. Joining is unconditionally safe; detaching is a foot-gun.

### H-03: Spawn-failure fallback re-introduces the exact bug the patch fixes — *under the exact conditions that caused the bug*

**File:** `src/agent/commands.zig:1702-1708`

Under provider latency (50-100 concurrent in-flight LLM calls across a battery), pthread allocation pressure climbs. Default macOS pthread limit is ~2000-4096; Linux default is `/proc/sys/kernel/threads-max` (system-wide, often 4k-30k) but RLIMIT_NPROC per-uid is 2-8k typical. With multiple agents firing lifecycle workers, you can hit the limit. When you do, `std.Thread.spawn` returns `error.ThreadQuotaExceeded` (or similar), the fallback runs `persistSessionCheckpointDetailed` **synchronously on the agent.turn() thread** — the *exact* 30-180s block this patch was written to avoid.

This isn't just theoretical: the patch was triggered by 9-of-100 timeouts under load. The fallback turns those into "tail timeouts hidden in operator log warnings" instead of a clean bench failure that everyone notices.

**Fix recommendation:**
1. Add a `lifecycle_async_spawn_failures: std.atomic.Value(u64)` Agent counter (or runtime-level) and surface it in `/doctor`.
2. Either drop the fallback entirely (just log and skip — same as the in-flight skip case; "next legitimate trigger picks up"), or cap the fallback's elapsed time with a flag-able timeout (`memory.lifecycle.fallback_sync_timeout_ms`, default 5_000 ms; bail with a log.warn after that).
3. Long-term: pool the lifecycle workers (a small global `Thread.Pool` shared across all agents, see also H-04). This is the right fix to point 4 in your prompt.

Skipping the work entirely on spawn-failure is *safer than running it sync*, because the underlying disease (50-100 LLM calls per session) is itself what's exhausting thread budget. Doing them inline only adds heat.

## MEDIUM

### M-01: Atomic ordering is over-strong but not incorrect

**File:** `src/agent/commands.zig:1665` (`swap(true, .acquire)`), `src/agent/root.zig:890` (`load(.acquire)`), worker `store(false, .release)` at `commands.zig:1694`.

You asked about `acquire`/`release` correctness. Verdict: **correct, but slightly over-spec'd** on the swap.

- `swap(true, .acquire)` — `acquire` ordering on a CAS-like is only meaningful for the *load* side. Since you're going to spawn-and-publish (write to `ctx.*`, then `thread.spawn`), what you actually need is **release** ordering on the post-spawn side so the spawned thread sees the populated `ctx`. `std.Thread.spawn` itself provides a synchronizes-with edge (it's a synchronization primitive), so that's fine. The `.acquire` here gives you nothing extra, but doesn't cost anything either.
- `load(.acquire)` in `waitForLifecycleIdle` paired with `store(false, .release)` in the worker — this is the **correct pair**. The `release` store happens-before the `acquire` load, so once you observe `false`, you have the worker's full memory effect (the `c.allocator.free(c.reason_owned)` and `c.allocator.destroy(c)` happen-before the loader returns).
- One subtle point: the **deinit** path uses `waitForLifecycleIdle` which sees `false` and then immediately frees Agent state. The `release` ordering guarantees the worker's last allocator calls are visible. **Good.**

**Fix recommendation:** Optional — downgrade `swap(true, .acquire)` to `swap(true, .monotonic)` (the spawn itself is the sync primitive). No-op semantically, slightly more honest about what you're depending on. Leave the worker's `store(false, .release)` and the waiter's `load(.acquire)` — those are load-bearing.

### M-02: 30s deinit timeout is unbounded under provider hang AND can fire in production hot-loop teardown

**File:** `src/agent/root.zig:906-908`, plus `src/session.zig:511`, `src/session.zig:93`.

The comment ("worker may segfault but at least the shutdown path is bounded") implies `Agent.deinit` is a shutdown event. It isn't always — `Session.deinit` and `recycleSessionInPlace` both call it in normal hot-loop operation. A 30s block on a TTL recycle path can cascade into the SessionManager's iteration locking (see H-02). On a real shutdown (process tear-down), 30s is also too long if the gateway is being SIGKILL'd by an orchestrator with a 10s grace.

**Fix recommendation:**
1. Make the timeout configurable (Agent field, default 30_000 ms) and have the *session recycle* path pass a smaller bound (5_000 ms) and a *fail-on-timeout* policy that skips recycling rather than UAF'ing.
2. Document on `Agent.deinit` that callers in hot paths MUST drain via `waitForLifecycleIdle` themselves first.

### M-03: "next legitimate trigger picks up what was missed" claim is not always true

**File:** `src/agent/commands.zig:1651-1655` (doc), `src/agent/root.zig:4985-5024` (hot-path callsites).

The doc-comment promises that dropped triggers don't matter because lifecycle re-fires on every Pass C / summary_seed. This is true for `summary_seed:auto` (fires every turn with no summary yet, per the inline comment), and broadly true for `compaction:auto` (fires every compaction). But **the durable_continuity_refreshed flag is set optimistically to `true` even when the trigger was dropped**:

```zig
commands.persistSessionCheckpointAsync(self, "compaction:auto");
self.last_turn_context.durable_continuity_refreshed = true;
```

If the in-flight guard was tripped (silent skip), `durable_continuity_refreshed=true` is a **lie** — the previous worker is still running with the OLD compaction's data; the new compaction's checkpoint is never written. Downstream consumers (gateway turn events, anything that reads `durable_continuity_refreshed`) get a false-positive.

**Fix recommendation:**
1. Change `persistSessionCheckpointAsync` to return `bool` (true=spawned-or-fell-back, false=skipped-because-in-flight).
2. In `refreshDurableContinuityAfterCompaction`, set `durable_continuity_refreshed` from that return value. Currently it's a write-truth lie when the guard fires.

### M-04: `Ctx` is a closure-typed local struct → type-monomorphization on every callsite

**File:** `src/agent/commands.zig:1671-1675`

Because `self: anytype`, `Ctx` is monomorphized per-callsite based on `@TypeOf(self)`. There are 2 callsites today (`compaction:auto`, `summary_seed:auto`), both with `self: *Agent`. So **only one `Ctx` type is actually generated** today; no real cost. But this is a paper-cut waiting to happen if anyone ever calls `persistSessionCheckpointAsync` with a non-`*Agent` (e.g. a test mock).

**Fix recommendation:** Either (a) constrain the function to `self: *Agent` and stop using `anytype` (consistent with the patch's added Agent-typed `waitForLifecycleIdle`), or (b) leave a comment noting the monomorphization is intentional. The whole `commands.zig` file uses `anytype` pervasively for a historical reason; if the codebase is migrating off it, this is a fine place to start. **Recommend (a)** since the function now demands `lifecycle_in_flight` and other concrete Agent fields anyway.

### L-02: Spin-wait at 10ms granularity wastes ~3 syscalls/sec while idle

**File:** `src/agent/root.zig:888-895`

`std.Thread.sleep(10ms)` loop is fine for the deinit case (rare, bounded). It's slightly wasteful for tests that call `waitForLifecycleIdle(5_000)` and complete in <100ms. Negligible; flagging only because a future contributor will be tempted to drop it to 1ms.

**Fix recommendation:** Either leave as-is with a comment ("10ms is intentional: tests complete in <100ms typical, prod-deinit is rare"), or migrate to a `std.Thread.ResetEvent` (`event.wait()` in waiter, `event.set()` in worker) — zero CPU, zero polling, deterministic. The ResetEvent approach is genuinely better and only ~5 LOC more.

### L-03: Tests' `waitForLifecycleIdle(5_000)` may be insufficient on a loaded CI

**File:** `src/agent/root.zig:7989, 8081, 8236`

5s is generous for a test that does a single LLM call (these tests mock the provider, so the worker should complete in <50ms). But CI on a hot ARM mac can have arbitrary GC pauses, and `extraction_persist.persistExtracted` can fan out to LLM calls even in mocked tests if a mock provider isn't wired all the way down. **If these tests ever go flaky, 5s is the first dial to turn.**

**Fix recommendation:** Bump to 30s in tests. There's no downside (only fires on actual completion); the upside is robustness. Or hoist a `TEST_LIFECYCLE_WAIT_MS` constant for easy tuning.

## Things explicitly OK (verified, no change needed)

- Atomic ordering on the wait path (release/acquire pair correct).

- The `defer` block ordering in the Worker.run (clears in-flight LAST is wrong — see below correction).

**Wait — correction on the `defer` ordering at `commands.zig:1693-1697`:**

```zig
defer {
    c.agent.lifecycle_in_flight.store(false, .release);  // (a)
    c.allocator.free(c.reason_owned);                     // (b)
    c.allocator.destroy(c);                               // (c)
}
```

Inside a single `defer` block, statements execute **top-to-bottom** (not reverse — they're regular statements, only the `defer` block itself is deferred). So this runs (a) → (b) → (c).

**This is a latent bug.** Releasing the in-flight flag at (a) BEFORE freeing `c.reason_owned` and `c` at (b)(c) means: a `waitForLifecycleIdle` that observes `false` immediately, AND a `Agent.deinit` that then proceeds, AND `Agent.deinit` tearing down `self.allocator` — can race with (b) and (c) which use `c.allocator` (== `self.allocator`). **Free-after-free / UAF on the allocator itself.**

**Fix:** swap the order — clear the in-flight flag LAST so the waiter only observes `false` after the allocator work is done:
```zig
defer {
    c.allocator.free(c.reason_owned);
    const allocator = c.allocator;
    const agent = c.agent;
    allocator.destroy(c);
    agent.lifecycle_in_flight.store(false, .release);
}
```

(Capture `agent` before `destroy(c)` since `c` is freed before we touch it for the atomic. Actually `c.agent` is an `*Agent` field — once `c` is destroyed, dereferencing `c.agent` is UB. Hence the capture.)

**Upgrading this to HIGH (H-04) post-write.** The original review missed this on first pass; flagging now.

## H-04 (added on re-read): Defer-block ordering races allocator teardown vs in-flight flag

**File:** `src/agent/commands.zig:1693-1697`

See "correction" block above. The flag is cleared BEFORE the allocator work completes. Under deinit-wait, the waiter can return as soon as the flag flips, and Agent teardown can race the worker's allocator calls. Fix as shown above (clear flag last, capture pointers before destroying `c`).

**Severity:** HIGH. This is the kind of bug that won't fire 99% of the time and will fire on one machine in production.
