## 2. Tables — schema + purpose

### `messages` — cold tier truth (immortal)
- Every turn, every word, persisted on write
- Columns: `user_id`, `session_id`, `role`, `content`, `created_at`
- Never mutated post-write. Source of truth for all retroactive reasoning.

### `sessions` — session lifecycle
- One row per session, identified by session_key (e.g. `agent:zaki-bot:user:1:main`)
- Tracks created_at, last_active, idle eviction, checkpoint state
- Foreign key target for `messages.session_id` and `memories.session_id`

### `memories` — canonical facts
- The dominant memory store. ~7000+ rows on Nova's account.
- Columns (~37 total): id, user_id, session_id, key, content, content_hash, memory_type, embedding (pgvector inline), metadata (JSONB), importance_score, confidence_score, access_count, **valid_to / invalid_at / expired_at / valid_at / reference_time** (bi-temporal), **subject / predicate / object_key** (typed-edge metadata), link_type, attribution, attributed_to, lemmatized, **is_latest** (supersede chain marker), parent_memory_id, source_session_id, source_snippet, seen_in_session_count, community_id, episodes (TEXT[]), source_channel, source_message_id
- Primary types via `memory_type`: `core` (immortal), `daily` (timeline summaries), `conversation` (autosaves)
- Key prefix conventions:
  - `extracted_<hex>` — typed extractions from LLM (V1.6+)
  - `compose:<hash>` — synthesized via `compose_memory` tool
  - `durable_fact/<ts>/<idx>` — session-end facts
  - `autosave_<role>_<nano>` — per-turn captures
  - `session_checkpoint_<ts>` — resume artifacts
  - `compaction_summary/<session>/<ts>` — Pass C archives
  - `summary_latest` / `timeline_summary/*` — lifecycle summaries
  - `dream_log/<date>` — V1.13 dream cycle output
  - `last_hygiene_at` / `timeline_index/current` — bookkeeping sentinels

### `memory_entities` — canonical entity nodes
- One row per unique entity (Person, Org, Project, Place, etc.)
- Columns: id, user_id, name, name_lower, entity_type, name_embedding (vector(1024)), linked_memory_ids (TEXT[]), created_at, updated_at
- ivfflat index on `name_embedding` for cosine coreference (Mem0 0.85 threshold)
- UNIQUE on (user_id, name_lower) — case-insensitive coreference

### `memory_edges` — typed predicate graph
- Connects nodes (entities or memory keys) via typed predicates
- V1.6 columns: id, user_id, source_key, target_key, predicate, attribution, confidence, weight, valid_from, valid_to, invalid_at, expired_at, is_latest
- **V1.14 NEW columns:** `fact` (TEXT), `temporal_anchor_unix` (BIGINT), `episodes` (TEXT[])
- UNIQUE INDEX on (user_id, source_key, predicate, target_key) WHERE is_latest — re-mentions bump weight, no duplicates
- Cascade-on-close: `setMemoryInvalidation` closes edges when source/target memory archived

### `memory_events` — audit trail
- Every edge add/close gets a row. Bi-temporal graph history.
- Columns: id, user_id, memory_id, event_type, payload (JSONB), created_at
- Currently the LARGEST table (191 MB) — heavy churn from session writes

### `memory_communities` — LPA clusters
- Output of label-propagation algorithm batch run
- Columns: community_id, label, size, last_recomputed_at
- Memberships joined via `memories.community_id`

### `memory_embeddings_e5_1024` — pgvector store
- 18 MB, ~9900 rows. Embedding for memory.content via Together intfloat/multilingual-e5-large-instruct.
- Used by `memory_recall` hybrid search (vector + keyword fusion)

### `working_memory` — V1.13 Layer 0
- 15 hot slots per (user_id, session_id) — persists across turns within a session
- Columns: user_id, session_id, slot_id (0-14), slot_type, content, source_key, importance, pinned, created_at, last_touched_at
- Slot 0 reserved for identity (pinned at session-create via `pinIdentityFromUserState`)
- Eviction: composite priority = importance × recency_decay (1hr half-life) × type_weight

### `extraction_queue` — V1.13 async pipeline
- Brokers extraction work between agent turn loop (producer) and heartbeat thread (consumer)
- Columns: id (BIGSERIAL), user_id, session_id, job_type, payload (JSONB), enqueued_at, started_at, completed_at, status, attempts, last_error
- job_type: `wiki_link` | `session_end` | `backfill`
- Atomic claim via `WITH ... FOR UPDATE SKIP LOCKED` — concurrency-safe across workers (per-cell pod context makes this rare today)

### `skill_executions` — V1.13 Layer 6
- One row per multi-tool turn at session-end
- Columns: id, user_id, session_id, skill_name, task_summary, steps_executed (JSONB), assumptions_made (JSONB), user_feedback, outcome_quality, created_at
- Capture-only today; recall block in prompt = V1.15 follow-up

### `channel_identity_bindings` — multi-channel identity
- Maps external channel accounts (Telegram, Slack, app) to canonical user_id
- Used during inbound message routing

## 3. Write paths — how memory grows

### Path A: Per-turn conversation persistence
```
User sends chat → /api/v1/chat/stream
  → SessionManager.processMessage()
    → INSERT INTO messages (role='user', content=...)
    → Agent.turnOutcome() — runs the LLM call
      → INSERT INTO messages (role='assistant', content=...)
      → optionally: agent calls memory_store tool
        → INSERT INTO memories (typed extraction route via extraction_persist)
```
**Always fires.** Even on filler turns, messages are persisted. memory_store fires only when agent decides a fact is worth durable.

### Path B: Per-3-turn entity_pipeline (V1.13)
```
turn_complete event in agent/root.zig
  → turns_since_extraction += 1
    → at 3, build recent_turn_text (12KB cap)
      → enqueue ExtractionJob{type: "wiki_link", payload: turn_text}
        → INSERT INTO extraction_queue
        ← agent turn returns immediately (~5ms)
heartbeat thread (every 1s):
  → claimNextExtractionJob → SELECT FOR UPDATE SKIP LOCKED
    → entity_pipeline.runOnTurn(turn_text)
      → LLM call: extract entity mentions (multilingual)
      → for each mention:
        → findEntityByCosine (0.85 threshold) → existing OR upsertEntity
      → for each entity pair: upsertMemoryEdgeRich(predicate=MENTIONS, fact="A co-occurred with B")
      → for each entity: upsertMemoryEdgeRich(speaker→entity, predicate=MENTIONED, fact="user mentioned X")
    → markExtractionJobDone OR markExtractionJobFailed (retry up to 3x)
```
**Async.** User-facing turn never waits.

### Path C: Session-end consolidation
```
Idle eviction in SessionManager.evictIdle (default 30 min)
  → commands.timelineSummarize()
    → LLM call with summarizer_system prompt
      → produces: bullet summary + JSON ExtractedMemory array
    → for each ExtractedMemory:
      → extraction_persist.persistExtracted (the unified write path)
        → step 1: predicate blacklist (REJECTED_PREDICATES)
        → step 2: MD5 content_hash dedup
        → step 3: cosine similarity dedup (>0.92)
        → step 4: contradiction judge LLM call (V1.8-1)
          → if conflict: setMemoryInvalidation on existing → cascades to memory_edges
        → step 5: derive entity key (cosine match OR new entity)
        → step 6: upsertMemoryWithMetadata (memories row)
        → step 7: upsertMemoryEdgeRich (V1.14 — fact + episode_key + anchor)
        → step 8: syncVectorAfterStore (vector index)
    → enqueueExtractionJob (job_type="session_end") — for further async processing
    → working_memory.captureSession (Layer 6 procedural trace if ≥5 tool calls)
    → setMemoryInvalidation on superseded facts (cascade)
```
**Heavy LLM work.** Multiple LLM calls per session_end. Session-mutex held during eviction.

### Path D: Working memory promotion (V1.13)
```
extraction_persist.persistExtracted detects predicate match in predicateToSlotType:
  open_loop ← TODO|WILL_DO|REMINDS_ME_TO|NEEDS_TO|PROMISED|OPEN_LOOP
  active_goal ← WORKING_ON|BUILDING|GOAL|FOCUSING_ON|WANTS_TO_FINISH
  decision ← DECIDED|CHOSE|PICKED
  emotional ← FEELS|MENTAL_STATE|STRESSED_ABOUT
  open_question ← ASKING|WONDERING
  temporal ← HAPPENS_ON|SCHEDULED_FOR|BIRTHDAY
→ working_memory.promoteSlot
  → pickSlotForWrite (eviction by composite priority)
  → upsertWorkingMemorySlot
```

### Path E: Identity pin at session-create (V1.13 follow-up)
```
SessionManager.getOrCreateInternal — first message in a new session
  → buildSessionAgent (wires extraction context)
  → working_memory.pinIdentityFromUserState
    → state_mgr.listIdentityFacts (top 8 identity-class memory rows)
    → bundle as bullet list, max 1KB
    → upsertWorkingMemorySlot(slot_id=0, slot_type="identity", pinned=true)
```

### Path F: Heartbeat-driven cron jobs
```
daemon.heartbeatThread tick:
  → drain extraction_queue (max 2 jobs/tick × 30s = 60s worst case)
  → fire scheduled cron jobs from cron.json:
    → if job.command == "dream":
      → substitute dream_system_prompt (V1.13 Day 5.2)
      → run agent turn — agent uses memory_timeline + brain_graph + compose_memory
      → agent writes memories key="dream_log/<date>"
    → other cron commands → user-defined prompts
```

## 4. Read paths — how the agent retrieves

### Read 1: Memory recall (hybrid search)
```
agent/memory_loader.loadTurnMemorySlot
  → loadContextWithRuntimeDetailed (when mem_rt present)
    → vector recall via pgvector cosine
    → keyword recall via tsvector lemmatized
    → score fusion (weighted)
    → V1.10-A supersede filter (drop superseded rows)
    → V1.11-iter11 unverified-history elision
    → identity pin via buildActiveIdentityBlock (UNLESS skip_legacy_identity)
    → graph_neighbors block (1-hop expansion of recall seeds)
    → community block (top-3 communities)
  → wraps everything in <memory_for_turn>...</memory_for_turn>
  → caps at DEFAULT_MEMORY_CONTEXT_MAX_BYTES (64KB post-V1.13)
```

### Read 2: Working Memory render (V1.13)
```
agent/root.zig per-turn:
  → working_memory.loadForRender (top-10 slots by composite priority)
  → renderBlock → <working_memory>...</working_memory>
  → goes into PromptContext.working_memory_block
  → rendered in volatile prompt BEFORE <memory_for_turn>
```

### Read 3: Brain graph navigation (agent tool)
```
agent calls brain_graph(action="local_graph", center_key="X", depth=2)
  → state_mgr.findEdgesByKeys / listEdgesForUser
    → SELECT memory_edges WHERE source_key=X OR target_key=X (depth 1)
    → repeat for depth 2 if requested
  → return JSON: nodes + edges + their rich fields (V1.14: fact, episodes)
```

### Read 4: Brain page rendering
```
GET /api/v1/users/:id/brain/graph
  → handleBrainTypedEdges in gateway.zig
    → query memory_edges + memory_entities + memory_communities
    → apply hygiene filter (drop autosave_*, session_checkpoint_*, dream_log/* etc.)
    → compute importance via memory/importance.zig (recency × edge_count)
    → return graph JSON for cytoscape rendering
```

### Read 5: Memory timeline (chronological)
```
agent calls memory_timeline(window="today" | "yesterday" | "all")
  → state_mgr query: memories WHERE category='daily' ORDER BY created_at DESC
  → returns chronological summaries for the agent
```

### Read 6: brain_graph tool (sub-actions)
- `local_graph(center_key, depth)` — neighborhood
- `communities` — LPA cluster labels
- `orphans` — typed memories with no edges
- `diff(date)` — what changed on that date
- `find_path(a, b)` — shortest path between nodes (BFS)

## 5. Cross-cutting concerns

### Bi-temporal validity
- Every memory + edge has `valid_from`, `valid_to`, `invalid_at`, `expired_at`, `is_latest`
- V1.14 adds `temporal_anchor_unix` to edges — distinguishes when-fact-became-true from when-we-wrote-it
- `setMemoryInvalidation` closes a memory + cascades to all incident edges (sets is_latest=false)
- Reads default to `WHERE is_latest=true` — superseded data hidden but preserved for audit

### Supersede chains
- When a corrected fact lands, the contradiction judge sets `superseded_by` metadata
- Old fact's `valid_to` = correction's `valid_from`
- Can answer "what did I know on April 1?" by querying `valid_from < ts AND (valid_to IS NULL OR valid_to > ts)`

### Multilingual coreference
- Entity names embedded via Together intfloat/multilingual-e5-large-instruct
- Cosine 0.85 threshold matches Arabic/Spanish/Mandarin variants of the same entity
- Verified working on test fixtures; real-world cross-language merging unverified at scale

### Cache stability (Context v2)
- Stable system prompt = ~25-70 KB; cached by Together vLLM byte-prefix matching
- V1.13 re-ordered into Tier 1 (comptime) → Tier 2 (deploy-config) → Tier 3 (workspace) → Tier 4 (runtime)
- Workspace edits (MEMORY.md changes) invalidate only Tier 3+; Tier 1+2 stays cached

### Truncation caps (V1.13 lifted)
- Extractor input: 16KB (was 4KB)
- Recent-turn-text: 12KB (was 3KB)
- Session-end transcript: 12KB (was 3KB)
- Mentions per turn: 40 (was 24)
- Facts per session: 250 (was 100)
- Recall candidates: 25 (was 10)
- Memory context bytes: 64KB (was 32KB)

### Failure-soft contract
- Every async operation is best-effort: log + continue on failure
- LLM extraction failures → outcome=llm_failed, queue retry up to 3x
- Schema migrations idempotent (`IF NOT EXISTS`)
- pgvector / postgres unavailable → degrades to inline behavior, not crash

## 6. Honest gaps and risks

### G-01 (REAL CONCERN): Multi-tenant heartbeat stall in B2C shared runtime
- Default role = `.shared`, max 2048 users per daemon
- One slow user's extraction (worst case 30s × MAX_JOBS_PER_TICK=2 = 60s) stalls heartbeat for ALL users on that daemon
- Acceptable for hundreds; needs attention before thousands
- **Mitigation V1.15+:** spawn extraction work into goroutines (Zig threads) with own provider clients

### G-02: `temporal_anchor_unix` only resolves to date precision (ISO-8601 YYYY-MM-DD)
- The minimal parser drops time-of-day from the LLM's output
- Acceptable for fact-anchoring ("user joined Google in 2018"); insufficient for fine-grained event sequencing
- **Mitigation V1.15+:** full ISO-8601 parser

### G-03: Episode chain doesn't include entity_pipeline source memory keys
- entity_pipeline (per-3-turn wiki_link path) passes `episode_key=null`
- Only extraction_classifier path (session_end) populates the episodes array
- Provenance traceable for typed facts but not for co-occurrence edges
- **Mitigation V1.15:** plumb the recent-turn-text builder's source memory keys through

### G-04: Edge-distance scoring is naive
- Today: BFS depth + LPA community membership
- No PPR, no hop-decay weighting, no query-aware edge re-weighting
- Production frameworks (HippoRAG, MemORAI) use Personalized PageRank
- **Mitigation V1.15+:** implement hop-decay × weight-product scoring as a stepping stone (cheaper than PPR)

### G-05: working_memory slot 0 content quality depends on `listIdentityFacts`
- The filter pulls "identity-class" rows; can include extraction errors / test pollution
- Verified Nova's account has both real identity facts and old "Agent Zero" test data
- Production users won't have test pollution but DO need a way to manually override slot 0
- **Mitigation V1.15:** admin CLI `nullalis admin pin-identity --user-id N --content "..."`

### G-06: `compaction_summary/*` memories filtered from /brain
- Pass C summaries land in memories table but `zaki_state.zig:5846` filter excludes them from /brain queries
- Same for `dream_log/*` (currently invisible too — would need explicit rendering)
- **Mitigation V1.15:** dedicated /brain/dreams + /brain/summaries rails on FE

### G-07: skill_executions never read back yet
- V1.13 ships capture; recall block in prompt = V1.15 work
- Currently the table accumulates but agent doesn't see prior runs
- **Acceptable for V1 ship** — capture without recall is half the loop, but the half that matters first

### G-08: Working memory and active_identity duplicate
- Pre-V1.13.4: identity rendered TWICE (once in WM slot 0, once in <active_identity>)
- V1.13.4 added `skip_legacy_identity` flag gated on 4 conditions; tested and shipping safely
- But: gate doesn't validate WM slot 0 ACTUALLY has identity-quality content
- **Risk:** if `pinIdentityFromUserState` returns 0 facts (truly empty user) AND skip_legacy_identity=true, agent has no identity
- Currently safe because: empty bundle → pinIdentitySlot not called → wm_block stays empty → render skipped → fallback to active_identity NOT triggered (by current logic)
- **Bug potential V1.15:** verify the wm_block-empty fallback path

### G-09: extraction_queue retry semantics
- Job retries up to 3 attempts before terminal failure
- Successful job = mark done (immutable)
- Failed terminal = audit only (no agent action)
- **Gap:** no way for user/admin to manually re-queue a failed job
- **Mitigation V1.15:** admin CLI `nullalis admin extraction-retry --job-id N`

### G-10: memory_events table is HUGE (191 MB)
- Every edge add/close writes a row
- Heavy churn = heavy table growth
- Currently no archive/rotate mechanism
- **Mitigation V1.15:** monthly partition + retention policy (keep 90 days hot, archive older)

### G-11: pgvector ivfflat lists tuned for 5-50K rows per user
- Currently fine (Nova has ~9900 embeddings)
- Above 100K, ivfflat search degrades; consider HNSW or larger lists count
- **Mitigation V1.16+:** monitor vector recall p95; switch index type when degraded

### G-12: No PII scrubbing on conversation messages
- `messages` table stores everything verbatim, forever
- GDPR / right-to-be-forgotten requires manual delete (no tooling yet)
- **Mitigation V1.15:** admin CLI `nullalis admin scrub --user-id N --before-date D`

### G-13: Cross-language entity coreference unverified at scale
- mDeBERTa cosine on Together embeddings — fixtures pass, real conversations untested
- "Alfred" vs "ألفريد" merging via cosine — works for canonicalized names; might fail for nicknames or variant scripts
- **Mitigation V1.15+:** controlled multilingual battery once we have real data

## 7. Capacity planning

| Resource | Current (Nova's account) | At 1k DAU | Constraint |
|---|---|---|---|
| `messages` rows | ~10K | ~50K-500K/day | Postgres scales; partition by month at ~10M |

| Together API spend | ~$0.01-0.05/active session | $1-5/DAU/month at heavy use | Sidecar (Groq Llama) for extraction can cut cost 5-10× |
| Heartbeat thread stall | 60s worst case | Same per pod | Per-cell pod (B2B) eliminates; goroutines fix B2C |

## 8. Verdict

**The system is production-grade for V1 B2C launch with hundreds of users.** Beyond that, G-01 (heartbeat stall) and G-10 (memory_events growth) need attention.

**Not breaking concerns blocking V1:**
- Edge richness (V1.14) is correct architecture; remaining gaps (G-02 to G-04) are refinements
- Working memory works; G-05/G-08 are quality concerns not correctness
- Procedural memory captures; G-07 (no recall yet) is acceptable for ship
- Multi-tenant stall (G-01) is the realistic risk; mitigation is well-understood (goroutines)

**Real concerns I'd flag:**
- G-12 PII scrubbing — legal/compliance for B2C launch
- G-10 memory_events growth — operational risk after a few months
- G-13 multilingual coreference at scale — verify before claiming international support

**Confidence:** HIGH on architecture, HIGH on correctness, MEDIUM on operational scaling, MEDIUM on quality polish.

V1.14 is mergeable. The audit reveals refinements worth tracking for V1.15+ but nothing blocking.
