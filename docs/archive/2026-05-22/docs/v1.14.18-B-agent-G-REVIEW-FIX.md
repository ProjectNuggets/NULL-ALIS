## Notes / Honest Deviations from Plan

1. **Fix 1 scope expansion (small, justified):** the dispatch said only
   add a mutex to `push`/`last`/`deinit`. I also changed `recallRecent`'s
   signature from `?*const NarrationRingBuffer` to `?*NarrationRingBuffer`
   (and the corresponding `rb_ptr` cast in `context_engine.zig:667-668`)
   because `last` now takes `*NarrationRingBuffer` (the mutex lock requires
   a mutable handle). Without this signature change, the build would not
   compile. The `recallRecent` doc-comment notes that the caller-visible
   semantics remain read-only. This is the *minimum* additional surface
   needed for Fix 1 to compile.

2. **Fix 2 simplification per dispatch caveat:** the dispatch suggested an
   optional `std.debug.assert(@import("builtin").mode != .Debug or
   self.allocator.ptr != std.testing.failing_allocator.ptr)` line and
   noted "if awkward, use a comment-only approach instead". The
   `std.testing.failing_allocator` symbol is defined in the testing
   module and would pull testing-only symbols into release-mode compiles
   when imported at module scope. I took the documented fallback: log.warn
   on dupe failure + updated the field-level doc-comment on
   `Agent.narration_ring_buffer` to point at the new warn surface. The
   sentinel does NOT silently leak — it errors on every dupe, and now
   every error logs a clear `did you forget to call NarrationRingBuffer.init?`
   hint.

3. **Fix 3 cleaned `key_owned` flag entirely:** the dispatch said "delete
   the `errdefer if (key_owned)` line". The `var key_owned = true;`
   declaration + three `key_owned = false;` assignments exist *only* to
   support that dead errdefer's gate. Leaving the variable would create
   an unused-mutation warning at best and dead-code clutter at worst.
   I removed the entire `key_owned` machinery (decl + assignments) since
   the explicit `allocator.free(key)` at each catch site is what actually
   does the cleanup. Function behavior is byte-identical.

4. **Fix 5 test name:** matches the dispatch suggestion verbatim. Used
   `return error.RecentThoughtsBlockMissing` / etc. instead of
   `unreachable` for the `orelse` branch (per existing test idiom in
   the file — see `mem_pos = ... orelse unreachable` in the G-07 test).
   Switched to named errors to make any future failure mode easier to
   diagnose without re-running the test under a debugger. Semantically
   equivalent; behaviorally identical for the passing path.

5. **Fix 6: §14.11 Sub-Gate C grep still passes.** The gate scans for
   `narration.recallRecent` (literal). My edit changed only the third
   positional arg from `3` to `narration.RECALL_DEPTH`. The literal
   `narration.recallRecent` token remains at the same call site.

6. **Fix 7 line number drift:** Fix 7's comment edit inserted one extra
   line (changed wording slightly longer than the original), causing the
   `promotion.promoteWMToDurableAtSessionEnd` call to move from line
   1576 → 1577 in `commands.zig`. The §14.11 grep gate (literal
   `promotion.promoteWMToDurableAtSessionEnd`) still passes — only the
   line number changed.

7. **Fix 8 scope:** the dispatch said "If you can't find a misleading
   comment in commands.zig, just add the clarifying doc-comment in
   narration.zig. Skip the commands.zig edit." I found the misleading
   comment in `root.zig:2748` (not `commands.zig`) — "We seed
   `current_iteration` to 0 here", which is incorrect because the field
   is seeded from `self.iteration_counter` (session-monotonic). I fixed
   the comment at its actual location (`root.zig`) and added matching
   field-level doc-comments at both `Agent.iteration_counter` (root.zig)
   and `NarrationObserver.current_iteration` (narration.zig). No
   `commands.zig` edit was needed for this fix.

## Fixes NOT Applied

- **LOW #6 (Frame-ownership dangling-pointer hazard):** Per dispatch
  guidance, this is now covered by Fix 1's mutex (the race window that
  would let a push interleave with a read is closed by the mutex). The
  borrow contract is also documented at the top of `NarrationRingBuffer`
  in Fix 1's doc-comment ("the caller must finish reading the inner
  slices BEFORE the next push or deep-copy them"). Adding the deep-copy
  variant requested in the review would change the `last()` call
  interface and force every existing caller to opt in; not warranted for
  the current single-caller surface.
