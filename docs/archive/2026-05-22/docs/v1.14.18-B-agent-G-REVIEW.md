## Findings

### HIGH â€” Ring buffer push is not thread-safe (latent race)
**File:** `/Users/nova/Desktop/nullalis-agent-G/src/agent/narration.zig:85-115` (`NarrationRingBuffer.push`)
**Issue:** `push` mutates `self.len`, `self.head`, and `self.frames[...]` without a mutex or atomic, but the host `NarrationObserver` is stored on `*Agent` and is reached from worker threads:
- `src/agent/root.zig:2491` spawns `ParallelToolWorker` threads via `std.Thread.spawn`.
- Each worker calls `tools_mod.setToolObserver(&self.agent.observer)` (`src/agent/root.zig:1739`).
- `self.agent.observer` is the wrapped narration observer (`src/agent/root.zig:2756`).
- Worker-side tool code (e.g. `src/tools/composio.zig:31`) can call `getToolObserver().recordEvent(&...)` concurrently with the main thread also recording events.
- Today the only worker-emitted event variant is `.system_notice`, which falls into the `else => {}` branch in `NarrationObserver.recordEvent` (no `emitFrame`/`push`). So the race is latent, not active â€” but a future event added to that switch (or a future worker call site emitting `.tool_call_start`/`.turn_stage`) silently turns it into a real data race on `frames`/`len`/`head`.

**Impact:** No active corruption today, but a single innocent-looking event-type addition in narration.zig's switch becomes a hard-to-diagnose use-after-free / leak under `-Dengines=...` + parallel tools. Exactly the F3 class of "second-order memory bug."
**Reproduction (latent):** add `.foo_bar => |e| self.emitFrame(...)` in `narration.zig` switch and run any parallel-tool benchmark; ThreadSanitizer flags the race on `self.len`.
**Suggested fix:** (a) document the invariant at the top of `NarrationRingBuffer` ("push is not thread-safe; call only from the main agent thread"), and (b) either add a `std.Thread.Mutex` to the ring buffer or assert that `current_iteration` setter / push runs only when no workers are in flight. Belt-and-suspenders: a `std.debug.assert(@import("builtin").single_threaded or main_thread_id == current_thread_id())` in `push` would surface a violation in tests instantly.

### MEDIUM â€” `failing_allocator` sentinel silently no-ops in tests instead of being "loud"
**File:** `/Users/nova/Desktop/nullalis-agent-G/src/agent/root.zig:648`
```zig
narration_ring_buffer: narration.NarrationRingBuffer = .{ .allocator = std.testing.failing_allocator },
```
**Issue:** Agent G's commit documents this sentinel as "any missed initialization surfaces loudly via a leak rather than silently writing into an unrelated allocator's arena." In practice, `std.testing.failing_allocator` causes `allocator.dupe` to return `error.OutOfMemory`, and `push` catches that and returns silently (`narration.zig:88`). `deinit` then iterates `0..len` with `len == 0`, so nothing is freed. **No leak, no crash, no log line â€” and recallRecent will return an empty slice forever.**

That makes the "loudness" property false: a test that exercises `turnOutcome` without calling `narration.NarrationRingBuffer.init(self.allocator)` will pass, with all frames silently dropped. The 60+ direct `Agent{}` constructions in `src/agent/root.zig`, `src/agent/compaction.zig`, and `src/session.zig` (only `fromConfig` re-binds) all sit in this state.

**Impact:** Hides "narration recall isn't surfacing frames" bugs behind passing tests. Anyone debugging `<recent_thoughts>` not appearing in production where `fromConfig` IS used will be lulled by the test green.

**Suggested fix:** either (a) log a warn from `push` when the allocator is `failing_allocator` (cheap pointer compare against a sentinel singleton), OR (b) drop the sentinel and require constructors to set the allocator explicitly (compile-time error via removing the default). Option (b) is more in spirit with "loud surface."

### MEDIUM â€” Dead `errdefer` in `promotion.zig` is misleading
**File:** `/Users/nova/Desktop/nullalis-agent-G/src/agent/promotion.zig:141-144, 174-175`
**Issue:** `promoteWMToDurableAtSessionEnd` returns `PromotionResult`, NOT `!PromotionResult`. Therefore `errdefer` can never fire â€” there's no error-return path. Yet the function registers two such errdefers:
```zig
errdefer {
    for (collected.items) |*p| p.deinit(allocator);
    collected.deinit(allocator);
}
...
var key_owned = true;
errdefer if (key_owned) allocator.free(key);
```
Both are dead code. The actual cleanup happens via explicit `allocator.free(key)` in each `catch |err| { ... continue }` path. The cleanup is correct in practice (no leak observed), but the `errdefer` patterns mislead future readers into thinking these cover error paths that don't exist.

**Impact:** Code-review confusion; future refactors might assume errdefer coverage that isn't actually there and remove the explicit cleanup.

**Suggested fix:** either change the return type to `!PromotionResult` and propagate genuine errors (more disciplined), or delete the dead `errdefer` blocks and add a comment "This function is failure-soft; cleanup is explicit at every catch site."

### MEDIUM â€” `empty` PromotionResult allocates upfront even on success
**File:** `/Users/nova/Desktop/nullalis-agent-G/src/agent/promotion.zig:117`
```zig
const empty = PromotionResult{ .promoted = allocator.alloc(PromotedSlot, 0) catch &.{} };
```
**Issue:** `empty` is computed unconditionally at function entry but only returned on early-failure paths. On the success path, `empty.promoted` (the 0-length allocation) is never returned and never freed. In std.zig's GPA, `alloc(T, 0)` returns a zero-sized slice that is not flagged as a leak â€” but this is allocator-defined behavior and the code reads as "may leak."

**Impact:** Tests pass with std.testing.allocator, but the code is brittle to allocator changes (e.g., a strict leak-checking allocator that bookkeeps zero-sized allocs would flag this).

**Suggested fix:** lazy-construct `empty` inside each early-return arm, e.g.:
```zig
const smgr = state_mgr orelse {
    log.info(...);
    return .{ .promoted = allocator.alloc(PromotedSlot, 0) catch &.{} };
};
```

### LOW â€” Recalled frames borrow ring-buffer-owned strings (dangling-pointer hazard if ring buffer is mutated mid-use)
**File:** `/Users/nova/Desktop/nullalis-agent-G/src/agent/narration.zig:117-153` (`last` doc-comment + body)
**Issue:** `last()` returns a heap-allocated slice of `RecordedFrame` STRUCTS whose `message`/`tool_name` slice fields still point at memory owned by the ring buffer. The doc-comment is clear: "do NOT free the inner strings." But there's no guard against a `push` (which can deinit the slot at `self.head`) executing between `last()` returning and the caller finishing with the frames. In single-threaded `context_engine.assemble`, no push occurs in the window between `recallRecent` and `renderRecentThoughtsBlock`, so the bug is latent. Combined with the HIGH-severity item above, parallel-tool dispatch is the realistic trigger.
**Impact:** Latent; today there is no execution path that interleaves push + read of the same buffer.
**Suggested fix:** either (a) have `last()` deep-copy the inner strings into a caller-owned slice (with an explicit `deinitRecalled` helper), OR (b) document at the call site that the ring buffer must not be pushed-to between recall and render. (a) is safer; (b) is cheaper.

### LOW â€” `iteration_counter` does not reset per-turn, monotonically increasing across an entire session
**File:** `/Users/nova/Desktop/nullalis-agent-G/src/agent/root.zig:3179` (`self.iteration_counter +%= 1`)
**Issue:** `iteration_counter` is initialized to 0 and bumped per ReAct iteration inside `turnOutcome` (line 3179). It is NOT reset between turns. So turn 2 of a long session starts at e.g. iter=17, turn 3 at iter=33, etc. The `<recent_thoughts iteration="N">` attribute therefore shows session-wide iteration numbers, not turn-local ones. This is consistent with the dispatch's wording ("Bumped per ReAct iteration") and may be intentional â€” but the `commands.zig` comment claims "We seed `current_iteration` to 0 here" (commands.zig:1574 area) which is misleading because `self.iteration_counter` may be non-zero at turn start. The `+%=` wrap on a `u32` is safe in practice (4B iterations between wraps).
**Impact:** Cosmetic â€” the displayed iteration numbers in `<recent_thoughts>` are surprising to users on long sessions. No safety issue.
**Suggested fix:** either reset `iteration_counter = 0` at the top of `turnOutcome`, OR clarify the comment in commands.zig (and the narration.zig doc) that iteration is session-monotonic, not turn-local.

### INFO â€” Iteration counter `+%=` is wrapping, not saturating (dispatch wording was imprecise)
**File:** `/Users/nova/Desktop/nullalis-agent-G/src/agent/root.zig:3179`
**Observation:** The dispatch called `+%=` "saturating wrap." In Zig, `+%=` is **wrapping** addition (overflow â†’ 0), not saturating (overflow â†’ `maxInt`). ## Strengths
