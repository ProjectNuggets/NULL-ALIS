# V1.8 Memory Audit Protocol — behavioral truth, layer by layer

**Phase 1 of the V1.8 memory pyramid sprint.** Talk to the agent. Watch what
fires at every layer. Code can be true and can be not true; behavior is truth.

---

## Goal

For each layer L0-L4 of the [memory pyramid](v1.7-pause-snapshot.md#v18-reframe—why-this-pause-is-the-right-move),
empirically verify what writes, what reads, what fires, what skips. Produce a
per-prompt truth table that the design doc (Phase 3) consumes.

The audit answers questions code-review CAN'T:
- Does Pass C extraction actually fire on real conversations?
- When the LLM emits the ===EXTRACTED=== JSON tail, do edges land in `memory_edges`?
- Does the lifecycle summarizer respect or violate its 15-min cooldown?
- When a triple-shaped fact is told to the agent, does memory_store route through
  the unified extraction path or the content-only path?
- For superseded facts, does the contradiction judge fire AND close the old row?
- For long sessions, does the agent's recall correctly find facts from 50 turns ago?

## Setup

### 1. Fresh tenant — isolated from polluted user_id=1/42

Use `user_id=999` (or any unused integer). Provision via:
```sql
INSERT INTO zaki_bot.users (user_id, workspace_dir) VALUES (999, '/tmp/audit-user-999/workspace');
```

OR use the gateway's `provisionUser` path (cleaner — exercises the same flow as production).

### 2. Force compaction Pass C to fire — reduce token_limit override

Without this, Kimi K2.5 (256K context) means we'd need 230K of conversation to
trigger Pass C. Audit can't run that long.

**Two options**:

**Option A (recommended)** — config override `token_limit_override = 16000` so
compaction triggers at:
- Pass A: 11.2K tokens (~70%)
- Pass C: 14.4K tokens (~90%)

A few long-prompt turns will trigger. Model behavior unchanged (still Kimi K2.5).

**Option B** — switch model to `meta-llama/Llama-3.1-8B-Instruct-Turbo` (128K real
context) + `token_limit_override = 16000`. Cheaper per call. **BUT** older Llamas
may emit the ===EXTRACTED=== JSON tail less reliably than Kimi — the audit would
then conflate "Pass C extraction broken" with "8B model can't follow structured
output." Stay with Kimi to isolate the variable.

**Decision**: **Option A** — override token_limit, keep Kimi K2.5. If audit reveals
that Pass C wiring is broken AT ALL, we already know it's not a model issue.

### 4. Snapshot harness — `scripts/v1.8-audit-snapshot.sh`

For each turn N, capture:

```sql
-- A. Conversation L0
SELECT 'messages' AS layer, COUNT(*) AS count
FROM zaki_bot.messages WHERE user_id = 999;

-- B. Atomic facts L2 + bi-temporal state
SELECT 'memories.live' AS layer, COUNT(*)
FROM zaki_bot.memories WHERE user_id = 999
  AND (valid_to IS NULL OR valid_to > EXTRACT(EPOCH FROM NOW())::bigint);

SELECT 'memories.archived' AS layer, COUNT(*)
FROM zaki_bot.memories WHERE user_id = 999
  AND valid_to IS NOT NULL AND valid_to <= EXTRACT(EPOCH FROM NOW())::bigint;

SELECT 'memories.by_category' AS layer, memory_type, COUNT(*)
FROM zaki_bot.memories WHERE user_id = 999 GROUP BY memory_type;

SELECT 'memories.by_source' AS layer,
  COALESCE(metadata->>'attribution', 'none') AS source, COUNT(*)
FROM zaki_bot.memories WHERE user_id = 999 GROUP BY source;

-- C. Vectors L3
SELECT 'entities' AS layer, COUNT(*)
FROM zaki_bot.memory_entities WHERE user_id = 999;

-- D. Graph L4
SELECT 'edges.active' AS layer, COUNT(*)
FROM zaki_bot.memory_edges WHERE user_id = 999 AND is_latest;

SELECT 'edges.by_predicate' AS layer, predicate, COUNT(*)
FROM zaki_bot.memory_edges WHERE user_id = 999 AND is_latest
GROUP BY predicate;

SELECT 'edges.by_attribution' AS layer, attribution, COUNT(*)
FROM zaki_bot.memory_edges WHERE user_id = 999 AND is_latest
GROUP BY attribution;

-- E. Communities (L4 derived)
SELECT 'communities' AS layer, COUNT(*)
FROM zaki_bot.memory_communities WHERE user_id = 999;

-- F. Most-recent rows of each kind (top 5)
SELECT 'recent.memories' AS layer, key,
  LEFT(content, 80) AS content_head,
  memory_type, valid_to, metadata->>'attribution' AS source
FROM zaki_bot.memories WHERE user_id = 999
ORDER BY created_at DESC LIMIT 5;

SELECT 'recent.edges' AS layer, source_key, target_key, predicate,
  attribution, weight, valid_from
FROM zaki_bot.memory_edges WHERE user_id = 999 AND is_latest
ORDER BY id DESC LIMIT 5;
```

Output → JSON for diffing. After each prompt: compute the *delta* from previous
snapshot (new rows, supersessions, etc.) → that's the "what fired" report for
that turn.

## Audit prompt sequence

Each prompt designed to exercise a specific layer. We send sequentially, snapshot
after each, document what we observe.

### T1 — Identity write
**Prompt**: `"My name is Nova. I'm building a personal second-brain system called nullalis."`

**Expected (per design)**:
- L0: 1 user message + 1 assistant message in `messages`
- L2: at least 1 memory row (identity fact). Either via memory_store (if agent calls it) OR via post-reply lifecycle summarizer
- L3: vector for the new memory
- L4: edges if extraction happens → `(nova, name_is, nova)` and/or `(nova, builds, nullalis)`. If LLM-extracted: attribution = `extraction_classifier`.

**What we measure**: Does the agent call memory_store proactively? If not, does the lifecycle summarizer pick it up?

### T2 — Triple-shaped preference
**Prompt**: `"I prefer Helix over NeoVim for editing code. I switched last month."`

**Expected**:
- L2: memory row (preference fact)
- L4: edges `(nova, prefers, helix)` and `(nova, replaces_prior, neovim)` — IF extraction parses the triple from the LLM's JSON tail
- L4: link_type for "prefers" should map to `preference` (V1.7a-5 wiring)

**What we measure**: Triple extraction. Predicate-to-link_type mapping. THIS IS THE
KEY TEST FOR THE 0-EDGES BUG.

### T3 — Recall
**Prompt**: `"What editor do I prefer?"`

**Expected**:
- L0: 2 new messages
- Read: agent should retrieve T2's memory via memory_recall. Surfaces in agent's context as graph_neighbors block (if T2 wrote an edge) OR as direct recall (if T2 wrote a memory).

**What we measure**: Does retrieval work? What ranking signal returned the answer?

### T4 — Contradiction / supersession
**Prompt**: `"Actually, I switched back to NeoVim last week. Helix had too many bugs."`

**Expected**:
- L2: new memory row (latest preference)
- L4: contradiction judge fires (V1.6 cmt6) → closes T2's preferred-helix row by setting `valid_to`
- L4: new edge `(nova, prefers, neovim)`, OLD edge `(nova, prefers, helix)` gets `is_latest=false`

**What we measure**: Bi-temporal close-out. Judge reliability. Edge supersession.

### T5 — Multi-entity
**Prompt**: `"I work at HRS. My colleague Alex and I are working on the MNDA contract for a client called WidgetCo."`

**Expected**:
- L2: multiple memories
- L4: many edges: `(nova, works_at, hrs)`, `(alex, works_at, hrs)`, `(nova, colleague_of, alex)`, `(nova, working_on, mnda)`, `(mnda, for_client, widgetco)`, etc.
- L3: new entity vectors for HRS, Alex, MNDA, WidgetCo (with coreference at 0.95)

**What we measure**: Multi-entity extraction quality. Entity dedup.

### T6 — Relational query
**Prompt**: `"Who works at HRS?"`

**Expected**:
- Agent uses brain_graph tool with action=local_graph (per the V1.7-ship S2c nudge) OR memory_recall as fallback
- If brain_graph fires: returns the local subgraph centered on the HRS node

**What we measure**: Does the prompt-nudge route the agent correctly? Does brain_graph
return useful results given T5's edges?

### T7 — Temporal query
**Prompt**: `"What did I learn about today?"`

**Expected**:
- Agent uses brain_graph tool with action=diff, date=YYYY-MM-DD (today)
- Returns births in today's window

**What we measure**: Time-shaped query routing. /brain/diff coverage.

### T8 — Topic / cluster query
**Prompt**: `"What topics am I focused on?"`

**Expected**:
- Agent uses brain_graph tool with action=communities
- Returns: empty if /brain/communities/recompute hasn't been called, OR clusters if it has

**What we measure**: communities-tool routing. Cold-start UX.

### T9 — Force compaction (long-context fill)
**Prompt**: 6-8 long prompts each ~3K chars (paste of varied content) until token
estimate > 14.4K (90% of 16K override).

**Expected**:
- compaction.auto: pressure_pct logs climb
- pass=A firing at 70%
- pass=C firing at 90%
- Pass C produces summary
- IF JSON tail emitted: extraction parses + writes edges

**What we measure**: THE CRITICAL ONE. Pass C firing + JSON emission + edge land.

### T10 — Force lifecycle summarizer (post-reply)
**Prompt**: any short prompt with `?reason=ttl_evict` simulated via session API
**OR**: trigger via 16-minute wait + new session creation.

**Expected**:
- lifecycle summarizer fires
- writes session_summary, summary_latest, timeline_summary
- IF triples in summary: writes durable_fact + edges

**What we measure**: Lifecycle summarizer reliability. Edge write half.

### T11 — Across-session recall
**Prompt** (NEW SESSION): `"What's my preferred editor?"`

**Expected**:
- Agent retrieves the LATEST preference (NeoVim per T4)
- Cross-session continuity via summary_latest + timeline_summary

**What we measure**: L5 episodic continuity. Does the agent remember across sessions?

### T12 — Forget + verify
**Prompt**: `"Forget that I work at HRS."`

**Expected**:
- Agent calls memory_forget OR memory_archive on the matching key
- L2: row supersedes (valid_to set)
- L4: cascade — edges from the archived memory key become orphaned (or get invalid_at)

**What we measure**: Forget path. Cascade-on-archive.

## Risks + mitigations

| Risk | Mitigation |
|---|---|
| Audit pollutes the test PG with 12+ prompts of fixture data | Use user_id=999 (unused). Drop the user at end via `DELETE FROM zaki_bot.users WHERE user_id=999;` cascading. |
| Probe logs change behavior (Heisenbug) | Probes are log-only, no logic change. Run with + without to verify identical state mutations. |
| Token_limit_override breaks something else | Override only for the audit env (separate config.json). Production config untouched. |
| Lifecycle summarizer cooldown blocks T10 timing | Either set cooldown to 0 in audit config OR use the simulated `ttl_evict` trigger reason (bypasses cooldown). |
| Kimi rate-limit on Together | 12 prompts × ~2 reqs each = 24 calls. Well under any rate limit. |

---

## What this audit does NOT cover (deliberate)

- Performance profiling — we measure CORRECTNESS, not latency. Profiling is a separate sweep.
- Multi-tenant isolation — covered by existing PG smoke tests.
- FE rendering — orthogonal; covered by `docs/v1.7-frontend-handoff.md`.
- Subagent / delegate paths — flag-gated off in default V1; not in this audit's scope.
