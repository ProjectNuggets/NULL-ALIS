## Section 4 — Discovered surprises (delta from design doc)

Things grep-verification found that differ from the design doc's assumptions:

1. **Pass C has its own JSON-tail prompt** at `src/agent/compaction.zig:887`, separate from `summarizer.zig:184`. V1.8-5 must touch BOTH locations. Design doc only cited the lifecycle one.

2. **summarizer module location** is `src/memory/lifecycle/summarizer.zig`, not `src/agent/summarizer.zig`. Design doc was imprecise.

3. **`syncVectorAfterStore` lives at `src/memory/root.zig:1840`**, not `src/agent/`. Already a robust API: handles durable outbox + no-provider no-op + batching. V1.8-2 is purely a wiring change, no API extension needed.

4. **13 callers of `syncVectorAfterStore` already exist** — including 2 in `tools/`, 2 in `compaction.zig`, 5 in `commands.zig`, 2 in `agent/root.zig`. The pattern is well-trodden. `extraction_persist.persistExtracted` is the conspicuous omission. Reinforces "wiring miss not architectural gap" verdict.

5. **`forgetMemory` exists in TWO places**: `src/zaki_state.zig:418` (no-op shim, likely markdown-engine stub) and `src/zaki_state.zig:4110` (real impl). V1.8-3 touches the real impl only; verify the shim doesn't need parallel update during Phase 6.

6. **`gateway.zig` is 25,827 lines.** Threading provider+model into `bindMemoryStoreUnifiedContext` may need ~10 LOC of variable hoisting if the symbols aren't already in scope at line 1562. Phase 6 confirms.

7. **`forgetMemory` does NOT use a transaction** today — bare DELETE. Adding edge UPDATE in V1.8-3 should wrap both in BEGIN/COMMIT for atomicity. ~5 LOC overhead.

8. **No background scheduler** in nullalis today (per [[project_lifecycle_investigation_2026_04_20]]). V1.8-10 daily summary needs one — confirms why it's queued, not must-ship.

## Section 6 — Sequencing recommendation for Phase 5/6

Build order (I recommend; Phase 5 locks):

1. **V1.8-7 first** (audit on Kimi). Why: confirms what Pass C reachability actually is. May surface unknowns that change V1.8-5 scope.
2. **V1.8-6 second** (eval suite, against `153da60` baseline). Why: gives every subsequent commit a measurable target. Discovers regressions early.
3. **V1.8-1** (judge wire — 1-line atomic). Why: smallest, safest, immediately unlocks edge close-outs which the eval suite will measure.
4. **V1.8-2** (vector coverage). Why: independent of others, fixes the most visible smoking gun (`vector_score=n/a`).
5. **V1.8-3** (forget cascade). Why: independent, surgical.
6. **V1.8-5** (tighten prompts). Why: should land BEFORE V1.8-4 so we measure baseline LLM compliance under tightened prompts before adding the deterministic floor underneath. Otherwise we can't tell whether V1.8-4 is masking a regression in V1.8-5.
7. **V1.8-4** (cheap deterministic — biggest single commit). Why: built ON TOP of fixed pipeline; the floor under improved tier 2b.
8. **V1.8-9** (identity profile). Why: depends on V1.8-1+2+3+4 producing reliable identity facts to materialize.

Each commit closes its target gap and the eval suite measures the F1 delta. Atomic commits per item.
