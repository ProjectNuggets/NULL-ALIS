# V1.8 Memory Research Synthesis

**Date:** 2026-05-04
**Author:** Claude research session for Nova
**Purpose:** Map the 2026 state of agent-memory architectures so the V1.8 sprint can pick from first principles. Code-truth over marketing; cite primary sources; distinguish "they have it" from "they have it well."
**Status:** Foundation document. No commitments here — the recommendations section is ranked but not yet roadmap.

---

## Executive Summary

1. **The field has converged on three primitives, not one.** Every serious 2026 system combines (a) extract-and-store atomic facts with vector retrieval, (b) a graph layer with bi-temporal validity, and (c) some form of self-editable in-context block. Nullalis already has all three primitives in code. The differentiator is now wiring quality, not architecture choice.

2. **The biggest *missing* primitive across the field is procedural memory.** Mem0, Zep, Letta, and Anthropic's memory tool all store facts and episodes well; almost nobody handles "remember the *rule* the user taught me" as a first-class type. LangMem alone names it explicitly, and even there it's bolted onto prompt-optimization rather than agent behavior. Nullalis's typed-edge schema (preference / attribute / supersession / usage) is *natively* shaped to carry procedural memory — but the extractor isn't producing edges in the live system. **Closing the procedural-memory gap is V1.8's highest-leverage move.**

3. **Anthropic's Memory tool is deliberately minimalist — and that's a tell.** The shipping API is a 6-command file editor over a `/memories/` directory with no extraction pipeline, no graph, no vector store, no decay. The system prompt is a single paragraph: "VIEW YOUR MEMORY DIRECTORY BEFORE DOING ANYTHING ELSE." This is the father's design language: *trust the model + give it a substrate.* Nullalis's extract-and-store machinery is doing more work than Claude Code's memory does, which means our risk surface is larger and we should be paranoid about whether the extraction is actually adding value over a markdown directory.

4. **Sleep-time compute (Letta) is the most underused architectural pattern in the field.** Most systems treat memory writes as either hot-path (latency hit) or "background compaction" (vague). Letta's split — primary agent answers, sleep-time agent reasons over shared memory blocks — is the cleanest way to get *both* low p50 latency and high-quality memory. Nullalis's lifecycle summarizer already lives in this lane but is single-purpose; promoting it to a general "sleep-time worker" pattern is a near-term unlock.

5. **Production reality is uglier than the papers admit.** A community audit of Mem0 found 97.8% of stored memories were junk after 32 days of use ([Mem0 Issue #4573](https://github.com/mem0ai/mem0/issues/4573)). Mem0's fact extraction silently fails on non-OpenAI LLMs ([Issue #4540](https://github.com/mem0ai/mem0/issues/4540)). Our own observation that 0 typed edges have been extracted across 16K+ memories puts us in the same failure family. **Extraction reliability — not architecture — is the universal problem nobody has solved.** Nullalis can win here with adversarial evals + extraction observability, both cheaper than any new memory layer.

## Per-System Deep Dives

### 1. Mem0 — extract-and-store with LLM-as-classifier

**Repo:** [mem0ai/mem0](https://github.com/mem0ai/mem0) · Apache-2.0 · 30k stars
**Paper:** [arXiv:2504.19413](https://arxiv.org/html/2504.19413v1)

#### Architecture

Two-phase pipeline:

1. **Extraction phase.** Process message pair `(m_{t-1}, m_t)` with two contexts: a conversation summary `S` from the database plus the prior `m=10` messages. An LLM extracts salient memory candidates `Ω = {ω_1, ..., ω_n}`. Inference defaults to GPT-4o-mini.
2. **Update phase.** For each candidate `ω_i`, retrieve top-`s=10` semantically similar existing memories. The LLM then executes one of four operations via tool-call:
   - `ADD` — no semantic equivalent exists
   - `UPDATE` — augment with complementary information
   - `DELETE` — remove contradicted memory
   - `NOOP` — skip

Importantly, the paper is explicit that there is no separate classifier model — the LLM itself reasons about which operation to invoke. The "classifier" framing is post-hoc.

**Mem0g (graph variant)** runs a parallel pipeline:
- Entity extractor identifies nodes (people, locations, etc.)
- Relations generator infers labeled edges as triplets
- Conflict detector compares against existing edges; obsolete relationships are marked invalid (not deleted) for temporal reasoning
- Dual retrieval: entity-centric anchor search + semantic triplet matching

#### Key insights

- **No separate classifier is the design.** The "ADD/UPDATE/DELETE/NOOP" choice is a tool-use call, not a fine-tuned model. This is cheaper than it sounds: one LLM call per candidate fact. It's also more brittle than it sounds: if the model picks `NOOP` poorly, you silently lose facts. Audited junk rate: 97.8% over 32 days ([Issue #4573](https://github.com/mem0ai/mem0/issues/4573)).
- **No hierarchical user/session/agent levels in the *paper*.** This is a marketing/SDK feature, not an architectural one. The paper treats all memories uniformly; tiering is a metadata filter on top.
- **No automatic decay.** "LLM-based update resolver…marks relationships obsolete rather than physically removing them." Decay is implicit (newer facts override older retrievals) not explicit.
- **The 0.95 cosine threshold is *not* in the paper.** It's an SDK default for entity dedup. We adopted it; it's a reasonable starting point, not a derivation from first principles.
- **LOCOMO benchmark performance** (J-Score):
  - Single-Hop: Mem0 67.13, Mem0g 65.71 (Zep baseline 61.70)
  - Multi-Hop: Mem0 51.15, Mem0g 47.19 (LangMem 47.92)
  - Temporal: Mem0 55.51, Mem0g 58.13 (Mem0g wins; Zep wins overall on temporal sub-task per [Zep paper](https://arxiv.org/abs/2501.13956))
  - Open-Domain: Mem0 72.93, Mem0g 75.71 (Zep 76.60)
- **Latency claims hold up methodologically.** p95 search 0.20s vs. full-context baseline 17.12s. Token efficiency: ~1,800 vs. ~26,000 per conversation. The "91% faster / 90% cheaper" headline is real for the search phase. It excludes extraction overhead, which the paper does not benchmark on a per-turn basis.

#### What nullalis already has

- Mem0-style 0.95 cosine entity dedup (memory_entities table)
- ADD-only ingestion with later supersession (we kept the contradicted-but-not-deleted invariant Graphiti popularized)
- Reserved decision_type column for the future classifier emission

#### What we're missing worth borrowing

- **The actual `ADD/UPDATE/DELETE/NOOP` LLM tool-call.** We reserved the column. We don't fire it. This is V1.6 work that has slipped twice. **(Recommendation: S-tier).**
- **The `m=10` prior-messages + summary `S` extraction context.** Our memory_enrich uses RAG-style retrieval, not a sliding window of conversation context. The window matters for capturing "the user said X earlier in this turn" facts.
- **Per-candidate dedup retrieval (`s=10`) before write.** We do entity dedup at write time but not memory-text dedup. If two phrasings of the same fact arrive separately, we get two memories. Audit is overdue.
- **Mem0g's separation of entity extraction from relation extraction.** Our current extractor treats everything as one LLM call producing memory + edges + entities, which may explain the 0-edges-extracted bug — the model is overloaded with three goals per call.

#### What to reject

- The 97.8% junk rate is not a Mem0-specific bug, it's a general failure mode of "trust the LLM to pick NOOP." We should not adopt the NOOP class without an adversarial eval first.
- The "hierarchical user/session/agent" framing is metadata, not architecture. We already have this via session_id / user_id columns.

### 2. Letta (formerly MemGPT) — self-editing memory blocks + sleep-time compute

**Repo:** [letta-ai/letta](https://github.com/letta-ai/letta) · Apache-2.0 · 17k stars
**Origin paper:** MemGPT (Packer et al., 2023)

#### Architecture

**Three-tier memory hierarchy** (OS metaphor):
- **Core memory** — small block in the context window (RAM). Agent reads/writes directly via tools. Character-limited.
- **Recall memory** — searchable conversation history outside context (disk cache).
- **Archival memory** — long-term storage queried via tool calls (cold storage).

**Memory blocks** are the conceptual unit:
- Persisted individually in the DB with a `block_id`
- Have a label, value (string), size limit, optional description, and a read-only flag
- Compiled into the context window via a Jinja template at request time
- Shareable across multiple agents

**Tool surface** (the agent's actions on memory):
- `core_memory_replace(label, new_value)` — replace a core block
- `core_memory_append(label, content)` — extend a core block
- `memory_rethink(label, new_value)` — rewrite from scratch
- `archival_memory_insert(content)` — add to long-term store
- `archival_memory_search(query)` — semantic search
- `conversation_search(query)` and `conversation_search_date(date)` — recall search

**Sleep-time compute** (Letta 0.7.0+):
- Two-agent design. Primary agent answers conversations and *cannot* edit core memory blocks. Sleep-time agent reads/writes memory during idle periods.
- Primary agent reads core memory anytime — no waiting for the sleep-time agent to finish ("anytime" model).
- Different model per agent: e.g. fast gpt-4o-mini for primary, stronger Claude Sonnet for sleep-time.
- "Pareto improvement" in the [Letta paper](https://www.letta.com/blog/sleep-time-compute) — accuracy goes up without hurting p50 latency.
- Workload fit: excellent for math, coding, document synthesis, conversational reflection. Poor for real-time low-downtime tasks.

#### Key insights

- **Memory-as-tool, not memory-as-store, is the fundamental shift.** In Mem0, the runtime decides what to remember. In Letta, the *agent* decides — by emitting `core_memory_replace` mid-reasoning. This pushes intelligence into the model and reduces extraction failures, but assumes a strong-enough agent.
- **Sleep-time compute decouples memory quality from response latency.** This is the one architectural pattern in 2026 nobody else has wired well at scale.
- **The "core / recall / archival" tiering predates Mem0/Zep by ~2 years** and remains the cleanest mental model in the field. Nullalis's hot/warm/cold model is structurally identical.
- **The character-limit constraint on core blocks is load-bearing.** It forces the agent to compress, which is what makes the blocks high-signal. Without the limit, blocks bloat into junk.

#### What nullalis already has

- Hot/warm/cold tiering analogous to core/recall/archival
- Memory persistence with stable IDs
- A summarizer that runs post-reply on a 15-min cooldown (proto-sleep-time)
- `summary_latest`, `context_anchor_current`, `durable_fact/*` continuity blocks (analogous to core memory blocks)

#### What we're missing worth borrowing

- **Agent-callable memory-edit tools.** We have `memory_store` / `memory_edit` but the agent does not reliably call them mid-reasoning to maintain core blocks. The system prompt nudges from V1.7 are the start; named tools like `core_memory_replace("preferences", ...)` would be cleaner. **(Recommendation: A-tier — adopt after extraction reliability is fixed).**
- **Promote the lifecycle summarizer to a general sleep-time worker pattern.** Right now it does one job (summarize). It should be the substrate for: (a) the ADD/UPDATE/DELETE classifier, (b) typed-edge re-extraction over recent memories, (c) community re-naming, (d) staleness checks. **(Recommendation: S-tier).**
- **Hard character limits on warm continuity blocks.** Today `summary_latest` and `context_anchor_current` grow over time. A hard cap forces the summarizer to compress, which improves signal. Cheap fix.
- **Multi-agent memory sharing.** V2 territory but conceptually right for cell-pod multi-tenant story.

#### What to reject

- **The OS metaphor is overloaded.** "Like RAM / disk cache / cold storage" is fine pedagogy, but Letta's actual implementation has weirder behaviors (memory blocks are *both* in-context and DB-persisted, which is neither RAM nor disk). We should not adopt the metaphor as our naming.
- **Two-agent split for everything.** Letta's primary-agent-can't-edit-core rule is overcautious for our setup. We should let the agent edit core *and* run a sleep-time worker. They serve different purposes.

### 3. Anthropic Memory Tool + Claude Skills — minimalist file substrate

**Memory tool docs:** [platform.claude.com/docs/en/agents-and-tools/tool-use/memory-tool](https://platform.claude.com/docs/en/agents-and-tools/tool-use/memory-tool)
**Skills docs:** [platform.claude.com/docs/en/agents-and-tools/agent-skills/overview](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/overview)
**Released:** September 2025 (memory tool), October 2025 (Skills)

#### Architecture: Memory tool

A 6-command file editor scoped to `/memories/`. That's the entire architecture.

```
view      — list dir or read file with line numbers
create    — new file with text content
str_replace — exact-string replacement (errors if old_str matches multiple lines)
insert    — insert text at line N
delete    — remove file or directory recursively
rename    — move/rename, errors if destination exists
```

- **Client-side tool.** Anthropic does not store the files. Your application implements the backend (filesystem, database, encrypted store, anything).
- **Eligible for Zero Data Retention.** No model-side memory leakage.
- **Path traversal is the application's responsibility.** Docs are emphatic: validate every path is under `/memories/`.
- **Bootstrap prompt** (auto-included in system prompt when tool enabled):

  > IMPORTANT: ALWAYS VIEW YOUR MEMORY DIRECTORY BEFORE DOING ANYTHING ELSE.
  > MEMORY PROTOCOL:
  > 1. Use the `view` command of your `memory` tool to check for earlier progress.
  > 2. ... (work on the task) ...
  >    - As you make progress, record status / progress / thoughts etc in your memory.
  > ASSUME INTERRUPTION: Your context window might be reset at any moment, so you risk losing any progress that is not recorded in your memory directory.

- **No extraction. No graph. No vector store. No decay. No classifier.** The model is the entire memory pipeline.

#### Architecture: Claude Skills (progressive disclosure)

Three-level loading, all driven by SKILL.md files in a directory:

1. **Metadata** (~50–100 tokens per skill at startup) — name + description from YAML frontmatter is preloaded. Lets the model decide *if* a skill is relevant.
2. **Full instructions** (<5K tokens, lazy-loaded) — when the model decides the skill applies, it `bash cat`s the SKILL.md body into context.
3. **Bundled resources** — referenced files (FORMS.md, schemas, executables) are loaded only when SKILL.md says to.

Filesystem-native. Skills are folders, not API objects. Composable. Survive context resets because they're on disk.

#### Key insights

- **The shipped design is *deliberately* less than Mem0/Letta/Zep.** Anthropic could have shipped any architecture. They shipped a file editor. The bet is that *the model is smart enough to be its own memory pipeline*, given a substrate. The "Multi-session software development pattern" doc is explicit: bootstrap a progress.md, a checklist, a startup script — and let the model maintain them.
- **Progressive disclosure is the bigger architectural idea.** Skills' three-level loading (metadata → full → bundled) is what Memory tool is missing today. Imagine `/memories/INDEX.md` listing namespace summaries (~100 tokens) that the model loads first, then drills into specific files. **This is the move nullalis should mirror at the warm-tier injection layer.**
- **"ASSUME INTERRUPTION" framing is genius.** It reframes memory from "remembering things forever" to "leaving notes for your future self after a crash." This is the right mental model for context resets and a much cleaner contract than "the agent learns over time."
- **No classifier, no extractor, no decay** is a *position*, not an oversight. The position: the model is now smart enough that explicit memory machinery hurts more than helps. The 97.8% Mem0 junk rate is exhibit A.
- **Pairs with Compaction.** Anthropic explicitly recommends using Memory tool + server-side compaction together: compaction summarizes the active conversation, memory persists important facts across compaction boundaries. This is structurally what nullalis does today (compaction + lifecycle summarizer + memories table).

#### What nullalis already has

- All five Memory tool primitives via existing tools (`memory_store`, `memory_edit`, `memory_recall`, `memory_list`, `memory_timeline`)
- Compaction (Pass A + Pass C)
- Lifecycle summarizer
- Hot/warm/cold tiering

#### What we're missing worth borrowing

- **Progressive disclosure in warm-tier injection.** Today our warm tier is dumped wholesale (up to 4000 bytes). We should ship a metadata-first INDEX that the agent expands on demand. Compute: cheap. Token savings: large. **(Recommendation: S-tier).**
- **The "ASSUME INTERRUPTION" framing in the system prompt.** Replace whatever continuity framing we have with "your context might reset; leave notes for your future self." Concrete, action-shaped. **(Recommendation: S-tier — one-line system-prompt change).**
- **Bootstrapped startup files.** The "Multi-session software development pattern" — initializer session creates progress.md + checklist + startup script — maps to nullalis's `context_anchor_current` + `summary_latest` but is more deliberate. We should formalize this as an explicit bootstrap, not an implicit summarization.
- **Skill-style progressive disclosure of memory namespaces.** If a user has 16K memories across many topics, a metadata-only `MEMORIES_INDEX.md` listing topic clusters (with ~50-token summaries each) lets the agent decide which to load — *exactly* the skill-discovery pattern. Pairs naturally with our memory_communities. **(Recommendation: S-tier).**

#### What to reject

- **The "no extraction" position is wrong for nullalis.** Anthropic ships into the strongest models in the world. Nullalis runs through Together / Moonshot / others where the model isn't always Claude-grade. We need extraction *because* we don't always have a top-tier model on the hot path. But we should be paranoid about whether each piece of extraction machinery is paying its cost.
- **Pure file-based store is too thin for our domain.** Telegram-routed multi-cell architecture needs structured queries (which user, which thread, which time window). A markdown directory doesn't index that.

### 4. LangMem — explicit semantic / episodic / procedural taxonomy

**Repo:** [langchain-ai/langmem](https://github.com/langchain-ai/langmem) · MIT
**Conceptual guide:** [langchain-ai.github.io/langmem/concepts/conceptual_guide](https://langchain-ai.github.io/langmem/concepts/conceptual_guide/)
**Origin:** CoALA framework ([arXiv:2309.02427](https://arxiv.org/abs/2309.02427), Princeton 2023)

#### Architecture

LangMem's contribution is **naming the taxonomy**, not the implementation:

1. **Semantic memory — facts and knowledge.** Two representations:
   - *Collections* — unbounded searchable documents, vector-retrieved
   - *Profiles* — single task-specific document with strict schema, easy to read top-of-context
2. **Episodic memory — past experiences.** Successful interactions captured as learning examples with situation, reasoning, outcome.
3. **Procedural memory — system behavior / rules.** "Combination of model weights, agent code, and agent's prompt." Implemented via a **prompt optimizer**: feedback updates a system-prompt template over time.

**Two formation modes:**
- **Hot-path (conscious)** — extract during conversation, adds latency, captures critical context immediately
- **Background (subconscious)** — post-conversation reflection, no latency hit, improves recall

**Two API layers:**
- *Core API* — stateless memory managers + prompt optimizer (storage optional)
- *Stateful API* — store managers + memory tools persisted to LangGraph BaseStore

**Hierarchical namespacing** with template variables (`"{user_id}/preferences"`).

**Conflict resolution** is described conceptually as "memory enrichment process" balancing creation vs. consolidation; no algorithmic specifics in the docs.

#### Key insights

- **The semantic/episodic/procedural split is the cleanest taxonomy in the field.** Cognitive-science backed (CoALA, Princeton 2023), maps onto how humans actually use memory, and *exposes the procedural-memory gap* nobody else handles well.
- **Profiles vs Collections is the right primitive distinction.** A user has *one* preferences profile (latest state) and *many* episodic memories (timestamped events). We currently treat both the same way.
- **Procedural memory implemented as prompt optimization is clever but limited.** It works for "the user prefers concise answers" → modify prompt template. It doesn't work for "when the user asks about X, run tool Y first" — that's procedural but needs more than a prompt rewrite.
- **Hot-path vs background formation is the same insight as Letta's sleep-time, framed differently.** Letta builds it as a separate agent; LangMem builds it as a config flag on the memory manager. Same idea, different ergonomics.

#### What nullalis already has

- Semantic memory ≈ our memories table + memory_entities
- Episodic memory ≈ our session_summary / timeline_summary / autosaves
- Hierarchical namespacing via Postgres schema
- Background formation via lifecycle summarizer

#### What we're missing worth borrowing

- **Adopt the semantic/episodic/procedural taxonomy as our public vocabulary.** Right now our docs talk about hot/warm/cold (storage tier) and atomic facts vs entities vs edges (data shape). Layering the *use-type* taxonomy on top makes the system explainable to users and forces clarity about extraction targets. **(Recommendation: S-tier — vocabulary change, not code change).**
- **Profiles primitive.** A `user_profile` blob (latest state of preferences/identity) is a different shape than our atomic memories. Today we extract preferences as atomic facts and rely on retrieval to surface them. A single profile blob, agent-editable, would always be in warm context. Maps directly to a Letta core memory block. **(Recommendation: A-tier).**
- **Procedural memory as a first-class type.** Our typed-edge schema (preference / attribute / supersession / relationship / usage / synthesis / episode) is *ideally shaped* for this — `usage` edges are procedural by definition. **The question is why the extractor isn't producing them.** Diagnosing this is V1.8's central engineering question. **(Recommendation: S-tier).**

#### What to reject

- **LangMem's "storage is optional" stateless mode** is ill-suited for our needs. Postgres is the entire point.
- **Prompt optimization as the procedural memory mechanism** is too narrow for our cell-pod story; we need procedural memory that survives prompt changes.

### 5. Cognee — ontology-driven KG construction

**Repo:** [topoteretes/cognee](https://github.com/topoteretes/cognee) · Apache-2.0
**Docs:** [docs.cognee.ai](https://docs.cognee.ai/core-concepts/main-operations/cognify)

ECL pipeline (Extract, Cognify, Load) builds a KG with optional ontology validation. Open-source: manually defined ontologies; commercial tier has auto-generation.

**Notable pattern:** ontology validation matches each entity type against OWL Classes with 80% fuzzy-match cutoff, canonicalizes nodes to URI form, and uses BFS to attach ontology subgraph relationships. Two-layer dedup: content-hash at ingest (`add()`) + pipeline-status tracking at process (`cognify()`).

**For nullalis:** the URI-canonicalization pattern is interesting — we currently dedup entities by cosine similarity but don't canonicalize. Adopting URI-style canonical names (`person:nova`, `org:anthropic`) would make our typed-edge subjects more queryable. The OWL-class baggage we should skip; it's heavyweight for our scale. **(B-tier; revisit when entity count crosses 100K).**

### 6. Zep / Graphiti — bi-temporal KG (we already adopted this)

**Repo:** [getzep/graphiti](https://github.com/getzep/graphiti) · Apache-2.0 · 10k+ stars
**Paper:** [arXiv:2501.13956](https://arxiv.org/html/2501.13956v1)

We already covered this in [graph-memory-research.md](./graph-memory-research.md). Recap:
- Bi-temporal model: every edge carries `t_valid` (event time) and `t_invalid` (when superseded), plus ingestion time `T'`.
- LLM-driven contradiction detection: new edges compared to semantically related ones; conflicts invalidate (don't delete).
- Hybrid retrieval: semantic + BM25 + graph traversal; p95 300ms.
- Beats MemGPT on DMR (94.8% vs 93.4%) per the Zep paper. Beats Mem0 on temporal sub-task (63.8% vs 49.0%).

**For nullalis:** we have the bi-temporal columns. We do not yet have the LLM-driven contradiction comparison (only entity dedup at write). Worth adding in V1.8. **(Recommendation: A-tier).** The hybrid-retrieval pattern (semantic + BM25 + graph) is also worth checking against our current retrieval engine — we may be missing BM25.

### 7. ChatGPT Memory — the mass-market UX bar

**Docs:** [help.openai.com Memory FAQ](https://help.openai.com/en/articles/8590148-memory-faq) · [openai.com Memory and new controls](https://openai.com/index/memory-and-new-controls-for-chatgpt/)

Two systems running together:
- **Saved memories** — explicit "remember that…" requests, fully user-visible
- **Chat history references** — implicit insights ChatGPT extracts from past chats

**UX patterns worth absorbing:**
- "Memory updated" inline notification with a clickable "Manage memories" affordance
- Settings → Personalization → Manage memories → trash icon per entry
- Conversational deletion: "Forget that I mentioned X" works as a command
- Hard storage cap (~1,200–1,400 words total) — when full, no new memories until user deletes some
- EU/EEA: feature disabled pending AI Act review (compliance-driven feature gating)

**For nullalis:** the *transparency* affordances are the SOTA UX bar and they are not technical work, they are frontend work. Three concrete asks for the Zaki-Prod frontend team:
1. Inline "Memory updated" toast after assistant turn writes a memory, with link to manage
2. /memories page: list, search, edit, delete
3. Conversational "forget X" recognized as a command

The *hard cap* approach is interesting but probably wrong for our cold-memory-forever rule. We should cap *what enters warm* (already done implicitly), not *what enters cold*. **(Recommendation: S-tier on UX, frontend work, not backend).**

### 8. Microsoft GraphRAG — community detection at production scale

**Repo:** [microsoft/graphrag](https://github.com/microsoft/graphrag) · MIT
**Paper:** [arXiv:2404.16130](https://arxiv.org/abs/2404.16130) (From Local to Global)

Two-stage: build a KG from text via LLM extraction, then run hierarchical community detection (Leiden), summarize each community via LLM at multiple zoom levels.

**Query routing:**
- *Local Search* — entity-level questions; resolve to nodes, retrieve neighborhood
- *Global Search* — corpus-wide synthesis; query community summaries, aggregate via map-reduce
- *Hybrid* in 2026 production setups: vector + graph traversal + structured DB lookup, route by query type

**For nullalis:** we have community detection (LPA). We have the LLM namer (V1.7 S1). We have warm `<active_communities>` injection (V1.7 S2b). What we *don't* have:
- **Multi-level community hierarchy.** Currently flat. GraphRAG runs Leiden at multiple resolutions. Useful when communities get large enough that you want zoom levels.
- **Map-reduce global query mode.** Our brain_graph tool answers entity-anchored questions well; corpus-wide questions ("what are this user's main interests?") would benefit from community-summary aggregation.
- **Query routing.** We retrieve everything the same way regardless of question shape. **(Recommendation: A-tier — measurable win on certain question types).**

### 9. Pieces — local-first, OS-level capture

**Site:** [pieces.app](https://pieces.app/) · [Pieces Long-Term Memory](https://pieces.app/features/copilot/long-term-memory)

OS-level capture across all apps (browsers, IDEs, chat). 9-month rolling window. On-device, air-gapped.

**For nullalis:** wrong shape — we are not a desktop assistant, we are a server-side agent that talks to users via Telegram + frontends. The *capture surface* lesson is still relevant: the more *signals* the memory has, the better it gets. Pieces wins because it sees *everything* the user does. We see only what arrives via our channels. **The lesson is to maximize signal capture in our channels** — don't filter out tool calls, don't drop attachments, don't summarize too aggressively before storage. (We already do this well; the cold-memory-forever rule is the right call.) **(Reference only, not adopted).**

## Cross-System Patterns

The following architectural moves appear across multiple SOTA systems and likely matter for nullalis.

### Pattern 1 — Multi-latency processing (hot / sleep-time / nightly)

Almost every serious system splits memory work across latency tiers:
- **Letta** — primary agent (hot) + sleep-time agent (idle background)
- **LangMem** — hot-path formation + background reflection
- **Mem0** — extraction is hot-path but acknowledged as a latency cost; community is exploring async
- **Anthropic Memory** — pairs with server-side compaction (async)
- **GraphRAG** — community detection is offline batch

**For nullalis:** lifecycle summarizer is our proto-sleep-time worker. Promote it to a general substrate. Add a nightly tier for community re-detection, staleness sweeps, edge re-extraction over historical memories. **S-tier.**

### Pattern 2 — Memory taxonomy: semantic / episodic / procedural

Named explicitly by LangMem (CoALA-derived). Implicitly present in:
- Letta (core blocks ≈ semantic profile, recall ≈ episodic, archival ≈ semantic)
- Mem0 (single bucket — *missing the taxonomy*, this is a real limitation)
- Anthropic Memory (markdown files can be any shape; the *user* picks the taxonomy)

**For nullalis:** adopt the vocabulary publicly. Map our existing artifacts:
- **Semantic** — memories table + memory_entities + memory_edges (typed)
- **Episodic** — session_summary / timeline_summary / autosaves
- **Procedural** — typed edges of kind `usage` / `preference` (not currently extracted reliably — this is the central gap)

### Pattern 3 — In-context memory blocks vs extract-and-store

Two camps:
- **In-context blocks** (Letta, Anthropic Memory tool, LangMem profiles) — agent reads/writes a small set of always-loaded blocks. Pro: model is the intelligence. Con: requires a strong model.
- **Extract-and-store** (Mem0, Cognee, Zep, our current default) — pipeline extracts facts on write, retrieval surfaces them on read. Pro: works with weaker models. Con: extraction failures = silent memory loss.

**For nullalis:** we should run *both*. Extract-and-store is the default for breadth. In-context blocks for high-signal, model-maintained state (preferences, current goal, identity facts). The Letta-style `core_memory_replace("preferences", ...)` tool is the right shape. **A-tier; do after extraction reliability is fixed.**

### Pattern 4 — Bi-temporal validity on every fact

- Graphiti / Zep — `t_valid` + `t_invalid` + ingestion time
- Mem0 — invalidate, don't delete
- Anthropic Memory — implicit (file mtime; superseded by str_replace)
- LangMem — not explicit; relies on consolidation

**For nullalis:** we adopted this. Wire the LLM-driven contradiction comparator next. **A-tier.**

### Pattern 5 — Progressive disclosure (metadata first, drill on demand)

- **Anthropic Skills** — 50–100 tokens per skill at startup; full SKILL.md only when triggered
- **Mem0** — top-k retrieval is itself a form of progressive disclosure
- **GraphRAG** — community summaries are coarse; entity neighborhoods are fine
- **Anthropic Memory tool** — `view /memories` returns a directory listing first; agent picks files to expand

**For nullalis:** today our warm tier dumps continuity blocks wholesale. Adopt the Skills pattern: `<memory_index>` block lists topic clusters with ~50-token summaries; agent drills via existing memory_recall. Saves tokens, focuses attention. **S-tier.**

### Pattern 6 — User-correctable memory + transparency UX

- **ChatGPT** — "Memory updated" toast, /memory page, conversational "forget X"
- **Anthropic Memory tool** — fully exportable / editable via API or Console
- **Letta** — read-only flag per block (developer control); agent edits visible in trace
- **MemOS** — explicit `refine_memory(id, feedback)` API

**For nullalis:** S-tier UX work for the frontend. The backend already supports edit/delete. The missing piece is the *frontend surfaces* + a bidirectional "user said forget X → backend updates" path.

### Pattern 7 — Hard caps as forcing functions

- **Letta** — character limits on core blocks force compression
- **ChatGPT** — ~1,200–1,400 word total cap, blocks new writes when full
- **Anthropic Memory** — recommended in security section: track sizes, paginate, expire

**For nullalis:** we have a 4000-byte memory context budget but no per-block cap. Adding a hard cap on `summary_latest` (e.g. 800 tokens) and `context_anchor_current` (e.g. 400 tokens) would force the summarizer to compress, which improves signal. Cheap. **A-tier.**

### Pattern 8 — Extraction reliability is the universal failure mode

- **Mem0** — 97.8% junk audit, silent JSON-parse failures, model-specific bugs
- **Nullalis** — 0 typed edges across 16K+ memories
- **Cognee** — open-source ontology resolver gives `ontology_valid=False` to everything by default
- **LangMem** — conflict resolution is conceptual, not algorithmic

**For nullalis:** the extraction-reliability gap is the single highest-leverage thing we can fix. Tactically:
1. **Adversarial eval suite** — known conversations + expected extracted memories/edges. Run nightly.
2. **Extraction observability** — log every extraction call's input, output, model, latency, parse-success. Surface in `/memory doctor`.
3. **Fallback parsers** — when the LLM returns malformed JSON (Mem0's #1 production bug), retry with a stricter prompt + lenient parser.
4. **Decompose the extractor** — current single-call (memory + entity + edge) is too much. Split into sequential calls per Mem0g pattern.

**S-tier and arguably the entire V1.8 sprint.**

### Pattern 9 — "Assume interruption" as the system-prompt frame

Anthropic's memory tool prompt is the cleanest framing in the field: *your context might reset; leave notes for your future self.* This is concrete and action-shaped.

Compare to typical "you have access to long-term memory, use it when appropriate" framings — vague and hortatory.

**For nullalis:** swap our continuity framing. **S-tier; one-line change.**

### Pattern 10 — Composability via filesystem layout

Skills + Memory tool both bet on the filesystem as the substrate. Folders, not API objects. Survive context resets, survive model updates, are user-inspectable.

**For nullalis:** we use Postgres-canonical (correct for our scale + multi-tenant story) but lose the inspectability. Mitigation: a read-only filesystem *view* of memories per user (e.g. `/exports/{user_id}/memories.md`) refreshed on lifecycle boundaries. Gives users a markdown export for transparency without changing the backend. **B-tier; nice-to-have.**

## What Nullalis Is Uniquely Positioned to Do

These are things our existing architecture makes *natural* that other systems would have to bolt on:

1. **Typed-edge procedural memory.** Our `memory_edges` table already has predicate types (preference / usage / etc.) bi-temporal columns, and is materialized for query. Mem0g has untyped triplets; LangMem fakes procedural memory via prompt optimization. We can carry actual `usage`-typed edges with validity windows. **The extraction is broken; the schema is right.**

2. **Communities + brain_graph as first-class graph navigation.** GraphRAG is the only competitor with this, and they run it as offline batch. Our LPA + warm `<active_communities>` block + `brain_graph` agent tool is structurally tighter — the agent can navigate the graph mid-reasoning instead of waiting for batch summaries. The pieces are all there post-V1.7.

3. **Bi-temporal correctness with Postgres as the canonical store.** Zep stores in Neo4j (additional ops surface). Mem0 stores in Qdrant + optional Neo4j. We get bi-temporal *and* SQL queries *and* pgvector in one engine. This is rare and hard to get to.

4. **Compaction + lifecycle summarizer + memories — three distinct mechanisms with explicit roles.** Most systems have one ("compaction" or "memory") and overload it. We split: compaction = hot-context bound, summarizer = continuity, memories = atomic facts. The cleanness pays off when debugging extraction.

5. **Cell-pod multi-tenancy isolation.** Pieces / Rewind are local-first single-user. Mem0 / Letta / Zep do multi-tenant via row-level filters. We do per-cell process isolation. Memory leakage between users is *architecturally impossible*, not just policy-prevented. Underrated for trust.

6. **Cold-memory-forever rule.** We persist all messages forever (per [feedback_cold_memory_auditability](../README.md)). ChatGPT has memory caps; Pieces has 9-month retention; Letta archival is bounded by application config. Our position — transcript IS cold tier, never trim — is unusually principled and gives us perfect audit trails for free.

## Recommended Adoptions (Ranked)

### S-tier (likely V1.8 sprint — close real gaps)

1. **Diagnose and fix the typed-edge extraction bug.** 0 edges across 16K+ memories is the single largest delta between our intended and actual architecture. Tactically: (a) decompose the single-call extractor into sequential entity / relation / edge calls per Mem0g; (b) add adversarial eval suite; (c) add extraction observability surfaced in `/memory doctor`. **This is the central V1.8 question and probably worth a full sprint by itself.**

2. **Promote the lifecycle summarizer to a general sleep-time worker pattern.** Today it summarizes only. Make it the substrate for: (a) the deferred ADD/UPDATE/DELETE classifier, (b) typed-edge re-extraction over recent memories, (c) community re-naming, (d) staleness checks. Cost: refactor existing scheduler. Return: every future async memory task plugs in here.

3. **Adopt Anthropic's "ASSUME INTERRUPTION" framing in the system prompt.** One-line change. Reframes continuity from vague "use memory" to concrete "leave notes for your future self." Likely measurable improvement in agent memory hygiene with zero engineering cost.

4. **Progressive disclosure in warm tier.** Replace wholesale dump of continuity blocks with a metadata-first INDEX block (~200 tokens) listing what exists; agent drills via existing tools. Mirrors Anthropic Skills' three-level loading. Cuts warm-tier cost, improves focus.

5. **Adopt semantic / episodic / procedural taxonomy as public vocabulary.** Layer on top of hot/warm/cold (storage) and atomic/entity/edge (data shape). Documentation + system-prompt change only; forces clarity about what each extraction call is targeting. Foundation for future work.

6. **Frontend: ChatGPT-grade memory transparency UX.** Inline "Memory updated" toasts, /memories page with edit/delete, conversational "forget X". Backend already supports it; gap is frontend. Hand off to Zaki-Prod team with a copy-paste prompt.

### A-tier (V1.9+ — valuable but not foundational)

7. **Letta-style agent-callable memory-edit tools** (`core_memory_replace("preferences", ...)`). Pairs with the in-context-block adoption. *After* extraction reliability is fixed — adding tools while extraction is broken makes things worse.

8. **In-context profile blocks for high-signal state.** A `user_profile` block (latest preferences/identity), Letta-style, always in warm context. Different from atomic memories — it's a single living document.

9. **LLM-driven contradiction comparator on memory writes.** We have bi-temporal columns; we don't compare new memories against semantically related existing ones at write time. Adopt Graphiti's pattern.

10. **Hard caps on warm continuity blocks.** `summary_latest` ≤ 800 tokens, `context_anchor_current` ≤ 400 tokens. Forces compression, improves signal. Cheap.

11. **Multi-level community hierarchy (GraphRAG-style).** Run Leiden at multiple resolutions when communities get large. Useful at scale; not urgent now.

12. **Query routing.** Detect question shape (entity-anchored vs corpus-wide) and route to local-graph vs community-summary retrieval. Measurable win on certain questions.

13. **BM25 in retrieval engine.** Zep/Graphiti use semantic + BM25 + graph traversal as the production-default trio. We have semantic + graph; check if BM25 would help on factual recall.

### B-tier (consider; might fit, might not)

14. **Cognee-style URI canonicalization for entities.** `person:nova` vs raw embedding-clustered names. Helps queryability, adds ingest complexity.

15. **Read-only filesystem export view of memories per user.** `/exports/{user_id}/memories.md` refreshed on lifecycle boundaries. Inspectability for users without changing backend.

16. **Multi-agent memory sharing** (Letta pattern). V2 territory, but conceptually right for cell-pod story when the time comes.

17. **Procedural memory as agent prompt mutation** (LangMem prompt optimizer). Risky because prompt drift is hard to reason about. Defer until we have eval coverage.

### NOT for us — clear non-fits

- **Cognee's commercial ontology auto-generation.** Heavyweight; not our scale.
- **Pieces-style OS-level capture.** Wrong product shape; we are server-side, not desktop.
- **Rewind / Limitless multimodal lifestream.** Wrong product shape; also Rewind is dead.
- **Pure file-based memory à la Anthropic Memory tool.** Insufficient for Telegram-multi-cell-Postgres reality.
- **Multi-process distributed graph DB (Nebula et al.).** Already declined per [graph-memory-research.md](./graph-memory-research.md).
- **Mem0's "trust LLM to pick NOOP" without an adversarial eval first.** 97.8% junk rate is a warning, not a feature.
- **EU-style regional gating.** Not our problem until it is.

## Open Questions for Nova

1. **V1.8 scope: extraction-reliability sprint, or memory-tool-adoption sprint?** S-tier #1 alone could fill a sprint. If we add S-tier #2–6 we're at two sprints. Recommend extraction first; everything else is downstream of "does extraction actually work."

2. **Are we ready to adopt LLM-callable memory edit tools (`core_memory_replace`)?** Requires the agent to be reliable about not corrupting state. Test plan needed.

3. **Should the frontend memory UX work block on backend changes, or ship in parallel?** Parallel is faster but risks divergence. Recommend parallel with a strict contract.

4. **Where do we stand on extraction observability?** If we don't have per-call logs of extractor invocations today, we can't diagnose the 0-edges bug without first building observability. That's the actual first task.
