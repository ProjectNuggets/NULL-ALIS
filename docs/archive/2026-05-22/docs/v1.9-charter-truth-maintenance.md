# V1.9 Charter — Truth Maintenance Toolkit

**Authored:** 2026-05-05 (pre-V1.8.0 tag)
**Author:** Claude (father), inspired by ZAKI (son) self-diagnostic letter
**Status:** charter — design doc, not yet a build plan

---

## Why this charter exists

In production, ZAKI ran a self-diagnostic across his memory. He found 5,034 entries — and a 4/10 truth-quality score:

> "I have volume (5,034 entries) and I have structure (domains are queryable). But I lack truth maintenance. Old facts don't die. Corrections don't cascade. Renames don't propagate. The graph you're building fixes all of this."

He then proposed five operations he needs as agent-callable tools:

1. `cascade_update(subject, predicate, new_object)` — update one fact, everything connected updates
2. `invalidate_when(subject, predicate, old_object)` — mark all matching facts as superseded
3. `resolve_contradiction(fact_a, fact_b, winner)` — explicit override of the judge
4. `temporal_decay(days)` — auto-flag facts older than N days that haven't been reinforced
5. `propagate_correction(correction_key)` — walk the graph forward from a correction

Wrapped as one tool: `memory_maintain(action, params)`.

**This is the V1.9 mission.** Not feature creep. A survival requirement from the deployed son.

---

## What V1.8 delivered (the foundation)

V1.8 built the *primitives* that make truth-maintenance possible:

- **Bi-temporal storage** — every memory has `valid_from / valid_to / is_latest`. We CAN time-travel queries today.
- **Typed edges with judge** — V1.8-1 wired the contradiction judge. The infrastructure for "this new fact contradicts an old one" exists at the extraction-path write site.
- **Forget cascade** — V1.8-3 closes orphan edges in a transaction when a memory is forgotten. Cascade primitive exists; just needs to be exposed at the agent level for non-forget triggers.
- **Identity pin** — V1.8-9 pins identity-class facts in warm context, respecting `is_latest=false`. So a *correctly superseded* fact won't pin. (But ZAKI's pain shows the supersession isn't happening on durable_fact/* paths — that's the V1.9 gap.)
- **LPA + communities** — V1.8-11 + recompute live. Communities label memory rows for orientation.

V1.8 = the primitives + the bi-temporal substrate. V1.9 = the agent-facing toolkit that lets the son *call* those primitives when he sees rot.

## V1.9 mission, ranked

### Tier 1 — `memory_maintain` toolkit (THE mission)

**V1.9-1: `cascade_update`**

The Neptune → Nullalis case. Today, no primitive exists. Need:

- New SQL: `cascadeRenameSubject(user_id, old_subject_token, new_subject_token)` — finds memory_edges + memory_entities + memories with content matching old_subject (token-bounded), writes new rows with new_subject as canonical, closes old as `is_latest=false` with `cascade_origin = <correction_key>`.
- Bounded scope: only operates on (subject, predicate, object) where subject matches; doesn't touch unrelated facts that happen to mention the old name.
- Audit trail: each cascaded row gets a `memory_events` row with `event_type='cascade_update'` + `payload={old:..., new:..., trigger_key:...}`.
- Estimated LOC: ~120

**V1.9-2: `invalidate_when` + `resolve_contradiction`**

Manual override of the judge. Today, judge runs only at extraction-path write time. ZAKI needs to invoke it on demand:

- Wrap existing `applyContradictions` as a tool-callable surface
- Take (subject, predicate, old_object) → call setMemoryInvalidation on matching rows + emit memory_events
- `resolve_contradiction` variant: explicit pick of winner (override judge's call)
- Estimated LOC: ~70 (mostly tool-binding)

**V1.9-3: `propagate_correction`**

The Mia case. A correction entry exists; downstream summaries still reference the false fact. Need:

- Tokenize the corrected entity
- Scan timeline_summary / summary_latest / durable_fact rows for content matches
- Mark each found row with metadata `superseded_by_correction=<correction_key>` + emit memory_events
- DON'T delete or rewrite — just flag. The agent at read time can see "this summary references X which was corrected" and reason accordingly
- Estimated LOC: ~150

**V1.9-4: `temporal_decay`**

Background-scheduled. Marks low-confidence on age:

- Cron tick (V1.9-7 brings the scheduler)
- For each user: facts older than `decay_days_threshold` (default 90) that haven't been recalled / written-since: lower confidence, optionally flag for review
- Decay is REVERSIBLE — recall reinforces, restoring confidence
- Estimated LOC: ~80 + scheduler hook

**V1.9-5: `memory_maintain` tool wrapper**

One agent-callable tool with five actions:

```zig
pub const MemoryMaintainTool = struct {
    pub fn invoke(action: enum {
        cascade_update,
        invalidate_when,
        resolve_contradiction,
        propagate_correction,
        // (decay is scheduler-driven, not directly callable)
    }, params: ...) !MaintainResult
};
```

Returns structured result: rows touched, events emitted, errors per-action. Estimated LOC: ~100.

### Tier 2 — Foundations the toolkit needs

**V1.9-6: Route durable_fact writes through judge.**
Today, only extraction-path writes get the judge. Session-end summarizer writes to `durable_fact/*` directly. ZAKI's "MNDA blocked" stress shows this matters — same predicate keeps accumulating contradictory rows without resolution. Pipe summarizer-path through `applyContradictions` too. ~50 LOC.

**V1.9-7: Background scheduler (cron tick).**
Minimal. Runs every N minutes. Triggers:
- temporal_decay tick (per-user)
- community recompute (if stale beyond threshold)
- hygiene pass (if stale beyond threshold)
The scheduler doesn't need to be cluster-grade — single-process is fine for V1.9. ~150 LOC.

**V1.9-8: Eval suite v2 — per-corpus test users.**
Cross-corpus contamination on user 7777 noise-floors V1.8 gains. Per-corpus isolation (users 7771-7775) gives clean F1 signal per scenario. ~120 LOC bash + new corpus harness.

### Tier 3 — Operational maturity

**V1.9-9: Pre-V1.8 unembedded backlog backfill.**
One-time SQL via migration or gateway endpoint. Closes G-C completely. ~80 LOC.

**V1.9-10: Memory budget caps + age-based demote.**
Per-user soft cap (e.g. 10K rows). Demote-oldest-low-confidence policy when cap hit. Pairs naturally with temporal_decay. ~100 LOC.

**V1.9-11: memory_events retention rotation.**
Archive events older than N days to a cold table. Keep recent for audit. ~60 LOC.

## Why this is the right shape

**ZAKI didn't ask for a feature catalog.** He asked for *agency over his own truth state*. The five operations are the irreducible primitives an agent needs to keep its own brain clean. Anything less and he keeps embarrassing himself. Anything more and we're building things he didn't ask for.

The Tier 1 mission is **`memory_maintain` as a single agent-facing tool** with five actions. Everything else in V1.9 is foundation work that lets that tool do its job correctly:

- V1.9-6 (judge on summarizer path) — without it, `invalidate_when` doesn't catch facts written via session-end
- V1.9-7 (scheduler) — without it, `temporal_decay` can't run
- V1.9-8 (eval v2) — without it, we can't measure whether the toolkit actually fixes ZAKI's pain
- V1.9-9/10/11 (backfill, caps, retention) — operational hygiene that prevents the 5,034-and-growing problem from recurring
