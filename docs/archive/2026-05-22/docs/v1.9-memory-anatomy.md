## TL;DR

ZAKI's memory is a **bi-temporal graph store with five layers**:

1. **Memory rows** (`memories` table) — the substrate
2. **Typed edges** (`memory_edges` table) — semantic relationships between rows + entities
3. **Entity nodes** (`memory_entities` table) — coref'd canonical names (Helix, Eli, OmniCorp...)
4. **Communities** (`memory_communities` table) — LPA clusters over the edge graph
5. **Audit trail** (`memory_events` table) — every lifecycle event (upsert, edge_added, edge_closed, episode, etc.)

Every row in `memories` belongs to one of **four artifact roles** (W1.3 taxonomy):

- **continuity** — current product truth, retrievable, injectable
- **audit** — cold deep-dive records, not injected
- **index** — discovery surfaces (timeline index)
- **user** — anything user-authored that doesn't match a system pattern

And each row carries a **memory_type category**:

- `core` — important enough to be **mutable** + **immortal** (W-INT-01 guard preserves close-out cols on upsert)
- `daily` — session-scoped, append-only at construction, mutable only via explicit cleanup paths
- `conversation` — short-lived, low-priority
- `custom` — user-defined alternative to the three standard categories

### W-INT-01 immortality guard (zaki_state.zig:2536-2549)

The `upsertMemory` ON CONFLICT path uses CASE-guards:

```sql
ON CONFLICT (user_id, key) DO UPDATE SET
  session_id    = CASE WHEN memory_type = 'core' THEN NULL ELSE EXCLUDED.session_id END,
  content       = EXCLUDED.content,
  memory_type   = CASE WHEN memory_type = 'core' THEN 'core' ELSE EXCLUDED.memory_type END,
  valid_to      = CASE WHEN memory_type = 'core' THEN valid_to    ELSE NULL END,
  invalid_at    = CASE WHEN memory_type = 'core' THEN invalid_at  ELSE NULL END,
  expired_at    = CASE WHEN memory_type = 'core' THEN expired_at  ELSE NULL END,
  is_latest     = CASE WHEN memory_type = 'core' THEN is_latest   ELSE TRUE END
```

**Effect:** A core row, once written, preserves:
- Its core-ness (cannot be demoted via re-upsert)
- Its bi-temporal close-out columns (cannot be resurrected via re-upsert)
- Its session_id = NULL (forced global)

This is why agent self-modification of core rows is "blocked" — re-upsert can't undo prior close-out, demotion, or session-scoping. To mutate a core row, the agent must call EXPLICIT operations: `memory_demote` (core→daily), then archive/edit on the demoted row.

### Bi-temporal model

- `valid_to` IS NULL → row is currently valid
- `valid_to` IS NOT NULL → row was closed at that timestamp; still in DB as audit trail
- `invalid_at` → when contradiction was detected (V1.6 judge)
- `expired_at` → when the fact stops being relevant (often = valid_to)

The `MEMORIES_VALIDITY_FILTER` in SQL is:
```sql
(valid_to IS NULL OR valid_to > EXTRACT(EPOCH FROM NOW())::bigint)
```

A retrieval that uses this filter sees only "currently valid" rows. Closed-out rows are invisible to agent recall but visible to `/brain/timeline`.

## Layer 3 — Predicates (the mutability decision tree)

Every operation that touches a memory row first asks **the predicates** in `src/memory/root.zig`. These determine what the agent can do.

### `isInternalMemoryKey(key)` — internal-only prefixes

```
autosave_user_*     → TRUE
autosave_assistant_* → TRUE
last_hygiene_at      → TRUE
__tombstone__/*      → TRUE
__bootstrap.prompt.*  → TRUE
otherwise            → FALSE
```

Used to hide from `/brain` views + skip in some loader paths. Internal keys are NEVER agent-editable.

### `isAppendOnlyMemoryKey(key)`

```
session_summary/*, timeline_summary/*, session_checkpoint_*,
timeline_index/current, autosave_user_*, autosave_assistant_*,
compaction_summary/*, summary_fallback/*, compaction_dropped/*
                     → TRUE
otherwise            → FALSE
```

Append-only means: written once, never agent-edited. Cleanup only via explicit paths (purge_topic).

### `isSystemManagedMemoryKey(key)`

```
context_anchor_current        → TRUE
summary_latest/*              → TRUE
durable_fact/*                → TRUE          ← ZAKI's MNDA zombies are HERE
isAppendOnlyMemoryKey(key)    → TRUE
otherwise                     → FALSE
```

System-managed = "the agent runtime owns this; agent self-modification is blocked." This is the broader umbrella that catches the durable_fact/* family ZAKI tried to invalidate.

### `isMutableMemoryEntry(key, category)`

```
isTombstoneKey || isMarkdownLineKey || isAppendOnlyMemoryKey   → FALSE
category == .core   → TRUE
otherwise           → FALSE
```

Surprising fact: **only `core` rows are "mutable" at this layer.** Daily and conversation rows are NOT mutable per this predicate. (Their typical lifecycle is "write once, then expire," not "mutate.")

### `isEditableMemoryEntry(key, category)`

```
if !isMutableMemoryEntry(key, category)   → FALSE
if isInternalMemoryKey(key)               → FALSE
if isSystemManagedMemoryKey(key)          → FALSE
otherwise                                 → TRUE
```

This is the **gatekeeper for memory_archive, memory_forget, memory_edit**. The tools call `lookupMemoryLifecycleEntry`, which returns `.editable` or `.protected` based on this predicate.

**Decision matrix:**

| Key shape | category | Editable? | Why |
|---|---|---|---|
| `user_name` | core | ✅ YES | user-shape, core, not internal/system |
| `project_neptune_launch` | core | ✅ YES (in theory) | same as above |
| `project_neptune_launch` | daily | ❌ NO | only core is mutable |
| `durable_fact/1776371382/2` | core | ❌ NO | system-managed |
| `summary_latest/<sid>` | core | ❌ NO | system-managed |
| `timeline_summary/<sid>/<ts>` | any | ❌ NO | append-only |
| `extracted_abc123` | core | ✅ YES (in theory) | not in system-managed list |
| `autosave_assistant_123` | any | ❌ NO | internal |

## Why ZAKI's cleanup attempts were blocked (the diagnosis)

ZAKI tried `memory_archive` on user-authored zombie entries like `project_neptune_launch`. He got "protected from archival." Per the predicates, user-authored core SHOULD be editable. But he was blocked.

**Three possible reasons (need empirical confirmation, but anatomy points here):**

1. **Some user-authored "core" entries were actually written with `memory_type='daily'`** — the user (or older code path) used memory_store with category=daily. Those are NOT mutable per `isMutableMemoryEntry`.

2. **The W-INT-01 guard preserves CLOSE-OUT columns on re-upsert** — even if archive succeeded once, a subsequent upsert wouldn't resurrect the row. But the FIRST archive call should still pass `isEditableMemoryEntry`.

3. **Some "user-authored" keys are actually classified under continuity** — e.g., the V1.7 cmt9.6 unification routed extracted_* into "continuity" role. If user calls memory_store with key=`extracted_<hash>`, it lands as system-managed despite having user-typed content.

**The fix is not to relax the guard** — the guard is right for "agent must not gaslight the user." The fix is to give ZAKI a separate mechanism: **annotate via metadata, don't mutate the row.**

V1.9-3 propagate_correction already does this for prose memories: it sets `metadata.superseded_by_correction` WITHOUT touching the row's content/category. The metadata write goes through a different SQL path that doesn't trigger the immortality guard's CASE-WHEN.

**That means: V1.10 closes ZAKI's pain by EXTENDING THE METADATA APPROACH, not relaxing the guard.**

## Layer 4 — Read paths

### `mem.recall(query, limit, session_id?)` — the primary retrieval surface

- BM25 keyword + pgvector cosine merged via reciprocal-rank fusion
- Returns `[]MemoryEntry` ordered by hybrid score
- Filters: `MEMORIES_VALIDITY_FILTER` applied (closed-out rows hidden)
- Used by: `tools/memory_recall.zig`, `agent/memory_loader.zig`

### `mem.list(category?, session_id?)` — full enumeration

- Returns all rows for a tenant, optionally scoped
- Used by: `memory_purge_topic` (for topic-string scan), `/brain/list`
- Filters: validity per `MEMORIES_VALIDITY_FILTER`

### `mem.get(key)` — direct key lookup

- Returns `?MemoryEntry`
- Used by: `lookupMemoryLifecycleEntry` (the editability gate), `appendDirectEntry` in memory_loader

### `agent/memory_loader.zig::loadTurnMemorySlot` — warm-context builder

The orchestration that runs every agent turn. Builds the `<memory_for_turn>` block:

```
<memory_for_turn>
  <active_identity source="pinned">          ← V1.8-9 — context-invariant facts
  - Eli Vance is a software engineer
  - Eli works at OmniCorp
  ...
  </active_identity>
  
  [Memory context]
  pending_conflicts: ...                     ← V1.7 sentinel surfaced
  summary_latest/...: ...                    ← continuity bucket
  durable_fact/...: ...
  extracted_<hash>: ...                      ← V1.7 cmt9.6: extracted_* joined continuity
  ...
  
  <graph_neighbors>                          ← V1.7a-2 1-hop expansion
  ...
  </graph_neighbors>
  
  <active_communities source="lpa">          ← V1.7-S2b top-3 named communities
  Helios launch deadline (4 members), ...
  </active_communities>
</memory_for_turn>
```

### `/brain/*` HTTP endpoints (read-only browse surfaces)

- `/brain/graph` — entity graph with edges + community labels
- `/brain/timeline` — chronological memory list
- `/brain/memory/<key>` — single-row drilldown including provenance + community + edges
- `/brain/communities` — LPA clusters
- `/brain/orphans` — memories with zero live edges
- `/brain/diff` — births/deaths in a window
- `/brain/local-graph` — k-hop expansion from a center key
- `/brain/documents` — supermemory-style document view

`BRAIN_USER_KEY_FILTER` excludes the 13 hidden prefixes from these endpoints (autosave/checkpoint/summary families etc.).

### Mutation paths

| Tool / SQL | Effect | Bypasses W-INT-01? |
|---|---|---|
| `memory_archive` (tool) | sets valid_to/invalid_at/expired_at + cascade close-out edges | NO — calls setMemoryInvalidation directly (UPDATE without CASE-guard); but pre-checks `isEditableMemoryEntry` |
| `memory_forget` (tool) | DELETE FROM memories | NO — same pre-check |
| `memory_edit` (tool) | UPDATE content | NO — same pre-check |
| `memory_demote` (tool) | UPDATE memory_type from 'core' to 'daily' | YES — explicit demote-only SQL (`zaki_state.demoteMemory`) bypasses CASE-guard |
| `memory_purge_topic` (tool) | bulk DELETE by topic on agent-generated keys only | YES — only operates on `isPurgeableFamily()` keys (autosave/checkpoint/summary families) |
| `memory_maintain cascade_update` (V1.9-1) | rewrites memory_edges; no memory rows touched | N/A (edge graph) |
| `memory_maintain invalidate_when` (V1.9-2) | UPDATE memory_edges is_latest=false | N/A (edge graph) |
| `memory_maintain resolve_contradiction` (V1.9-2) | calls setMemoryInvalidation on loser_key | NO — but bypasses pre-check (no `isEditableMemoryEntry` gate) |
| `memory_maintain propagate_correction` (V1.9-3) | UPDATE memories.metadata SET superseded_by_correction | YES — modifies metadata (not protected by CASE-guard) |
| `memory_maintain temporal_decay` (V1.9-4) | UPDATE memories.confidence_score | YES — modifies confidence_score (not protected) |

**Key observation:** the metadata + confidence_score columns are **NOT** in the W-INT-01 CASE-guard. They can be modified on core rows freely. **This is the path forward for V1.10 truth maintenance on prose-level zombies.**

## Layer 6 — Edges, entities, communities, events (the graph)

### `memory_edges` — typed semantic relationships

```
id                bigint  PK
user_id           bigint  NOT NULL
source_key        text    NOT NULL    typically extracted_<hash> or entity_id
target_key        text    NOT NULL    typically entity_id
predicate         text    NOT NULL    SCREAMING_SNAKE_CASE: PREFERS, WORKS_AT, etc.
attribution       text                'extraction_classifier', 'session_end_loop', 'cascade_rename', etc.
confidence        double
weight            double  NOT NULL DEFAULT 1.0
valid_from        bigint
valid_to          bigint              bi-temporal close-out
invalid_at        bigint
expired_at        bigint
is_latest         boolean NOT NULL DEFAULT TRUE
created_at        timestamptz NOT NULL DEFAULT now()
```

Unique partial index: `(user_id, source_key, predicate, target_key) WHERE is_latest`.

### `memory_entities` — coref'd canonical names

```
id                text   PK    32-char hex (deriveEntityKey or random)
user_id           bigint NOT NULL
name              text   NOT NULL    "Helix", "Eli Vance", "OmniCorp"
name_lower        text   NOT NULL    LOWER(name) for coref lookup
entity_type       text   NOT NULL DEFAULT 'PROPER'
name_embedding    vector(1024)        nullable; populated when coref runs
linked_memory_ids text[]              cache of linked memory keys
```

### `memory_communities` — LPA clusters

```
community_id      int     PK
user_id           bigint  NOT NULL
name              text                LLM-named ("Helios launch deadline", etc.)
name_source       text                'llm' | 'fallback'
member_set_hash   text                cache invalidation key
generated_at      timestamptz
```

LPA propagates over `memory_edges` filtered by `is_latest` AND endpoint validity. Per V1.8-11: endpoints can exist in EITHER `memories` OR `memory_entities`.

### `memory_events` — audit log

```
id              text       PK
user_id         bigint     NOT NULL
memory_id       text                  optional (some events are graph-only)
event_type      text       NOT NULL   upsert | edge_added | edge_closed | episode | compose | demote | extraction | resolve_contradiction
payload         jsonb      NOT NULL DEFAULT '{}'   structured event details
created_at      timestamptz NOT NULL DEFAULT now()
```

This table is **append-only by design**. Every lifecycle change writes a row. The 1,567+ events for user 7777 we saw earlier are mostly `upsert` + `episode` + `edge_added` + `edge_closed`.

### `memory_provenance` — source attribution

```
memory_key      text       PK
user_id         bigint     NOT NULL
session_id      text                  origin session of the write
snippet         text                  conversation excerpt that triggered the write
created_at      timestamptz
```

Powers `/brain/memory/<key>` "where did this come from?" surface. NOT involved in mutability rules.

### `memory_embeddings_e5_1024` — vector store

Per-key 1024-dim e5-1024 embeddings. Joined to `memories.key` for cosine retrieval. Lifecycle: written by `syncVectorAfterStore` after every memory_store/persistExtracted; deleted alongside memory_forget.

## Categories deep-dive

| Category | Lifecycle | Mutability | Use case |
|---|---|---|---|
| `core` | Promoted by repeated corroboration OR explicit user-store-with-category=core | Mutable per `isMutableMemoryEntry` | Important facts the user wants to persist permanently |
| `daily` | Default for transient session writes; auto-aged | NOT mutable per predicate | Working-memory shape, ephemeral |
| `conversation` | Per-message context | NOT mutable | Session-scoped chat snippets |
| `custom { name }` | Caller-defined | Treated as `daily` for mutability | Plugin extension point |

The asymmetry — only `core` is "mutable" while `daily` is not — is intentional. Daily/conversation are write-once; if you want to "edit" them you write a new core row that supersedes via the contradiction judge.

## Bidirectional supersede pointers (V1.9-3, the V1.10 foundation)

When `memory_maintain action=propagate_correction` runs, it writes:

**On each flagged target row (`UPDATE memories.metadata`):**
```json
{
  "superseded_by_correction": "durable_fact/1778049787/0"
}
```

**On the correction itself (`UPDATE memories.metadata`):**
```json
{
  "superseded_targets": [
    "timeline_summary/agent:zaki-bot:user:1:main/1776595003",
    "timeline_summary/agent:zaki-bot:user:1:main/1776592052",
    "durable_fact/1776631790/0",
    ...
  ]
}
```

**This metadata write bypasses the immortality guard.** The W-INT-01 CASE-WHEN does NOT mention `metadata` — it can be freely UPDATEd even on core rows. That is the architectural seam V1.10 will use to make zombies invisible without violating the V1.7 design intent.

## What the agent CAN and CANNOT do today (V1.9 mutability matrix)

### CAN do

✅ Write new memories (memory_store, compose_memory, extraction_persist)
✅ Cascade-rename entities (V1.9-1)
✅ Invalidate edges by pattern (V1.9-2)
✅ Resolve contradictions by key — if loser is non-protected (V1.9-2)
✅ Propagate corrections — flag prose rows with metadata (V1.9-3)
✅ Decay confidence on stale rows (V1.9-4)
✅ Survey edge-graph contradictions (V1.9-7)
✅ Demote core rows to daily (memory_demote)
✅ Purge agent-generated keys by topic (memory_purge_topic)
✅ Edit/archive/forget non-system, non-internal, mutable user-shape rows

### CANNOT do (per current design)

❌ Edit/archive/forget `durable_fact/*` (system-managed)
❌ Edit/archive/forget `summary_latest/*` (system-managed)
❌ Edit/archive/forget `timeline_summary/*` (append-only)
❌ Edit/archive/forget `compaction_summary/*` (append-only)
❌ Resurrect closed-out core rows via re-upsert (W-INT-01)
❌ Demote core rows then resurrect them (W-INT-01 preserves close-out cols)

**The deliberate gap:** prose-level zombies inside system-managed continuity families (durable_fact, summary_latest, timeline_summary) cannot be MUTATED by the agent, only ANNOTATED via metadata.

## What V1.10 must preserve

1. **The W-INT-01 immortality guard stays.** It's protective. Don't relax.
2. **Memory rows are append-mostly.** Truth evolves via NEW writes + supersede metadata, not via mutation of historical rows.
3. **The bi-temporal model + metadata-write seam.** This is the V1.10 working surface.
4. **The four-role taxonomy** (continuity / audit / index / user). Don't add a fifth without serious thought.

## What V1.10 must close

1. **Prose-level contradiction detection** — V1.9-7 surveyor is edge-graph only. The MNDA case lives in durable_fact prose. Need an LLM-judge prose surveyor that scans clusters of durable_fact entries with similar `metadata.subject` for content-level contradictions, then writes `superseded_by_correction` metadata.
2. **Loader-side supersede filter** — when `memory_loader` walks recall results, if `metadata.superseded_by_correction` is set, EITHER skip the row (cleanest) OR annotate it as "(superseded — see correction <key>)" inline. Either way, zombies become invisible-or-marked at retrieval time. This is the small commit that closes 80% of ZAKI's day-to-day pain.
3. **Timeline view UX in agent prompt** — when ZAKI answers "tell me about X," teach him to render chronological order with the current-state highlighted and superseded entries shown as historical context. Per Nova's intuition: "first this happened, then this, then that."

## Strategic frame for V1.10

The right V1.10 charter, in two sentences:

> *"V1.10 makes the timeline THE canonical view of any memory subject — not a list of facts, but a story of how truth evolved. Rows that have been superseded fade; the latest current truth lives at the top; the chronology is the audit trail."*

This frame:
- Honors the V1.7 immortality guard (rows preserved as historical evidence)
- Honors Nova's intuition (timeline > list)
- Honors ZAKI's diagnosis (prose zombies need LLM judging + metadata flagging, not row mutation)
- Sets up V1.11+ (multi-channel, etc.) by having the timeline UX as a clean primitive
