## D1.7 Spawned-task-ids Accumulator Pattern (INFO)

The recent D1.7 pattern (spawned_task_ids_acc at src/agent/root.zig:2483) is a new accumulator lifecycle that merits documentation for future audits.

**Pattern:**
```zig
var spawned_task_ids_acc: std.ArrayListUnmanaged([]const u8) = .empty;
var spawned_task_ids_transferred: bool = false;
defer {
    if (!spawned_task_ids_transferred) {
        for (spawned_task_ids_acc.items) |id| self.allocator.free(id);
        spawned_task_ids_acc.deinit(self.allocator);
    }
}
```

**Key invariant:** Each string ID is duped at line 4002; appended at line 4006. On every turn exit path, `toOwnedSlice` is called at one of:
- 3856, 3871 (audio reply path)
- 4036 (approval required path)
- 4178, 4231 (exhausted/loop summary paths)
- 4285 (normal exit)

On `toOwnedSlice` success, items[] is transferred to the TurnOutcome and becomes empty; the transferred flag is set **after** the catch (lines 3857, 3872, 4037, 4179, 4232, 4286). This ensures that if toOwnedSlice OOMs, the accumulated strings remain in items[] and the defer cleanup runs.

**Audited decision:** This pattern is correct. The transferred-flag-after-catch ordering is defensive and prevents leaks on OOM at the transfer point. Comments at lines 3847-3855 and 4283-4284 explain the fix for D1.7 finding 2.

**Verdict:** No issues. Pattern is well-implemented and documented inline.
