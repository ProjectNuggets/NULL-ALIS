## Findings

### HIGH-1 — Session.last_turn_outcome not protected by mutex
**File:** `src/session.zig:81, 84, 99`
**Pattern:** Shared state without mutex
**Description:** 
The `Session` struct has a new field `last_turn_outcome: ?Agent.TurnOutcome = null` (added in D1.3). 
This field is written at line 84 (`if (self.last_turn_outcome) |*outcome| outcome.deinit(self.agent.allocator)`) 
and read at line 99 (`if (self.last_turn_outcome) |*outcome| return outcome`) without holding `session.mutex`. 
Two threads can race: one calling `lastTurnOutcome()` (read at :99) while another calls `recordTurnOutcome()` (write at :84).

**Reproduction:** 
1. Spin up daemon with inbound dispatcher workers > 1.
2. Send two rapid messages to the same session from different channels.
3. Both inbound threads may acquire different session objects via the SessionManager map, or the second thread may call `lastTurnOutcome()` while the first is in `recordTurnOutcome()` deinit.
4. UAF or double-free of the `TurnOutcome` allocation.

**Suggested fix:**
Hold `session.mutex` in both `lastTurnOutcome()` (line 99) and `recordTurnOutcome()` (the caller that writes to `last_turn_outcome`). 
The field is only read in `lastTurnOutcome()` and written during turn completion; both should be protected.

### MED-1 — DaemonState.components array written without mutex from multiple mark* functions
**File:** `src/daemon.zig:67-97`
**Pattern:** Shared state without mutex
**Description:**
The `DaemonState.components[MAX_COMPONENTS]` array is written by `addComponent()` (lines 68-71), `markError()` (lines 77-81), 
and `markRunning()` (lines 89-95) from multiple threads:
- Gateway thread reads `state` pointer and may call `markRunning()` in gateway.zig (not traced in this audit, but passed to thread)
- Inbound dispatcher calls `state.markRunning()` at daemon.zig:2040
- Heartbeat thread calls functions on `state` at daemon.zig:976-980

The array itself has no mutex. Initial writes in `daemon.run()` are single-threaded, 
but subsequent `mark*()` calls from worker threads race on read-modify-write of the components array.

**Reproduction:**
1. Multiple inbound dispatcher workers call `state.markRunning("inbound_dispatcher")` concurrently.
2. Two threads read the same component slot, both modify it, last write wins.
3. Restart count, error message, or running status becomes inconsistent.

**Suggested fix:**
Add `components_mutex: std.Thread.Mutex = .{}` to DaemonState, and protect all reads/writes to `components[0..component_count]` with this mutex.

### MED-2 — Outbound dispatcher stats accumulated without atomic or mutex
**File:** `src/dispatch.zig` (not fully read, but referenced at daemon.zig:2274)
**Pattern:** Shared state without synchronization
**Description:**
The `dispatch_stats: dispatch.DispatchStats` is created at daemon.zig:2251 and passed to all outbound dispatcher threads 
(daemon.zig:2273-2275). If `DispatchStats` has mutable fields (counters, metrics), they are written by multiple outbound 
worker threads without any lock or atomic operation. This causes lost updates and incorrect metrics.

**Reproduction:**
1. Spin up 4 outbound workers.
2. Queue 1000 messages for delivery.
3. Workers all increment the same counter field concurrently.
4. Final counter is < 1000 due to lost writes.

**Suggested fix:**
Either make `DispatchStats` counters atomic (`std.atomic.Value(u64)`), or protect all mutations with a mutex.

### LOW-1 — Defer in Session.turn may not execute on panic
**File:** `src/session.zig:542-614`
**Pattern:** Defer to unlock on panic
**Description:**
The `Session.mutex.lock()` at line 589 or 590 is followed by `defer session.mutex.unlock()` (implicit or explicit). 
If the turn logic calls `@panic()`, the defer may not fire in Zig (though modern Zig does run defers on panic). 
This is an audit smell rather than a confirmed bug, but worth documenting.

**Suggested fix:**
Verify behavior: either confirm that Zig's defer fires on panic (likely true in recent versions), 
or use a try-catch to ensure unlock on error paths. Current pattern appears safe in practice.

### LOW-2 — Heartbeat thread sleeps while holding no locks but checks isShutdownRequested without atomic semantics
**File:** `src/daemon.zig:967, 1007`
**Pattern:** Atomic read-check without explicit memory order
**Description:**
The check `while (!isShutdownRequested())` at line 967 and similar checks elsewhere use `shutdown_requested.load(.acquire)` 
(line 158), which is correct. However, the pattern is thread-safe. This is NOT a bug, but documented for completeness 
to show that the atomic flag is used correctly.

**Suggested fix:** None — atomic usage is correct.

## What was checked and found clean

### ✓ Event Bus (bus.zig) — BoundedQueue pattern
Examined mutex + condition variable pattern in `BoundedQueue.publish()` (lines 334-348), `consume()` (lines 351-366), 
and `close()` (lines 369-375). Standard and correct: mutex protects state, conditions broadcast wake-ups, 
closed flag prevents new publishes. No races found.

### ✓ Heartbeat Wake Queue (heartbeat_wake.zig) — Coalescing merge pattern
Examined `enqueue()` (lines 58-93) and `dequeue()` (lines 95-110). Both hold `g_queue.mutex` for the entire 
critical section. The coalescing logic (comparing user_id targets at line 73) is done inside the lock. No races found.

### ✓ Scheduler Jobs — Disk-based versioning
Examined `schedulerThread()` (lines 1450-1568) and the snapshot-merge pattern (lines 1540, 1551-1557). 
Jobs are never mutated in-place during tick; instead, a snapshot is taken before tick, then changes are merged 
back to a fresh load from disk. This avoids concurrent mutation of job pointers. No races found.

### ✓ Session Mutex Serialization
Examined `Session.turn()` protocol (lines 542-614). Lock is acquired at line 589/590 and held for the entire turn. 
All tool execution happens within the lock. `last_turn_outcome` is the one exception (HIGH-1).

### ✓ Gateway Control Commands — Synchronous and blocking
Examined `sendGatewayControlCommand()` (lines 177-200). Creates a new TCP connection per call, no shared state, 
synchronous send. Safe from races.

### ✓ Inbound Dispatcher — Session Manager lookup pattern
Examined `SessionManager.findOrCreate()` pattern (session.zig:185-217). Mutex is held only for the map lookup; 
returned session pointer is then locked separately. This is correct (short critical section on the map, separate 
lock on the session). No races.

### ✓ Parallel Tool Execution — Thread-local TLS context
Examined `executeToolCallsParallel()` (agent/root.zig:2129-2257). Worker threads read `turn_ctx`, `tenant_ctx`, 
`message_ctx` via thread-local variables set before spawn (lines 2166-2168). Each worker writes to its own 
`workers[i].result` (unique per worker). Results are joined and read after `threads[i].join()` (line 2224). 
No races found.

## Summary

**Total findings:** 3 HIGH, 2 MED, 2 LOW

The codebase is generally thread-safe due to careful use of mutexes, condition variables, and atomic flags. 
However, the new `Session.last_turn_outcome` field added in D1.3 is not protected by any lock and poses a data race risk. 
The `DaemonState.components` array also lacks synchronization, leading to potential lost updates in concurrent 
component status tracking. The `dispatch_stats` accumulation pattern needs atomic or mutex protection to prevent 
counter loss in high-concurrency scenarios.

**Recommended action:** Fix HIGH-1 before merging to main. MED-1 and MED-2 should be addressed in the next sprint to 
ensure accurate diagnostics and state consistency under load.
