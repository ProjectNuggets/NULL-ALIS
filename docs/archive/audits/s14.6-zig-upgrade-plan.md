# S14.6 — Zig 0.14 → 0.15+ Upgrade Plan (2026-04-26)

**Audited at commit:** `224bea5`
**Current Zig version:** `0.15.2` (per `.zigversion` — single source of truth across `ci.yml`, `release.yml`, `flake.nix`, `Dockerfile`)
**Goal:** track upstream Zig releases, isolate deprecated stdlib calls, test ahead of Zig releases that affect us.

## Where Zig version is pinned

| Surface | Pin | Sync mechanism |
|---|---|---|
| Dev local | `.zigversion` | manual + flake.nix devshell |
| CI runs | `.github/workflows/ci.yml` | reads `.zigversion` (S3.1) |
| Release builds | `.github/workflows/release.yml` | reads `.zigversion` (S3.3) |
| Docker image | `Dockerfile` | reads `.zigversion` and downloads tarball (S3.1) |
| Nix devshell | `flake.nix` | references the pinned tarball |

**Single source of truth maintained:** `.zigversion` → all paths. To bump Zig, change `.zigversion` and CI + Docker + flake follow.

## Upgrade cadence

Zig minor versions ship roughly every ~6 months. Upgrades are rarely backward-compatible at the source level (stdlib APIs evolve, breaking changes called out in release notes).

**Upgrade trigger conditions:**
1. **Security-relevant fix in upstream stdlib** that affects our hot paths (libstd JSON parser, libstd HTTP client, allocator)
2. **New stdlib feature** we'd genuinely use (e.g. `std.http.Server` matures, eliminating our hand-rolled `std.net.Server` at `gateway.zig`)
3. **Compatibility cliff** — a dep we use bumps min-Zig requirement past our pin
4. **6-month rolling cadence** — at minimum every 6 months even if no specific trigger

## Deprecated stdlib calls to monitor

For each Zig release, scan release notes for deprecations matching our usage. Current high-touch surfaces:

| Surface | Files | What to watch for |
|---|---|---|
| `std.net.Server` | `src/gateway.zig`, `src/http_native/` | Our hand-rolled HTTP path. If `std.http.Server` matures + offers SSE support, S2.X migration becomes viable |
| `std.heap.GeneralPurposeAllocator` / `page_allocator` | everywhere | API renames have happened across versions; check for `.zig` signature changes |
| `std.json` | provider serializers, gateway parsers | Stringify API changed in 0.13 → 0.14; we're past that |
| `std.fmt.bufPrint` / `allocPrint` | logging, prompt construction | Format spec changes |
| `std.posix.getenv` | env var reads (NULLALIS_*) | Less likely to change but watch |
| `std.Thread.spawn` / `std.Thread.Mutex` / `std.atomic.Value` | daemon supervisor | Threading primitives are stable but watch for behavior tweaks |
| `@embedFile` | `src/migrations/0001_initial_schema.sql`, `src/skills/` | Comptime feature — stable |

## Pre-upgrade checklist (before bumping `.zigversion`)

1. **Read upstream release notes** for the target version end-to-end
2. **Run `zig fmt --check src/`** locally on the new compiler — surfaces any syntax-level deprecations
3. **`zig build test`** on new compiler — surfaces stdlib API breakage at test time
4. **Spike on a branch** before merging — `.spike/` harness exists for benchmark comparison
5. **Check deps** — `build.zig.zon` deps might have their own min-Zig bumps that we need to follow
6. **Update CI matrices** — both Linux + macOS jobs in `ci.yml` use `.zigversion`; both must build cleanly

## Post-upgrade checklist

1. Bump `.zigversion`
2. Push to a feature branch; verify CI green
3. Run `.spike/run.sh cold + polluted` — assert pass rate ≥ baseline
4. Update `internals/P1_tech.md` with new Zig version
5. Document any source-level migrations required in `docs/zig-upgrade-NOTES.md` (file doesn't exist yet — create on first real upgrade)
