# v0.2 Turn Origin Roadmap

## Current Runtime Status (March 12, 2026)

`TurnOrigin` enum values in runtime:
- `user`
- `heartbeat`
- `scheduler`
- `wake`
- `proactive`

Wired in execution paths today:
- `user`: default session turns
- `heartbeat`: heartbeat job turns (`job.id == "heartbeat"`)
- `scheduler`: non-heartbeat cron/scheduler turns (`delivery.mode = none`)
- `wake`: forced wake and wake-queue-triggered heartbeat/scheduler turns
- `proactive`: scheduler turns with delivery mode enabled

## Why `wake` and `proactive` exist

They are forward-compatible origin labels for policy and observability:
- finer tool policy per autonomous source
- cleaner ops analytics by source type
- clearer user-visible shaping for system-origin messages

## Current Wiring Notes

1. `wake`
- Assigned for explicit wake paths, including wake queue and forced heartbeat triggers.
- `heartbeat` remains interval-driven.

2. `proactive`
- Assigned for scheduler delivery-mode turns.
- Policy remains constrained by background tool gating.

3. Preserve backward compatibility
- Existing behavior and policy for `user`, `heartbeat`, `scheduler` remains unchanged.
- Metrics/logs should expose origin explicitly to reduce ambiguity during incidents.

## Next Expansion Scope

- No additional origin values are currently planned beyond these five.
- Expand proactive routing beyond scheduler delivery turns only when execution path, policy matrix, and telemetry are defined together.
