# V0.7 Backend Quota Guardrail Plan (Design-Complete Contract)

## Objective
Define an implementation-ready backend quota guardrail so seat/month usage cannot be bypassed by direct API calls, multi-tab bursts, or channel fan-in.

This is a contract-level implementation spec (not line-by-line code), sufficient for another engineer to implement without additional product decisions.

## Non-Goals
1. Billing provider integration.
2. Payment collection or entitlement sync.
3. UI-only enforcement.

## Enforcement Points and Precedence
## Request paths in scope
1. BFF chat stream endpoint (primary user path).
2. Gateway internal chat stream path.
3. Inbound channel fan-in path (Telegram/webhook and future channels).

## Precedence order
1. Hard system safety guards (panic/overload/drain) evaluate first.
2. Authentication/tenant binding check.
3. Quota guardrail check.
4. Session/queue processing.

Reasoning:
- Prevents expensive processing if quota is already exceeded.
- Keeps per-user and per-tenant policy decisions centralized and deterministic.

## Quota Dimensions
## Required dimensions (v1)
1. Per-user daily request cap.
2. Per-user monthly request cap.
3. Per-user short-window rate cap (e.g., per minute).

## Optional dimensions (v1.1)
1. Per-tenant daily/monthly cap.
2. Token-based caps (prompt/completion).

## Policy model
1. `disabled`: guardrail off (dev/lab only).
2. `shadow`: decisions computed and counted, requests still allowed.
3. `enforce`: requests blocked on quota violation.

## Response Contract (Enforce Mode)
When blocked, return deterministic JSON:
```json
{
  "error": "quota_exceeded",
  "reason": "user_daily_cap",
  "limit": 5,
  "used": 5,
  "window": "daily",
  "retry_after_s": 86399
}
```

HTTP status:
1. `429 Too Many Requests` for short-window rate violations.
2. `402` or `429` for daily/monthly seat limits; pick one globally and keep stable (recommended: `429` for operational consistency).

## Counter and Metrics Contract
Required counters:
1. `quota_allowed_total`
2. `quota_blocked_total`
3. `quota_blocked_by_reason_total{reason=...}`
4. `quota_shadow_blocked_total` (shadow-mode would-have-blocked)

Optional latency metric:
1. `quota_check_ms_total`

All counters must include route labels:
- `chat_stream_http`
- `chat_stream_sse`
- `webhook`
- `api`

## Config Schema Contract
Proposed config namespace:
```json
{
  "quota": {
    "mode": "disabled|shadow|enforce",
    "user_daily_cap": 5,
    "user_monthly_cap": 150,
    "user_rate_per_minute": 10,
    "tenant_daily_cap": 0,
    "tenant_monthly_cap": 0
  }
}
```

Defaults:
1. `mode=shadow` in staging.
2. `mode=enforce` in production after shadow validation.
3. `tenant_* = 0` means disabled dimension.

Override order:
1. Per-user override (if configured).
2. Tenant-level default.
3. Global default.

## Data Model and State Consistency
Persistent counters needed:
1. `(tenant_id, user_id, day_key)` request_count
2. `(tenant_id, user_id, month_key)` request_count
3. Short-window rolling bucket per user (in-memory + periodic persistence or direct DB counters).

Consistency requirements:
1. Atomic increment-and-check in authoritative store.
2. Idempotency key support to avoid double counting on retries.
3. Clock-source consistency (UTC windows).

## Shadow Mode Rollout and Promotion
## Rollout steps
1. Deploy in `shadow` mode.
2. Observe counters for 3-7 days.
3. Verify false-positive rate is acceptable.
4. Switch selected cohort to `enforce`.
5. Expand rollout gradually.

## Promotion gates
1. No unexplained block reasons.
2. Counter parity between shadow and observed behavior.
3. No incident-level user impact.

## Rollback
1. Flip `quota.mode` to `shadow` (or `disabled` emergency).
2. Keep counting to preserve diagnosis history.
3. Publish rollback reason and timestamp.

## Abuse/Breach Cases and Expected Handling
1. Direct API bypass of UI limit:
- Block via backend quota check.
2. Multi-tab burst:
- Short-window rate cap handles burst.
3. Channel fan-in (Telegram + web simultaneously):
- Shared per-user counter blocks cumulative overuse.
4. Retry storms:
- Use idempotency keys to prevent duplicate count inflation.
5. Clock skew:
- Use server-side UTC keys only.

## Implementation Acceptance Criteria
1. Same user cannot exceed configured cap by bypassing UI.
2. Shadow and enforce use identical decision logic.
3. Block responses are deterministic and machine-parseable.
4. Metrics expose allowed/blocked rates by reason and route.
5. Rollback to shadow mode is config-only, no code revert required.
