# 2026-05-21 — v1.14.18-A + v1.14.18-B Activation Audit

This is the second §14.10 activation audit. Unlike the first (2026-05-20), it did
**not** trust the "which PR targets this gap" bucketing. Two independent agents
traced the actual production call chain for every claimed-closed gap, and the
coordinator independently verified the load-bearing finding. The result changes
the prior audit's scorecard materially.

## Why this audit exists / what the prior audit missed

The 2026-05-20 audit bucketed the 19 latent-value gaps by **which sprint targets
them** and declared a gap "closed" when its PR merged. It never traced the call
chain to a runtime gate. That is a code-completion audit, not a behavioral-
activation audit — and §14.10's whole premise is "code-complete ≠ behaviorally
active." This audit closes that method gap. See §14.10 sharpening below.

## Root cause — one C3-dead gate killed three gaps

G1, G5, and G16 died at the *same line*. In `src/agent/commands.zig`
(`persistSessionSemanticSummary`), `procedural_memory.captureSession` (G1, G5)
and `promoteWMToDurableAtSessionEnd` (G16) were nested inside:

```zig
if (self.extraction_cfg.per_turn_enqueue_enabled and ...) {   // ← dead gate
```

`per_turn_enqueue_enabled` defaults `false` (`config_types.zig`) and its enabling
sites were **deleted in V1.14.7 C3**. The gate's only legitimate job is to
suppress the legacy extraction_**queue** enqueue.

History of the mis-scoping:
- **V1.13 Day 4.2** — `captureSession` placed inside the block when the gate
  defaulted **true** (C1). Correct at the time.
- **V1.14.7 C3** — gate flipped **false** to kill the enqueue path. `captureSession`
  taken down as silent collateral.
- **v1.14.18-A** — "capture-gate fix" repaired an *inner* tool-counter bug
  (`last_turn_tool_count` → `session_total_tool_count`) but never touched the
  *outer* gate.
- **v1.14.18-B** — added the WM promotion + reflection-trail capture *into the
  same dead block*.

Neither `captureSession` nor `promoteWMToDurableAtSessionEnd` writes to
`extraction_queue` — `captureSession` writes `skill_executions`, promotion writes
`durable_facts` via `mem.store`. The enqueue gate never logically applied to them.
CI stayed green throughout because every test (`promotion_reflection_pg_test.zig`)
calls these functions **directly**, bypassing the gate — proving the functions
work, never proving the agent reaches them.

## The re-activation fix — commit `b1c5c49d`

`captureSession` + `promoteWMToDurableAtSessionEnd` hoisted into a new block
*after* the gate, ungated, guarded only by extraction-state-manager availability,
failure-soft. The gated block now contains **only** the legacy extraction-queue
enqueue (correctly C3-dead). The inner `CAPTURE_TOOL_THRESHOLD` triviality filter
(skip pure-conversation sessions) is retained. No behavior of the enqueue path
changes.

Re-activates, end-to-end on the production session-end path: `skill_executions`
population, reflection-trail persistence to `assumptions_made_json`, and WM
`active_goal`/`decision` → `durable_facts` promotion — the exact substrate the
v1.14.19 sleep cycle depends on.

Verified: `zig build test -Dengines=base,sqlite,postgres -Dchannels=cli,telegram`
with `NULLALIS_POSTGRES_TEST_URL` set → exit 0, 0 failed, 0 errors logged.
**The behavioral proof is the next bench run** (see bench-gate readiness below).

Recommended follow-up (not blocking): a session-end-path integration test that
drives `persistSessionCheckpoint`/`persistSessionSemanticSummary` and asserts the
`skill_executions` + `durable_facts/transient_goal/` rows appear *without* the
flag set — closing the §14.11 Gate-C reachability proof in CI, not just at bench.

## G4 — TASK-PLANNER READ-BACK — activation plan

**Is the code there? PARTIAL BUILD.** The plan *model* is 100% complete; the
read-back-to-prompt path does not exist.

- **Exists:** `task_planner.zig` — `TaskPlan`/`TaskStep`/`StepStatus` structs,
  `parseTaskPlan`, `markStepDone`/`advanceStep`/`isComplete`. `root.zig` populates
  `active_task_plan` (a turn-loop local) and consumes it *only* for observer
  telemetry (`emitStepProgress/Done/Complete`).
- **Missing:** (1) a `renderPlanBlock(*const TaskPlan) ![]u8` — no `render*`/
  `format*` function exists; (2) a `PromptContext.task_plan_block` field;
  (3) a render clause in `buildVolatileSystemPrompt`; (4) `active_task_plan` must
  move from a `turnOutcome` loop-local to an `Agent` field so `context_engine.
  assemble()` can reach it (mirror `narration_ring_buffer`).
- **Reuse:** the v1.14.18-B `<recent_thoughts>`/`<known_weakness>` block pattern
  (`context_engine.assemble` → `PromptContext` field → `buildVolatileSystemPrompt`
  render clause) is a directly reusable template.

Activation steps: (1) promote `active_task_plan` to an `Agent` field; (2) write
`task_planner.renderPlanBlock` (`<task_plan>` with per-step `[done]`/`[→ current]`/
`[pending]` via `StepStatus.toSlice()`, failure-soft); (3) add `task_plan_block`
to `PromptContext`; (4) build it in `assemble()` with the defer-free + empty-guard
pattern; (5) render it in `buildVolatileSystemPrompt` at a fixed recall-stack
position (cache-stability invariant); (6) unit test + ordering test.

## G11 — BRAIN_GRAPH / RECALL ESCALATION — activation plan

**Is the code there? WIRING ONLY.** Every building block exists.

- **Exists:** `goal_loop` detects `stuck` and reaches a decision point
  (`root.zig:3790`); `executeTool` already runs a code-constructed `ParsedToolCall`
  (proven by `executeApprovedPendingTool`); the `forced_follow_through`
  synthetic-message-injection pattern is already in production
  (`root.zig:3219,3822`); `memory_recall` and `brain_graph` are registered,
  dispatchable tools.
- **Missing:** currently `stuck` and `met` both → identical `break`. No branch
  distinguishes them; no escalation action.
- **Smallest real change:** on `stuck`, instead of `break`, reuse the
  `forced_follow_through` pattern — append a SYSTEM message instructing a
  `memory_recall`/`brain_graph` call and `continue` the loop, gated by a one-shot
  counter (mirror `forced_follow_through_count < 2`). ~15 lines, confined to
  `root.zig`, no new module. (Stronger variant: construct the `ParsedToolCall`
  and run it directly via `executeTool`, guaranteeing recall actually fires.)

Activation steps: (1) split the verdict branch at `root.zig:3790` — `.met` keeps
`should_exit_goal_loop`, `.stuck` sets a new `escalate_on_stuck` flag; (2) add a
`stuck_escalation_count` guard; (3) before the exit `break`, if escalating and
under the one-shot cap, inject the recall directive and `continue`; (4) record
the escalation in `active_reflection_trail`; (5) `turnOutcome` integration test —
mock `goal_status="stuck"`, assert a recall invocation follows rather than an
immediate exit.

## §14.10 discipline sharpening (applied to AGENTS.md this session)

The first activation audit's method failure — bucket-by-PR instead of
trace-the-call-chain — is now closed in the discipline itself. §14.10 / §14.11
Gate C now require an activation claim to be backed by: (a) a grep-identified
**production** caller, AND (b) proof that caller is **reachable in default
config** — no gate defaulting it off, no test-only call path. "A PR targets it"
is not evidence of activation.
