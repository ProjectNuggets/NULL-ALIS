# nullALIS v1.1 Next Steps

Date: 2026-03-18
Status: planned (post-v0.1 baseline)

This document defines the next execution batch after the `v0.1` baseline freeze.

## Scope for v1.1

### P0 — Together API v0.1 hardening

Goal:
- Make Together behavior deterministic for chat and embedding paths under provider instability.

Work:
- Normalize provider error mapping and retry classes for Together-compatible requests.
- Confirm reasoning knobs and model-specific generation fields behave as expected.
- Add contract tests for request/response shape and fallback behavior.

Acceptance:
- No raw provider error leakage to product surfaces.
- Retry/fallback behavior is deterministic and test-backed.

### P0 — TTS end-to-end product wiring

Goal:
- Move from prep-only confidence to stable user-visible voice behavior.

Work:
- Keep current `tts_prepare` stage and ensure channel delivery semantics are explicit.
- Verify Telegram audio attachment path, failure mapping, and cleanup behavior.
- Keep `/settings` and runtime behavior aligned for `voice_replies`.

Acceptance:
- TTS behavior is predictable per mode and channel.
- Failures are explicit and non-blocking for text replies.

### P0 — Session lane policy finalization

Goal:
- Preserve correctness-first serial behavior while preventing avoidable `main` lane contention.

Work:
- Keep explicit session-key requirement strict.
- Enforce intended routing policy across app and channel ingress.
- Add smoke gates for lane ownership, lane distribution, and rejection reasons.

Acceptance:
- No accidental fallback to wrong lane.
- Lane behavior is observable and reproducible under burst tests.

### P1 — Compaction/summarizer hot-path guard

Goal:
- Prevent hidden long stalls in interactive turns.

Work:
- Keep interactive pre-provider compaction trim-first and bounded.
- Keep heavy summarization on lifecycle boundaries, best-effort.
- Separate telemetry for turn compaction vs lifecycle summarizer.

Acceptance:
- No recurring long pre-provider stalls in normal chat traffic.

### P1 — Runtime observability gate for latency

Goal:
- Convert performance debugging from guesswork to measured release gates.

Work:
- Use `llm_first_token`, `llm_first_token_upper_bound`, and `turn.profile` in ops dashboards/runbooks.
- Add canary gates based on stage attribution and tail latency.

Acceptance:
- Regressions are detected by gate thresholds, not operator intuition.

## Upstream Intake Policy (v1.1 window)

Allowed:
- Surgical cherry-picks with narrow blast radius and clear tests.

Deferred:
- Large multi-file behavioral rewrites (for example memory isolation hardening commit class like `8a967ea`) unless split into staged migrations with diagnostics-first rollout.

## Execution Order

1. Together API v0.1 hardening
2. TTS end-to-end wiring
3. Session lane policy finalization
4. Compaction/summarizer hot-path guard
5. Observability release gates
