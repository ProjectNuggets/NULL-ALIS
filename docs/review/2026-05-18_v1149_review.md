# V1.14.9 Code Review — Episode-Based Boundary Extraction

**Reviewed:** 2026-05-18
**Depth:** Deep (cross-file: chunker → runner → merger → persist; also verified Pass A / Pass C / session-end / manualCompactHistory call sites)
**Status:** Issues found. No BLOCKERs. The pipeline is structurally sound and matches the documented design. The findings below are real but small.

## Summary

The chunker / runner / merger / telemetry split is well-architected and the lifetimes are mostly correct. Tests cover the happy paths and the most important skip paths. The two HIGH findings are both about *propagation gaps* — V1.14.9 wired Pass A + session-end but missed `manualCompactHistory`, and the runner's parallel allocator-thread-safety assumption is asserted in a doc comment but not enforced or even runtime-checked. Several MEDIUM findings document real (small) leak paths on rare error branches. The pipeline as it stands is safe to merge; the HIGH items should be addressed in a follow-up before the next public-beta cut.

## HIGH

### H-01: `manualCompactHistory` does NOT propagate extraction judge fields

**File:** `src/agent/root.zig:999-1012`

The whole point of the V1.14.9 root.zig change was to fix the silent skip caused by missing `extraction_judge_provider` / `extraction_judge_model_name`. `autoCompactHistory` (lines 908-985) now propagates them correctly. But `manualCompactHistory` builds its own CompactionConfig literal (line 1002) and omits the extraction fields entirely:

```zig
return compaction.manualCompactHistory(self.allocator, &self.history, compact_provider, compact_model, .{
    .keep_recent = self.compaction_keep_recent,
    // ...
    .workspace_dir = self.workspace_dir,
    // NOT propagated:
    //   .extraction_judge_provider
    //   .extraction_judge_model_name
    //   .extraction_state_mgr
    //   .extraction_user_id
    //   .extraction_coref_embed
    //   .archive_memory / .archive_session_id / .archive_mem_rt
});
```

Any user-triggered manual compaction (e.g., `/compact` command) will silently skip the unified extractor — exactly the regression V1.14.9 was meant to close, but only for the auto path. Pass C inside manualCompactHistory is the largest, most extraction-valuable boundary (operator-initiated, usually after a long session).

**Fix:** Mirror the autoCompactHistory propagation. Probably worth extracting a `buildCompactionConfig(self)` helper to keep them in sync going forward.

```zig
fn buildCompactionConfig(self: *Agent) compaction.CompactionConfig {
    return .{
        .keep_recent = self.compaction_keep_recent,
        .max_summary_chars = self.compaction_max_summary_chars,
        .max_source_chars = self.compaction_max_source_chars,
        .token_limit = self.token_limit,
        .max_tokens = self.max_tokens,
        .message_timeout_secs = self.message_timeout_secs,
        .max_history_messages = self.max_history_messages,
        .workspace_dir = self.workspace_dir,
        .archive_memory = self.mem,
        .archive_session_id = self.memory_session_id,
        .archive_mem_rt = self.mem_rt,
        .extraction_state_mgr = self.extraction_state_mgr,
        .extraction_user_id = self.extraction_user_id,
        .extraction_coref_embed = self.extraction_coref_embed,
        .extraction_judge_provider = self.extraction_judge_provider,
        .extraction_judge_model_name = if (self.extraction_judge_model_name.len > 0)
            self.extraction_judge_model_name else null,
    };
}
```

Also check `forceCompressHistory` (line 1016) — it calls `forceCompressHistoryWithArchive` directly with a different signature; no CompactionConfig is built, so this drop path has no extractor wire at all. That may be intentional (emergency context-recovery skips extraction by design), but the docstring should say so explicitly given V1.14.9's "extraction at every boundary" framing.

### H-02: Parallel allocator thread-safety is doc-asserted, not enforced

**File:** `src/agent/extraction/runner.zig:307-343, 347-358`

The runner comment at line 307-312 says: "The agent's primary allocator is expected to be thread-safe (std.heap.GeneralPurposeAllocator with `.thread_safe = true` OR the C allocator)." This is correct for production (`main.zig:381` uses `std.heap.smp_allocator`), but:

1. **No runtime guard.** If a future caller wires `extractAtBoundary` from a non-thread-safe context (e.g., a test using `std.testing.allocator` which is DebugAllocator and IS NOT thread-safe for concurrent allocations from multiple OS threads), the runner will spawn 8 threads that race on the allocator. Failure mode = use-after-free or hashmap corruption, hard to debug.

2. **No test coverage for the parallel path.** The 4 runner skip-path tests cover empty window / empty model / no state_mgr / zero edges. None exercise the parallel execution path, so this assumption has never been tested.

3. **Footgun risk in commands.zig session-end.** `commands.zig:1484` constructs an ExtractionContext using `self.allocator`. That allocator IS smp_allocator in production (so safe), but the contract is implicit.

**Fix (cheap):** Add a runtime guard that falls back to sequential if `n_jobs > 1` and the allocator vtable doesn't match a known-thread-safe one. There's no public API for "is this allocator thread-safe?" so the pragmatic fix is:

   a. Document the contract more loudly on `ExtractionContext.extraction_concurrency` (not just in a deep-buried function comment).
   b. Add a debug-mode assertion in `extractEpisodesParallel`:
   ```zig
   if (builtin.mode == .Debug) {
       std.debug.assert(@intFromPtr(allocator.vtable) != @intFromPtr(std.testing.allocator.vtable));
   }
   ```
   c. Add a test that runs the parallel path with `std.heap.smp_allocator` and 3+ episodes (use a fake provider).

The session-end / Pass A / Pass C call sites all pass the same agent allocator, so production is safe today — but one wrong caller change introduces undefined behavior with no compile-time signal.

## MEDIUM

### M-01: Merger leaks `raw_dup` keys if phase 2/3 errors after phase 1 inserts

**File:** `src/agent/extraction/merger.zig:73-181`

```zig
var canonical_map: std.StringHashMapUnmanaged([]const u8) = .empty;
defer canonical_map.deinit(allocator);  // line 74 — only frees map storage
// ...
const raw_dup = try allocator.dupe(u8, e.name);
try canonical_map.put(allocator, raw_dup, canonical);  // line 111-112
```

If any error fires after line 112 but before the explicit cleanup at line 177-180 (e.g., the `seen_edges.put` at line 147 OOMs, or `out_entities.append` at line 171 OOMs), the deferred `canonical_map.deinit(allocator)` runs but only frees the hashmap internals — every `raw_dup` allocation is leaked.

Same shape applies to the `embeds` map (line 86-93): if a put fails after a successful `allocator.dupe(f32, ce)`, the cleanup iterates current entries but the just-failed entry's owned_embed is also leaked.

**Severity rationale:** Only fires on OOM or downstream allocation failure during merge — a boundary path that runs at most a few times per hour. But the test allocator (`std.testing.allocator`) WILL flag this in any test that injects an OOM into phase 2/3; we just don't have such a test today.

**Fix:** Add proper errdefer that walks the current contents of canonical_map and frees keys. Slightly cleaner: convert the loop body to a `findOrInsertCanonical`-style sub-function that owns its dup with proper errdefer.

```zig
errdefer {
    var it = canonical_map.iterator();
    while (it.next()) |kv| allocator.free(kv.key_ptr.*);
}
```
Place this errdefer right after the `defer canonical_map.deinit(allocator)`.

Same fix for the embeds map's value allocations.

### M-02: `findOrInsertCanonical` leaks new_entity on subsequent put failure

**File:** `src/agent/extraction/merger.zig:233-241, 254-260`

```zig
const new_entity = Entity{
    .name = try allocator.dupe(u8, e.name),     // succeeds
    .entity_type = e.entity_type,
};
const owned_embed = try allocator.dupe(f32, ce);   // succeeds
try canonical_entities.put(allocator, new_entity.name, new_entity);  // ← if this OOMs
try embeds.put(allocator, new_entity.name, owned_embed);              // ← or if this OOMs
return new_entity.name;
```

If either `put` errors:
- The `new_entity.name` is leaked (it's not in any owning collection yet).
- The `owned_embed` may be leaked if the second put failed.

Same pattern on the case-insensitive fallback path (line 254-260).

**Fix:** Standard Zig hashmap-insert pattern — `getOrPut` first, then populate the value slot only after the put succeeds:

```zig
const gop = try canonical_entities.getOrPut(allocator, dup_name);
errdefer if (!gop.found_existing) _ = canonical_entities.remove(dup_name);
if (!gop.found_existing) {
    gop.value_ptr.* = Entity{ .name = dup_name, .entity_type = e.entity_type };
}
```

Realistically: low-probability path. Note for hardening, not for tonight's merge.

### M-03: Per-episode transcript has no per-message OR total cap

**File:** `src/agent/extraction/runner.zig:381-400`

`buildEpisodeTranscript` is intentionally cap-free (line 377-380 comments it out): "The episode is small enough by construction to fit in the LLM's coherence sweet spot." But this assumes the chunker never lets a single oversized message into an episode. Let's check that assumption:

The chunker's `hard_cap` check (chunker.zig:134) only fires when `current_tokens + msg_tokens > max_episode_tokens` AND `non_empty` (current_start < idx). The `non_empty` guard means **the first message in any episode is never split out**, no matter how large it is. So a single 50KB tool result at the start of an episode produces a 50KB transcript shipped to the LLM, blowing past `max_episode_tokens=8000`.

Empirically rare (most messages are <2KB), but the F5 conv-43 session that motivated this work had tool-call results in the 5-15KB range. With Pass A drop windows likely containing tool messages, this can happen.

**Severity rationale:** Not a leak / not a crash — just a latent runaway-cost-and-coherence-degradation risk. The very failure mode V1.14.9 was meant to prevent (incoherent giant input) can recur if one message is large enough.

**Fix options (pick one):**
1. **Cap the per-message content in buildEpisodeTranscript** the same way buildTranscript does (`if (msg.content.len > 2000) msg.content[0..2000] else msg.content`). Simple, matches the established pattern.
2. **Add a hard split inside the chunker** for single-message episodes whose token count alone exceeds `max_episode_tokens`. More work; better fidelity.

Option 1 in 3 lines is the right tradeoff here.

### M-04: Telemetry `extraction_ms_total` and `hydration_ms` are stubbed to 0

**File:** `src/agent/extraction/runner.zig:233-248`

```zig
// Per-call timings TBD — full Phase 5+ instrumentation needs
// call-level timing injected into runExtractionOnEpisode +
// runHydrationCall. Stub for now; not blocking on density.
.extraction_ms_total = 0,
.hydration_ms = 0,
```

The telemetry module bills itself as the "single entry point" for boundary metrics (telemetry.zig:62), with timing fields prominently in the schema (telemetry.zig:48-49). Shipping V1.14.9 with these as constant-0 means downstream alerting (provider-contention slow-call detector, latency regression diff over time) has nothing to read.

The two log lines are also duplicative now: `boundary.metrics` (telemetry.zig:65) and `boundary.complete` (runner.zig:253) report overlapping data — comment says boundary.complete is "kept for grep-compat" but it'll drift over time.

**Fix:** Wrap `runExtractionOnEpisode` and `runHydrationCall` in `std.time.Timer`. Sum episode times for `extraction_ms_total`. Trivial — 4-6 lines. Then remove the boundary.complete line (or convert it to a debug log) once consumers migrate.

## LOW

### L-01: `inferBoundaryKind` mis-tags any future non-Pass-A caller that sets `enable_hydration=false`

**File:** `src/agent/extraction/runner.zig:275-279`

```zig
fn inferBoundaryKind(ctx: ExtractionContext) telemetry.BoundaryKind {
    if (!ctx.enable_hydration) return .pass_a;
    if (ctx.archive_mem == null) return .pass_c;
    return .session_end;
}
```

Already noted by the author. As long as `enable_hydration=false` is treated as a proxy for "Pass A", any future caller that wants extraction-only for a different reason will appear in metrics as `pass_a`. Not load-bearing per the comment, but for clean metrics, the right fix is an explicit `kind: ?BoundaryKind = null` field on ExtractionContext that callers set. Three callers, three small diffs.

### L-02: Chunker's max_episodes sampling drops the middle without telemetry on what was dropped

**File:** `src/agent/extraction/chunker.zig:177-194`

The warn log notes the sampling fired (good) but doesn't capture which episode-index range was dropped or how many tokens those middle episodes accounted for. If F5 conv-43-style sessions hit the cap, we'll know it fired but not what we lost.

**Fix:** Add `dropped_episodes={d} dropped_tokens={d}` fields to the warn log. Two extra accumulators in the if branch.

### L-03: Chunker emits zero-token episodes if all messages have empty content

**File:** `src/agent/extraction/chunker.zig:131, 166-173`

`non_empty = current_start < idx` means "the current episode has at least one message added." It does NOT mean "the current episode has any token content." Plausible scenario: a window of empty-content tool calls + role markers. `current_tokens` stays at 0 across many messages, no boundary fires, the tail flush at line 166 emits one episode with `estimated_tokens=0`. The runner then ships a transcript of role headers with no content to the LLM, which will return zero extractions (cost: 1 LLM call for nothing).

**Fix:** In `runExtractionOnEpisode`, short-circuit when `episode.estimated_tokens == 0` (or when transcript is whitespace-only after build). Cheap defensive guard.

### L-04: `slotIntentToWorkingMemoryType` silently drops `.preference`

**File:** `src/agent/extraction/runner.zig:45-55`

```zig
.preference => null, // preferences stay in canonical memory only
```

Returning null means the persist layer treats it as "no slot intent," which is correct behavior, but it also means an LLM-tagged preference fact loses its `slot_intent=preference` signal entirely on the way through. If anything downstream ever wants to filter on preference facts, the tag is gone.

Not actionable today — the comment is explicit and matches the schema design — but flag this if the working_memory layer later grows a preference slot type.

### N-02: Two slightly different "build a transcript" functions diverge

**File:** `src/agent/extraction/runner.zig:381 vs 420`

`buildEpisodeTranscript` and `buildTranscript` differ only in (a) cap behavior and (b) `*msg` vs `msg` iteration. Both write `ROLE: content\n`. Worth either:
- A shared `appendRoleLine(buf, allocator, msg, max_per_msg: ?usize)` helper, or
- A doc comment on the diff pointing out it's intentional (the cap difference is load-bearing).

Currently the divergence is subtle enough that a future "fix" to one will likely miss the other.

### N-03: Magic number `5` in chunker sampling

**File:** `src/agent/extraction/chunker.zig:183`

```zig
const keep_head: u32 = 5;
const keep_tail: u32 = max_episodes - keep_head;
```

`5` is documented in the module doc ("first 5 + last (cap-5)") but the function signature doesn't expose it. If `max_episodes < 5` you'd get `keep_tail` underflow. The assert at line 109 (`max_episodes > 0`) doesn't prevent it. Tighten to `assert(max_episodes >= 5)` OR make `keep_head` a parameter OR clamp: `const keep_head = @min(5, max_episodes / 2);`.

### N-04: `recordBoundary` failure-pct integer division loses precision at small N

**File:** `src/agent/extraction/telemetry.zig:99-105`

```zig
if (metrics.episodes_chunked > 2) {
    const failure_pct = (@as(u64, metrics.episodes_extracted_failed) * 100) / metrics.episodes_chunked;
```

`episodes_chunked > 2` means N≥3, so resolution is ~33%. At N=3, 1 failure = 33% (alert off), 2 failures = 66% (alert on). Fine for the intent, but the docstring says ">30% failure" which a 33% bucket only barely crosses. Cosmetic; the integer math is correct.

## What was NOT a concern after review

- **Thread safety of merger StringHashMap/cosine loop** (your concern #1): Merger is only called sequentially in `extractAtBoundary` after `wg.wait()` returns (runner.zig:185-194). No race possible in the current call graph. Confirmed.
- **Episode lifetime vs window content** (your concern #2): The chunker's contract (chunker.zig:97-100) is explicit, the slice fields point into `window`, and every caller (extractAtBoundary line 148 + 158) frees only the `[]Episode` array via `allocator.free(episodes)`. Caller is the same function that owns `window`. Lifetime is safe.
- **Worker function allocator** (your concern #3): smp_allocator is thread-safe; curl-subprocess provider is process-isolated. See H-02 for the doc-vs-enforcement gap.
- **Pass A wire fix in root.zig** (your concern #6): autoCompactHistory propagation is correct. The miss is manualCompactHistory — see H-01.
