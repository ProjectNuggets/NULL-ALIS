# v0.1 Low-Hanging Clean Patches

This document tracks the reliability-first, low-risk patches that were implemented without widening product scope.

## Implemented

1. `runtime_info.integrations` Composio readiness
- Replaced static `connected_accounts_state="unknown"` with best-effort readiness signals.
- Added toolkit probes for:
  - `gmail`
  - `googledrive`
  - `googlecalendar`
- Added per-entity connected-account readiness fields:
  - `gmail_connected`
  - `google_drive_connected`
  - `google_calendar_connected`
- Added `api_reachable` and explicit account state labels:
  - `disabled`
  - `not_configured`
  - `connected`
  - `not_connected`
  - `unverified`
  - `api_unreachable`

2. `/runtime` operator output hardening
- `/runtime` now prints concise readiness lines from `runtime_info` summary + integrations:
  - state/scheduler/degraded
  - turn/session context
  - Telegram configured/connected
  - Composio enabled/configured/entity/account state
  - Gmail/Drive/Calendar readiness line

3. Integration preflight script
- Added `scripts/preflight-integrations.sh` for one-pass checks:
  - startup self-check health (`state_effective`, `scheduler_backend`, `degraded`)
  - Composio toolkit availability probe
  - per-entity connected-account readiness probe

## Scope Guardrails

- No new broad feature surfaces were added.
- No background auth flow changes were introduced.
- No scheduler/autonomy behavior was expanded.
- Changes are diagnostics/operator readiness improvements only.

## Deferred UX Discussion

- `composio connect` remains blocked for background origins (`heartbeat`/`scheduler`/etc) and is only allowed in explicit user turns.
- Rationale for v0.1: prevent autonomous auth loops, accidental account-linking in the wrong session/entity, and unsolicited connect prompts.
- Follow-up item: design a user-visible "auth needed" handoff UX so background tasks can request connect cleanly without performing auth themselves.
