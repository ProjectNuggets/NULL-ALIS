# v0.2 Operational Model Runbook (Agent-Executable)

Status: execution plan  
Date: 2026-03-12  
Owner lane: platform/runtime  
Scope: scalable multi-user operations with bounded noisy-neighbor impact

## 1) Goal

Ship a production operating model where:
1. User A does not materially degrade User B experience.
2. Conversational continuity is preserved (`main/thread/task/cron` lanes).
3. Scaling is horizontal and predictable.
4. Security and tenant isolation remain strict.

This runbook is start-to-finish and can be executed by an AI agent.

## 2) Current State (Ground Truth)

From current code:
1. One gateway process handles many users with a worker pool and request queue:
- `gateway.max_workers`, `gateway.max_queued_requests` in `src/config_types.zig`
- acceptor + worker queue in `src/gateway.zig`
2. Tenant runtime is per-user in-memory and cached:
- `tenant.runtime_cache_max_users`, `tenant.runtime_idle_ttl_secs`
- `getTenantRuntime(...)` in `src/gateway.zig`
3. Turn serialization is per `session_key` lane:
- `session.mutex` in `src/session.zig`
4. Session key ownership is enforced:
- `/api/v1/chat/stream` validation in `src/gateway.zig`
5. Canonical key helpers exist:
- `userMainSessionKey`, `userCronSessionKey` in `src/zaki_session.zig`
6. Load harness supports lane strategies:
- `scripts/load-burst.py`

Operational implication:
1. Correctness is good.
2. Shared runtime resources still couple users under high load.
3. Tail latency rises as concurrent active turns rise.

## 3) Target Operating Model

## 3.1 Standard Tier (Default)
1. Deploy multiple gateway cells (pods/groups).
2. Sticky route user traffic by `user_id` to one cell.
3. Keep existing lane model inside each cell:
- `main` for continuity
- `thread` for active conversation lanes
- `task` for independent concurrent work
- `cron` for scheduler/proactive jobs
4. Enforce per-cell hard quotas:
- max in-flight turns
- max queue depth
- DB pool cap
- provider concurrency budget
- tool concurrency budget

Expected result:
1. Noisy-neighbor impact becomes bounded per cell.
2. Scale by adding cells, not by overloading one process.

## 3.2 Premium Tier
1. Dedicated tenant cell (or dedicated tenant stack).
2. Dedicated runtime quotas and provider budget.
3. Optional dedicated DB schema/cluster according to contract.

Expected result:
1. Strongest isolation.
2. Highest infra and ops cost.

## 4) SLOs and Hard Gates

Interactive turn SLO targets (per tier):
1. `p50 < 8s`
2. `p95 < 20s`
3. `p99 < 35s`
4. error rate `< 1%`

Isolation gate:
1. In noisy-neighbor test, unaffected cohort p95 degradation `<= 15%`.

Overload policy:
1. Prefer early bounded failure (`429/503 + retry hint`) over unbounded queueing.

## 5) Phase Plan (Execution Order)

## Phase 0: Baseline Freeze (mandatory)
Steps:
1. Record git SHA and active config profile.
2. Run:
```bash

3. Capture baseline canary artifacts:

python3 scripts/load-burst.py --token <TOKEN> --mode multi-user --users 100 --requests 100 --lane-strategy main_only --json > docs/reports/<date>/baseline-100.json
```

Exit criteria:
1. Build and tests pass.
2. Baseline reports committed in `docs/reports/<date>/`.

## Phase 1: Client Session-Lane Discipline
Steps:
1. Enforce app client contract:
- always send `X-Zaki-User-Id`
- always send explicit `session_key`
2. Adopt lane policy from `docs/session-key-policy.md`.
3. Add per-lane in-flight UX guard in clients (no double-send on same lane unless explicit fork).

Exit criteria:
1. Same-user overlap on split lanes improves p95/p99 by `>= 30%` vs `main_only`.
2. No session ownership regression errors.

## Phase 2: Cell Routing (Sticky by User)
Steps:
1. Introduce user-sticky routing at ingress/LB:
- primary key: `X-Zaki-User-Id`
- fallback: auth user id extracted at edge
2. Keep current gateway internals unchanged.
3. Validate stickiness with request tracing logs.

Implementation options:
1. Ingress hash routing by header.
2. Edge router service with consistent hash ring.

Exit criteria:
1. Same user consistently lands on same cell in steady state.
2. Cross-cell thrash rate is near zero outside deploy events.

## Phase 3: Per-Cell Quotas and Backpressure
Steps:
1. Configure per-cell limits:
- `gateway.max_workers`
- `gateway.max_queued_requests`
- Postgres pool caps in state config
2. Add/enable backpressure policy:
- reject when queue cap reached
- return retry hint
3. Separate interactive and background pressure budgets (worker slots or queue partitions).

Exit criteria:
1. No unbounded request queue growth.
2. Under stress, failure is bounded and explicit, not silent timeout drift.

## Phase 4: Provider Throughput Isolation
Steps:
1. Keep topology:
- chat primary = OpenRouter
- chat fallback = Together
- embeddings = Together
2. Add per-cell provider concurrency/token budgets.
3. Add budget telemetry fields to diagnostics and alerting.

Exit criteria:
1. Provider-side saturation in one cell does not collapse all cells.
2. Fallback chain behavior remains correct.

## Phase 5: Canary Rollout and Promotion Gates
Steps:
1. Rollout sequence:
- 10% cells
- 25%
- 50%
- 75%
- 100%
2. At each step, run:
```bash
python3 scripts/load-burst.py --token <TOKEN> --mode multi-user --users 20 --requests 20 --lane-strategy main_only --json > docs/reports/<date>/canary-20.json
python3 scripts/load-burst.py --token <TOKEN> --mode multi-user --users 50 --requests 50 --lane-strategy mixed_real --json > docs/reports/<date>/canary-50-mixed.json
python3 scripts/load-burst.py --token <TOKEN> --mode multi-user --users 100 --requests 100 --lane-strategy main_only --json > docs/reports/<date>/canary-100.json
```

Promotion gate:
1. SLOs pass.
2. Error budget pass.
3. Isolation gate pass.

Rollback trigger:
1. p95 breach for two consecutive canary windows.
2. error rate > 1%.
3. sustained queue saturation.

## Phase 6: Premium Dedicated Tier Activation
Steps:
1. Add deployment mode `dedicated_tenant` at control plane.
2. Provision dedicated runtime quota pack per tenant.
3. Route premium tenants to dedicated pool/namespace.
4. Publish premium SLO and limits.

Exit criteria:
1. Premium tenant performance no longer depends on standard-tier tenant load.

## 8) Security and Isolation Constraints (Non-Negotiable)

1. Preserve `session_key` user ownership checks.
2. Preserve tenant ownership lock behavior across nodes.
3. No broadening of background OAuth connect permissions.
4. Keep runtime truth user-scoped on operator surfaces.

## 9) Capacity Planning Starter (Initial, recalibrate with live data)

Assume heavy interactive turns:
1. one cell handles roughly 20-40 active heavy users before p95 risk increases.
2. scale by adding cells, not by increasing one cell indefinitely.

Planning equation:
1. `required_cells = ceil(active_heavy_users / target_heavy_users_per_cell)`
2. keep 20-30% spare headroom for burst and failover.
