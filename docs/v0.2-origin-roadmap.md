# v0.2 Turn Origin Roadmap

## Current Runtime Status (v0.1)

`TurnOrigin` enum values in runtime:
- `user`
- `heartbeat`
- `scheduler`
- `wake`
- `proactive`

Wired in execution paths today:
- `user`: default session turns
- `heartbeat`: heartbeat job turns (`job.id == "heartbeat"`)
- `scheduler`: non-heartbeat cron/scheduler turns

Reserved (defined but not assigned by runtime paths yet):
- `wake`
- `proactive`

## Why `wake` and `proactive` exist

They are forward-compatible origin labels for policy and observability:
- finer tool policy per autonomous source
- cleaner ops analytics by source type
- clearer user-visible shaping for system-origin messages

## v0.2 Wiring Plan

1. Wire `wake`
- Set origin to `wake` when a turn is explicitly triggered through the wake queue (`/internal/wake-heartbeat` and cron `next_heartbeat` wake flow).
- Keep `heartbeat` for interval-driven sweeps.

2. Wire `proactive`
- Assign origin to autonomous outbound turns that are not direct scheduler/heartbeat checks (future bus-driven proactive dispatch).
- Keep strict default policy: no auth/connect flows, no unconstrained shell/network exploration.

3. Preserve backward compatibility
- Existing behavior and policy for `user`, `heartbeat`, `scheduler` remains unchanged.
- Metrics/logs should expose origin explicitly to reduce ambiguity during incidents.

## Scope Notes

- No additional origin values are currently planned beyond these five.
- If a new autonomous class appears, add origin only with a concrete execution path and policy rules.
