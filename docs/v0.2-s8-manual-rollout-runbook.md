# v0.2 S8 Manual Rollout Runbook (Staged Strict Identity Mapping)

Status: active (operator-run)  
Applies to: tenant + Postgres deployments using staged strict identity mapping  
Last updated: 2026-03-12

## 1) Purpose

This runbook defines the exact manual procedure for S8 staged strict rollout.

Goals:
1. expand strict identity mapping channel-by-channel
2. keep user-impact bounded with canary evidence at each stage
3. guarantee deterministic rollback using config-only policy changes

Non-goals:
1. no automation controller in this phase
2. no API contract changes
3. no schema changes

## 2) Required Config Knobs

Set and verify:
1. `tenant.identity_mapping_enforcement = "staged_strict"`
2. `tenant.identity_mapping_strict_channels = [...]`

Optional (do not tune unless needed):
1. `tenant.identity_mapping_positive_ttl_secs` (default `300`)
2. `tenant.identity_mapping_negative_ttl_secs` (default `30`)

## 3) Stage Order

Promotion order is conservative and fixed:
1. Stage 1: `telegram`
2. Stage 2A: `slack`
3. Stage 2B: `line`
4. Stage 2C: `lark`
5. Stage 2D: `whatsapp`
6. Stage 3: remaining inbound channels (one channel at a time)

Rule:
1. promote only one channel set per stage window
2. do not combine two pending stages in one promotion decision

## 4) Pre-Stage Checklist

Before every stage:
1. capture `/internal/diagnostics` baseline snapshot
2. capture runtime status (`nullalis status --user-id <id>` and `nullalis doctor --user-id <id>`)
3. verify mapping coverage for target channel(s):
- mapped users coverage list
- unmapped backlog list
4. verify no unresolved cross-user leakage alerts
5. verify current canary posture and cached runtime posture are unchanged from prior stage

## 5) Canary Commands and Artifacts

Run canaries with fixed posture and save JSON output:

```bash
python3 scripts/load-burst.py \
  --url http://127.0.0.1:3000/api/v1/chat/stream \
  --token "$NULLALIS_INTERNAL_TOKEN" \
  --mode multi-user \
  --users 20 \
  --lane-strategy main_only \
  --timeout-secs 300 \
  --json > docs/reports/<date>-backbone/s8-stage-<n>-20.json

python3 scripts/load-burst.py \
  --url http://127.0.0.1:3000/api/v1/chat/stream \
  --token "$NULLALIS_INTERNAL_TOKEN" \
  --mode multi-user \
  --users 50 \
  --lane-strategy mixed_real \
  --timeout-secs 300 \
  --json > docs/reports/<date>-backbone/s8-stage-<n>-50.json
```

If stage risk is medium/high, run 100-user sample too:

```bash
python3 scripts/load-burst.py \
  --url http://127.0.0.1:3000/api/v1/chat/stream \
  --token "$NULLALIS_INTERNAL_TOKEN" \
  --mode multi-user \
  --users 100 \
  --lane-strategy main_only \
  --timeout-secs 360 \
  --json > docs/reports/<date>-backbone/s8-stage-<n>-100.json
```

Required markdown interpretation:
1. `docs/reports/<date>-backbone/s8-stage-<n>.md`

## 6) Promotion Gates (Hard)

A stage can be promoted only if all pass:
1. error rate `< 1%`
2. strict rejects are expected and explainable (mapped to onboarding/mapping gaps)
3. no cross-user leakage evidence
4. no `/api/v1/chat/stream` contract regression
5. mandatory build/test gates pass:
- `zig build test --summary all`
- `zig build -Dengines=base,sqlite,postgres`

Decision outcomes:
1. `GO` — promote next stage
2. `HOLD` — keep current stage, run remediation
3. `ROLLBACK` — remove current stage channel(s) from strict list

## 7) Rollback Procedure (Config-Only First)

If any hard gate fails:
1. edit config: remove failing channel(s) from `tenant.identity_mapping_strict_channels`
2. keep `tenant.identity_mapping_enforcement = "staged_strict"`
3. restart gateway/daemon
4. capture post-rollback diagnostics snapshot
5. publish rollback report with reason and evidence links

Do not start with code revert unless config rollback fails to stabilize.

## 8) Required Stage Report Template

Each stage report must include:
1. stage number and channel set
2. exact config before/after
3. canary artifact links (`.json`, `.md`)
4. strict reject reason distribution
5. mapping coverage delta for promoted channels
6. decision (`GO/HOLD/ROLLBACK`)
7. owner + timestamp

## 9) Operational Defaults

1. keep conservative staged progression as default policy
2. keep TTL defaults unchanged during rollout unless incident requires tuning
3. never promote two stages in one window
4. if evidence is incomplete, decision is `HOLD`
