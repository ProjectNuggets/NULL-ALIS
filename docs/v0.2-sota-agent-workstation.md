# v0.2 SOTA Agent Workstation

Status: Active execution plan  
Owner: Program branch `program/v0.2-sota-exec`  
Last updated: 2026-03-10

## Purpose

This is the workstation-grade execution spec for a 6-agent SOTA program.  
It is written so any implementation agent can execute a package with minimal ambiguity, strict regression control, and concrete completion gates.

Program constraints:
1. Balanced delivery: reliability + proactivity + UX in parallel.
2. Hard gates + canary before any SOTA claim.
3. No agent can claim complete without full validations.

4. Rebase lane branch onto latest `program/v0.2-sota-exec` before final validation.
5. Include evidence section in PR description with command outputs.

## A1 — Runtime Truth + Scheduler Determinism

Branch: `program/v0.2-sota-exec-a1-runtime-truth`  
Primary objective: eliminate contradiction surfaces and scheduler ambiguity.

Primary files to touch:
1. `src/diagnostics/runtime_truth.zig`
2. `src/status.zig`
3. `src/doctor.zig`
4. `src/tools/runtime_info.zig`
5. `src/tools/schedule.zig`
6. `src/main.zig`
7. `src/cron.zig`

Implementation scope:
1. Unify scheduler capacity source and messaging across CLI/tool/runtime.
2. Guarantee user-scoped truth parity across `status`, `doctor`, `runtime_info`, diagnostics.
3. Remove contradictory Telegram readiness reporting between config and runtime state.

Do not touch:
1. Origin wiring logic.
2. UI rendering behavior.
3. New integration providers.

Required tests:
1. Parity tests for `status --user-id`, `doctor --user-id`, `runtime_info`.
2. Capacity-limit text/value consistency tests in schedule+cron paths.
3. Regression tests for tenant-postgres fallback labeling.

Completion gates:
1. No contradiction for same `user_id` across truth surfaces.
2. Full build/test gates pass on rebased lane.

2. Channel UX presentation logic.

1. Origin assignment tests by path.
2. Policy deny/allow tests per origin.
3. Dedupe/rate/budget enforcement tests.

Completion gates:
1. `wake/proactive` are observable and policy-bounded.
2. No policy regression for `user/heartbeat/scheduler`.

## Definition of Done (Per Agent, Strict)

1. Scope deliverables implemented.
2. Required tests added and passing.
3. Full gates pass on rebased lane.
4. No regression found in touched behavior.
5. Handoff bundle delivered:
- change summary,
- evidence commands + outputs,
- risks and known limits,
- "what was intentionally not changed."
