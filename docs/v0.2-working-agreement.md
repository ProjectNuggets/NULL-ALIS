# v0.2 Working Agreement (Swiss-Watch Protocol)

Status: active  
Applies to: all v0.2 execution slices  
Last updated: 2026-03-12

## 1) Branch and Commit Discipline

1. One queue item = one dedicated branch slice.
2. Base every slice from latest `v0.2-scale-exec-swisswatch` (or designated integration branch).
3. Commit cadence:
- checkpoint commit when scope is coherent
- final commit only after gates pass
4. No blind commits:
- each commit must include scoped change intent in message
- each commit must include validation notes in PR/summary

## 2) Scope Discipline

1. Never mix:
- runtime behavior changes
- infra deployment changes
- broad docs rewrites
in one slice unless explicitly required and documented.
2. If a change expands scope, split it into a new queue item.
3. “While here” edits are prohibited unless bug-critical.

## 3) Mandatory Validation Before Every Checkpoint

Run both:
```bash
zig build test --summary all
zig build -Dengines=base,sqlite,postgres
```

If either fails:
1. stop
2. fix or rollback the slice
3. do not continue to next queue item

## 4) Canary Artifact Rules

1. Every canary must produce:
- machine artifact (`.json`)
- short interpretation note (`.md`)
2. Artifact naming must include:
- date
- scenario
- rollout stage
3. No promotion decision without artifact links.

## 5) Rollback and Stop Conditions

Immediate rollback/hold if:
1. p95 breaches target for two consecutive canary windows.
2. error rate > 1% sustained.
3. queue saturation sustained (>80%) without recovery.
4. ownership/tenant isolation behavior is ambiguous.

Rollback scope rule:
1. rollback only failing slice first
2. preserve unrelated successful slices

## 6) Documentation Sync Rules

After each completed slice:
1. update `docs/v0.2-unfinished-ledger.md` status fields.
2. update `docs/v0.2-execution-queue.md` item status and next action.
3. update `docs/plan-v02.md` active phase pointer.

No “done” claim is valid unless docs are synchronized.

## 7) Ownership and Handoff Format

Every handoff includes:
1. what changed
2. what did not change
3. commands run
4. artifacts produced
5. risks and follow-ups

No handoff = no closure.

## 8) Safety Defaults

1. additive changes first; avoid breaking interfaces.
2. preserve tenant and session-key ownership boundaries.
3. preserve background OAuth-connect restrictions unless explicitly approved.
4. prefer deterministic failure with reason code over hidden fallback.

## 9) S8 Staged Strict Rollout Rule

1. Execute strict-channel promotions only via:
- `docs/v0.2-s8-manual-rollout-runbook.md`
2. No strict channel expansion without:
- canary JSON artifacts
- stage markdown decision report
- explicit `GO/HOLD/ROLLBACK` outcome
