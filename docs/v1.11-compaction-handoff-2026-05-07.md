## RESOLVED — Chat not responding (V4-Pro empty content) — fixed 2026-05-07

**Original symptom:** ZAKI doesn't respond to "good morning" — frontend shows placeholder prose followed by repeated "Worked" indicators with no actual text.

**Diagnosis (gateway log):**
```
agent.fromConfig max_tool_iterations=1000 model=deepseek-ai/DeepSeek-V4-Pro reasoning=high
turn.profile kind=direct llm_calls=1 retries=0 tool_iterations=0 tool_calls=0 first_token_ms=-1
```

DeepSeek V4-Pro at `reasoning_effort=high` emits non-empty `reasoning_content` but empty `content`. The model's actual reply was buried in the reasoning channel; the visible content channel was empty.

**Root cause traced to `agent/commands.zig:3829-3834`:** `composeFinalReply` returned the empty `base_text` (content) regardless of whether `reasoning_content` was available. The `reasoning_mode` user setting controlled whether reasoning was appended ALONGSIDE content, but there was no fallback for "content empty AND reasoning present."

**Permanent fix shipped (`d00fdcf`):**

When `content` is empty/whitespace AND `reasoning_content` is non-empty, surface `reasoning_content` as the visible reply. The user asked a question; an empty reply is never the right answer.

```zig
const final_base: []const u8 = if (has_visible_text)
    trimmed_base
else if (has_reasoning)
    std.mem.trim(u8, reasoning_content.?, " \t\r\n")
else
    base_text;  // true no-output case
```

`reasoning_mode == .on` still controls "show reasoning alongside visible content" — orthogonal axis. When fallback fires (no visible content), don't double-print.

**Six regression tests added** (commands.zig:4824+) to lock the contract across all 4 fallback combinations.

**Booth impact:** RESOLVED. Deep mode (V4-Pro at reasoning=high) now produces visible replies. No mode-flip workaround needed.

### Subagent "received" bug — DIAGNOSED, FIX IS FE-SIDE

The bug per project memory:
> Agent runs subagent, subagent completes, but reply is just "received" instead of the work result.

Backend is **correct**:
- `gateway.zig:395 appendSubagentCompletionToGatewaySession` saves completion event + publishes to `app_event_subscribers`.
- `/chat/events?session_key=...` SSE endpoint at `gateway.zig:8910` streams `subagent_completion` frames live.
- `tool_only_turn` event already includes `spawned_task_ids` so the FE knows what to wait for.

FE is the gap:
- ZERO references to `/chat/events`, `subagent_completion`, or `app_event_subscribers` anywhere in `zaki-prod/src` or BFF.
- The FE never subscribes to the live completion stream.
- Result: user sees `EMPTY_TURN_PLACEHOLDER` as the final reply, the actual subagent output lands silently in DB and only becomes visible on chat reload.

**FE-agent work order:**
1. BFF: proxy `/chat/events` SSE through to nullalis (or have the FE call directly via the chat URL prefix used elsewhere).
2. FE: open an `EventSource` on `/chat/events?session_key=<current_session_key>` whenever a chat session is active. Maintain at most one per session.
3. FE: handle `event: subagent_completion` frames (payload: `{event_id, session_id, content}`). When a frame arrives:
   - If a transcript entry exists for the matching `tool_only_turn` (was pushed at `ChatArea.tsx:3983`), REPLACE its placeholder text with the completion `content`.
   - Else, append a new assistant message bubble with the content.
4. FE: handle reconnect — on EventSource error, exponential-backoff reconnect; the gateway re-sends pending events on `loadCompletionEvents` (gateway.zig:8930) so the FE doesn't miss anything across reconnects.

This unblocks the actual demo-blocker. Fast follow.

### Approval drop bug — DIAGNOSED, FIX IS FE-SIDE

Same root cause family as the subagent bug. The FE drops what the backend sends.

Backend (correct):
- POST `/api/agent/sessions/:key/approve` (gateway.zig:10961) → routes to slash command `/approve allow-once` → `handleGenericToolApprove` (commands.zig:2487) → `executeApprovedPendingTool` (root.zig:1713) → runs the tool → `self.turn(synthetic)` produces continuation reply → HTTP responds with `{"status":"approved","message":"<continuation reply>"}`.

FE (the gap, ChatArea.tsx:5356):
```
await approveAgentSession(sessionKey, payload);
// discards response.data entirely — the continuation reply is thrown away
decrementSessionApprovalCount(...);
setNullalisApprovalRequest(null);
```

User experience: clicks approve → approval card disappears → tool result never renders → "tool dropped."

**FE-agent work order (interim, ~5 lines):**
1. In `ChatArea.tsx:handleApprovalAction`, capture the response data:
   ```
   const { data } = await approveAgentSession(sessionKey, { ... });
   if (approved && data?.message) {
     pushAssistantMessage(sessionKey, data.message);
   }
   ```
2. The `data` shape is `{ status: "approved" | "denied", message: string }` —
   `lib/api.ts` says it's `{ ok: boolean }` but that's stale; widen the type
   to include `status` + `message`.
3. Tool-result events emitted DURING the continuation turn are observed by
   nullalis's observer, so once the `/chat/events` SSE is wired (subagent
   fix), those events also flow naturally and the FE doesn't even need
   step 1 — the events stream paints the result. Step 1 is the
   minimum-viable patch until the SSE work lands.

Both bugs (subagent + approval) are the same architectural FE gap: the
FE doesn't subscribe to or render what nullalis is already publishing.
Fix `/chat/events` SSE wiring once and both demos stop dying.
