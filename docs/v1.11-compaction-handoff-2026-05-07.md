## CRITICAL OPEN ISSUE — Chat not responding (V4-Pro empty content)

**Discovered 2026-05-07 mid-day** during Nova's chat testing.

**Symptom:** ZAKI doesn't respond to "good morning" — frontend shows the old EMPTY_TURN_PLACEHOLDER prose followed by repeated "Worked" indicators with no actual text.

**Diagnosis (from gateway log):**
```
agent.fromConfig max_tool_iterations=1000 model=deepseek-ai/DeepSeek-V4-Pro reasoning=high
turn.profile kind=direct llm_calls=1 retries=0 tool_iterations=0 tool_calls=0 first_token_ms=-1 ...
```

DeepSeek V4-Pro with `reasoning_effort=high` is producing empty `content` while emitting reasoning_content during 10s of thinking. Provider reads zero tokens, agent returns empty text, frontend renders thinking indicator only.

**Two paths to fix:**

1. **Quick (config-only, 5 sec)** — switch user `assistant_mode` from `deep` to `balanced` in `~/.nullclaw/data/users/1/config.json`. Same V4-Pro model but `reasoning_effort=medium` should produce visible content. Validates the diagnosis.

2. **Real fix (V1.12)** — investigate `providers/compatible.zig:434` (where reasoning_content streams per V1.10 docs) to confirm both `reasoning_content` and `content` channels are drained correctly from V4-Pro's SSE stream. Consider falling back to `reasoning_content` as visible text when `content` is empty (with a clear "thinking" tag).

**Booth-impact:** HIGH. ZAKI must respond to user messages with visible text. Without this, the demo is broken regardless of brain page polish.

**Recommendation:** Try the quick fix first to confirm. If balanced works, ship a V1.11 hotfix that EITHER (a) makes balanced the default deep-mode behavior or (b) fixes the V4-Pro reasoning_content extraction.
