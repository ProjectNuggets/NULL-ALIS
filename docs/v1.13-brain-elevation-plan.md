# V1.13 — Brain elevation plan

**Branch:** `v1.13/brain-elevation`
**Goal:** Close the gaps from V1.12 deep-dive + add the missing cognitive layers (Working Memory, Procedural, Dream) without rewriting the foundation.
**Anchor commit:** `b32445a` (V1.12 forward-flow shipped + audited).

---

## Context

V1.12 closed the connectivity layer: per-3-turn entity extraction, multilingual coreference, idempotent COOCCURS edges. But the deep-dive surfaced four real gaps:

1. **HI-01 latency** — entity_pipeline runs synchronously on the turn loop. Interim 10s timeout caps blast radius, but slow Together responses mean we miss ~10-30% of triggers.
2. **No working memory** — open loops, recent decisions, emotional context all rebuild from scratch every turn. ZAKI feels like a "very good amnesiac."
3. **No procedural memory** — skills are static `.md` files, no execution traces, no learning from practice.
4. **No dream state** — idle time wasted; no pattern extraction, no narrative synthesis, no proactive prep.

Plus: truncation caps (4KB extractor input, 5-10 top_k recall) are bottlenecks given Kimi K2.6's 256K window.

**Insight from Nova:** the heartbeat thread (`daemon.zig::heartbeatThread`) is an existing background lane. Nothing currently maps to it for memory work. Day 2 leverages this directly — no new threading infrastructure needed.

**Insight from Nova:** Day 5 (dream state) maps cleanly to a `cron.json` entry firing at 3 AM. Existing scheduler runs it; it calls a CLI command; results land via existing extraction_persist + memory pipeline.

---

## Architectural rule

Every layer in V1.13 reuses what exists. No new tables unless schema demands it. No new threading models. The pipeline gets richer, not more complex.

---

## Day 1 — Working Memory (Layer 0)

The single biggest user-PoV delta. Today ZAKI rebuilds context from scratch every turn via prose recall. Working memory adds 10-15 hot slots that persist across turns within a session and render at the top of every prompt's volatile block.

### What lives in working memory

```
slot_id  type           content                                  importance  last_touched
───────  ─────────────  ───────────────────────────────────────  ──────────  ────────────
1        open_loop      "call Alfred about MNDA"                 0.9         t-2 turns
2        active_goal    "ship V1.12 brain pipeline"              0.95        t-0 turns
3        emotional      "user is stressed about launch pressure" 0.7         t-5 turns
4        identity       "user calls me ZAKI; I am ZAKI BOT"      1.0         t-0 (always)
5        relationship   "Alfred = boss; Mia = daughter"          0.85        t-1 turns
6        decision       "using nullALIS as project codename"     0.9         t-0 turns
7        recent_entity  "Mika (husky)" — re-mentioned 3x         0.6         t-0 turns
8        skill_state    "content-strategy: assumed SaaS"          0.5         t-7 turns
9        temporal       "user travels to parents next week"       0.7         t-3 turns
10       open_question  "user asked about S-tier brain"           0.8         t-0 turns
```

### Schema (minimal — one table)

```sql
CREATE TABLE IF NOT EXISTS zaki_bot.working_memory (
    user_id BIGINT NOT NULL REFERENCES users(user_id) ON DELETE CASCADE,
    session_id TEXT NOT NULL,
    slot_id INT NOT NULL,                  -- stable position; eviction reuses slot_id
    slot_type TEXT NOT NULL,               -- open_loop|active_goal|emotional|identity|relationship|decision|recent_entity|skill_state|temporal|open_question
    content TEXT NOT NULL,                 -- the actual text
    source_key TEXT,                       -- optional ref to memory key that birthed this slot
    importance FLOAT NOT NULL DEFAULT 0.5, -- 0..1, used for eviction
    pinned BOOLEAN NOT NULL DEFAULT FALSE, -- never evict
    created_at TIMESTAMPTZ DEFAULT NOW(),
    last_touched_at TIMESTAMPTZ DEFAULT NOW(),
    PRIMARY KEY (user_id, session_id, slot_id)
);
CREATE INDEX idx_wm_session ON zaki_bot.working_memory(user_id, session_id, last_touched_at DESC);
```

### Slot management (LRU + importance composite)

- 15 slots max per (user_id, session_id)
- Eviction: lowest `(importance × recency_decay)` evicted on overflow
- Promotion: a memory is promoted to working memory when:
  - User explicitly asks ("remind me to X" → open_loop slot)
  - Recalled 3+ times in current session
  - Active goal or open question detected by extractor
  - Identity facts always pinned (slot 1-2 reserved)

### Render path

In `prompt.buildVolatileSystemPrompt` (Context v2 volatile block):
```
<working_memory>
  Active goals: ship V1.12, content strategy for nullALIS
  Open loops: call Alfred about MNDA, dentist Mar 25
  Recent context: user is stressed about launch pressure
  Recent entities mentioned: Mika (husky), Alfred (boss), Mia (daughter)
  Identity: user is Nova; I am ZAKI BOT, built by NovaNuggets
</working_memory>
```

### Wiring

- `src/agent/working_memory.zig` (new module): CRUD + eviction + render
- `src/zaki_state.zig`: working_memory schema migration
- `src/agent/prompt.zig::buildVolatileSystemPrompt`: render WM block at top
- `src/agent/extraction_persist.zig`: when extraction emits an open_loop / active_goal predicate, promote to WM slot
- New tool `working_memory` (read-only for the agent): list current slots, manually pin/unpin

### Day 1 gate

- After 5 turns of conversation, working memory has populated slots
- Slots are visible in next-turn prompt (verify via byte-stable hash log)
- Identity facts are pinned and never evicted
- `zig build test` clean

### Day 1 commits (atomic)

1. `iter01a` migration + working_memory.zig CRUD

## Day 2 — Move entity_pipeline + session_end onto heartbeat lane (HI-01 real fix)

The existing background lane is `daemon.zig::heartbeatThread`. Today nothing maps to it for memory work. We move:

1. **entity_pipeline per-3-turn** trigger off the agent's turn-loop thread → onto a queued worker dispatched from heartbeat tick
2. **session_end entity_pipeline pass** off the session evictor's mutex-holding thread → onto same queue

### Architecture

```
agent turn N completes
  └── post-turn hook checks: turns_since_extraction >= 3?
        └── YES → enqueue ExtractionJob{user_id, session_id, snapshot_recent_text} to bus
              (returns immediately — turn loop continues)

heartbeatThread (every N seconds):
  ├── checks bus for pending ExtractionJobs
  ├── runs entity_pipeline.runOnTurn() per job
  └── writes results via existing extraction_persist + memory_edges paths
```

### Why this works

- `bus.zig` already has proactive event tagging with `source: "heartbeat"`
- `daemon.zig::runHeartbeatAgentTurn` already runs agent turns with full provider/state_mgr context
- `tools/root.zig::ChatStreamTenantLane.cron` is a recognized lane
- `background_safe` tool metadata already exists; entity_pipeline path is read+write but its writes are idempotent so safe in this lane

### Schema addition (minimal)

```sql
CREATE TABLE IF NOT EXISTS zaki_bot.extraction_queue (
    id BIGSERIAL PRIMARY KEY,
    user_id BIGINT NOT NULL,
    session_id TEXT NOT NULL,
    job_type TEXT NOT NULL,            -- 'wiki_link' | 'session_end' | 'backfill'
    payload JSONB NOT NULL,            -- snapshot_text or session_id
    enqueued_at TIMESTAMPTZ DEFAULT NOW(),
    started_at TIMESTAMPTZ,
    completed_at TIMESTAMPTZ,
    status TEXT NOT NULL DEFAULT 'pending', -- pending|running|done|failed
    error TEXT
);
CREATE INDEX idx_xq_pending ON zaki_bot.extraction_queue(user_id, enqueued_at)
  WHERE status='pending';
```

### Wiring

- `src/agent/extraction_queue.zig` (new): enqueue + claim + complete
- `src/agent/root.zig`: per-3-turn trigger now enqueues instead of running inline
- `src/agent/commands.zig`: session_end ditto
- `src/daemon.zig::heartbeatThread`: poll queue, run pending jobs (concurrency-safe via SELECT FOR UPDATE SKIP LOCKED)
- Existing `entity_pipeline.runOnTurn` unchanged — just called from a different thread

### Truncation tune (Day 2 includes)

- Bump `MAX_INPUT_BYTES` from 4KB → 16KB (Kimi has 256K, no reason to truncate so aggressively)
- Bump session-end transcript build cap 3KB → 12KB
- Add session-end **prefix dedup** (ME-05 fix): hash the prefix already-seen on per-3-turn runs, skip in session_end if hash matches recent

### Day 2 gate

- Per-3-turn trigger returns in <50ms (just enqueues)
- Heartbeat polls and processes within 60s
- Failure-soft: queue retries on transient error, marks failed after 3 attempts
- Live test: 3 multilingual turns → entity_pipeline runs in background, edges land within 60s, agent turn loop never blocks
- `zig build test` clean

### Day 2 commits

1. `iter02a` migration + extraction_queue.zig
2. `iter02b` per-3-turn trigger refactored to enqueue-only
3. `iter02c` session_end ditto + prefix dedup
4. `iter02d` heartbeatThread queue worker
5. `iter02e` truncation cap bumps + tests

## Day 3 — Graph-aware recall

Today `memory_recall` returns prose blobs. The agent never navigates the typed-edge graph it built. Day 3 makes the graph first-class for the agent.

### Three sub-deliverables

#### 3.1 Hybrid recall = vector + keyword + centrality

`memory_loader.loadContextWithRuntimeDetailed` already does vector+keyword. Add a centrality boost:

```
score = (1 - vector_distance) * 0.5
      + keyword_score * 0.3
      + centrality_score * 0.2

centrality_score = 1 - 1 / (1 + edge_count_for_this_memory_or_entity / 8)
```

Memories with more edges (hubs in the graph) get boosted in recall. Orphans get penalized. Implementation: SQL query joins `memory_edges` and computes degree per-row; the LATERAL JOIN pattern is fine for the recall path.

#### 3.2 brain_graph tool — agent prompt nudge

The tool exists but agent rarely calls it. Add to system prompt:
> When asked about a person, project, or concept that has rich connections in your memory, prefer `brain_graph` to navigate the local subgraph (depth 2) before composing a response. The graph view shows you who/what is connected to whom, with what predicates and weights.

Plus: cache `brain_graph` results within a turn (already cacheable per metadata).

#### 3.3 Graph-context injection in volatile prompt

After working memory block, add a small graph-context block:
```
<related_to_current_turn>
  Mentioned entities in this turn: Alfred (12 edges, central), nullALIS (8 edges, central)
  Linked to Alfred: KPMG, Google, MNDA, NovaNuggets
  Linked to nullALIS: ZAKI, Neptune, Helix, Together
</related_to_current_turn>
```

Driven from working_memory (Day 1) + last-turn entity mentions.

### Day 3 gate

- Hybrid recall returns hub-memories first when query touches a hub-entity
- Agent invokes brain_graph at least 1×/session organically (no forced prompt)
- Graph context block visible in next-turn prompt
- `zig build test` clean

### Day 3 commits

1. `iter03a` centrality column on memory_edges materialized view OR LATERAL JOIN in recall
2. `iter03b` hybrid recall scoring + tests
3. `iter03c` brain_graph prompt nudge in system prompt
4. `iter03d` graph context block in volatile prompt

## Day 4 — Procedural memory (Layer 6)

Skills today are static `.md` files. Every invocation reads from scratch. No improvement. Day 4 adds execution traces.

### Schema

```sql
CREATE TABLE IF NOT EXISTS zaki_bot.skill_executions (
    id BIGSERIAL PRIMARY KEY,
    user_id BIGINT NOT NULL,
    skill_name TEXT NOT NULL,        -- e.g. 'content-strategy'
    session_id TEXT,
    task_summary TEXT,                -- what user asked
    steps_executed JSONB,             -- [{step, tools_used, output_excerpt, success}]
    assumptions_made JSONB,           -- ["assumed SaaS model", ...]
    user_feedback TEXT,               -- "positive — asked for more"
    outcome_quality FLOAT,            -- 0..1, agent self-assessed or inferred
    created_at TIMESTAMPTZ DEFAULT NOW()
);
CREATE INDEX idx_skill_exec ON zaki_bot.skill_executions(user_id, skill_name, created_at DESC);
```

### Capture path

In `agent/root.zig` after a turn that used a skill (detected via tool_call to skill_registry or invocation pattern):
- Capture: skill_name, task_summary (from user message), steps_executed (from tool_call sequence), assumptions (from agent's reasoning_content), outcome_quality (initial: heuristic; later: user feedback signal)
- Persist to `skill_executions`

### Recall path

When agent invokes a skill:
- Query `skill_executions` for top-3 most-recent runs of that skill for this user
- Inject into prompt:
  ```
  <skill_history skill="content-strategy">
    Last execution (2026-04-15): assumed SaaS model — user didn't confirm.
    Pillar 4 (Open Agent Movement) resonated.
    Refinement: ask "Who is the customer?" before pillars.
  </skill_history>
  ```

### Day 4 gate

- After 3 invocations of the same skill, recall surfaces prior runs
- Agent demonstrably builds on prior assumptions instead of starting cold
- `zig build test` clean

1. `iter04a` migration + skill_executions table

## Day 5 — Dream state (3 AM cron)

The simplest, lowest-risk delivery of Layer 7 — runs as a daily cron job at 3 AM via the existing scheduler.

### Cron entry (`~/.nullalis/cron.json`)

```json
{
  "id": "dream_cycle_3am_user_1",
  "user_id": 1,
  "schedule": "0 3 * * *",
  "kind": "command",
  "command": "nullalis admin dream-cycle --user-id 1",
  "enabled": true
}
```

(Or pure cron-spec equivalent depending on the existing schedule format — implementation decides.)

### What `dream-cycle` does (sequential, ~2-5 min total)

```
1. Pattern extraction (LLM call):
   Read last 7 days of conversation_messages.
   Find recurring themes, emotional patterns, mentioned-but-untracked entities.
   Output: pattern_memories with link_type=pattern.

2. Narrative synthesis (LLM call):
   For each high-density entity cluster (community), build a short narrative.
   "The nullALIS naming journey: from Panther (Mar 15) → reconsidered (Mar 27)
   → corrected to nullALIS (Apr 12)."
   Persist as compose_memory with link_type=synthesis.

3. Brain hygiene:
   For typed orphans, run cosine search; if top-3 above 0.85, auto-link with attribution=hygiene_job.

4. Importance recompute:
   Update importance_score using recency × edge_count × novelty × retrieval_count.

5. Working memory pre-populate (Day 1 dependency):
   Stage 5-10 high-importance slots ready for next session start.

6. Idle reflection note:
   Compose a single memory key=dream_log/<date> capturing what was found.
   Renders on /brain as today's reflection.
```

### Wiring

- `nullalis admin dream-cycle` CLI command (new in `cli.zig` or admin subcommand)
- `src/agent/dream.zig` (new module): orchestrates the 6 steps
- All steps reuse existing pipelines:
  - Pattern extraction → `extraction_persist.persistExtracted` with new link_type=pattern
  - Narrative → `compose_memory` tool
  - Hygiene → existing `listOrphanMemories` + `upsertMemoryEdge`
  - Importance → `memory/importance.zig` recompute
  - Working memory → Day 1 `working_memory.zig` API
- Dedup: skip if last dream_log/<date> exists for today

### Day 5 gate

- Cron entry installs cleanly via `nullalis cron add ...`
- Manual `nullalis admin dream-cycle --user-id 1` runs end-to-end in <5min
- `dream_log/<date>` memory landed; visible on /brain
- 5-10 patterns extracted, 1-3 narratives composed, all orphans either linked or marked
- Working memory has next-session-ready slots
- `zig build test` clean

### Day 5 commits

1. `iter05a` `nullalis admin dream-cycle` CLI command + dispatch
2. `iter05b` `agent/dream.zig` orchestrator
3. `iter05c` pattern extraction + narrative synthesis prompts

## Truncation + cap tunes (folded across days)

| Cap | Today | New | Day | Rationale |
|---|---|---|---|---|
| `MAX_INPUT_BYTES` | 4KB | 16KB | 2 | Kimi 256K window; we're throwing away signal |
| Session transcript build cap | 3KB | 12KB | 2 | Same reason |
| `top_k` recall (default) | 5-10 | 15 | 3 | Graph-aware needs more candidates |
| `MAX_FACTS_PER_SESSION` | 100 | 250 | 4 | Long sessions hit cap; bumped after WM dedup |
| `MAX_MENTIONS_PER_TURN` | 24 | 40 | 2 | Long multi-entity turns get clipped |

All bumps are additive and can be reverted via config without code changes.

---

## Risk + mitigation table

| # | Risk | Mitigation |
|---|---|---|
| 1 | Working memory bloat — 15 slots feel too many in context | Render only top-10 by composite priority; cap volatile-block size separately |
| 2 | Heartbeat queue backlog if user chats faster than worker drains | Worker batches up to 10 jobs per tick; alert at backlog >50 |
| 3 | Centrality boost dominates → agent always returns same hubs | Cap centrality contribution at 0.2 of total score; recency stays >0.4 |
| 4 | Skill execution capture is wrong shape, polluting recall | Day 4 starts with capture-only mode (no recall); enable recall after 1 week of capture |
| 5 | Dream cron eats LLM budget if pattern extraction unbounded | Hard cap 8 LLM calls per dream cycle; fail-soft if cost >$0.05 |
| 6 | Hygiene job creates spam edges | Existing 0.85 threshold + attribution tag (already in V1.12) |
| 7 | Pattern extraction over-fits 7 days, misses long-arc patterns | Day 5 looks at 7-day window primarily; weekly meta-cycle reads 30-day window |

## What this plan deliberately does NOT do

- **No new tables** beyond: `working_memory`, `extraction_queue`, `skill_executions` (3 total — minimum viable)
- **No new threading model** — Day 2 reuses existing `heartbeatThread`
- **No new cron infrastructure** — Day 5 uses existing scheduler
- **No new LLM provider** — all uses Kimi K2.6
- **No FE work in this branch** — backend-only; FE rendering folds into next session's brain page work
- **No backfill of pre-V1.6 memories** — separate one-time admin job, not in branch

Stays scoped. Stays additive. Stays merge-safe.
