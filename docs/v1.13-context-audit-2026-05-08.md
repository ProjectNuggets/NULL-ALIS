# V1.13 — Context audit: what gets sent every turn

**Branch:** `v1.13/brain-elevation`
**Audit goal:** Map every section emitted in stable + volatile prompts. Find duplications. Verify caching boundaries. Project context-window-fill over a long session.

---

## STABLE prompt (cached after turn 1; rebuilt only when workspace files / tools / persona change)

| # | Section | Source | Typical bytes | Cache behavior |
|---|---|---|---|---|
| 1 | Identity | workspace MD files (`AGENTS.md`, `USER.md`, `MEMORY.md`, `SOUL.md`) | 5–30 KB | Stable across turns; invalidates on file change |
| 2 | Tools | tool catalog | 8–15 KB | Stable; invalidates when tools list changes |
| 3 | Link-type vocab | `memory_root.ALL_LINK_TYPES` (comptime) | ~1 KB | Stable forever |
| 4 | **Brain Architecture** (V1.13) | comptime string in `prompt.zig` | ~3 KB | Stable forever |
| 5 | Response Protocol | comptime string | ~3 KB | Stable forever |
| 6 | Channel Attachments | comptime string | ~500 B | Stable forever |
| 7 | Capabilities (optional) | derived from config + tools | 2–5 KB | Stable per session config |
| 8 | Persona | `SOUL.md` front-matter | ~500 B | Stable per session |
| 9 | Turn Classification | comptime string | ~1 KB | Stable forever |
| 10 | Task Decomposition | comptime string | ~1 KB | Stable forever |
| 11 | Safety | comptime string | ~1 KB | Stable forever |
| 12 | Skills (optional) | workspace `skills/` dir | 0–10 KB | Stable per session |
| 13 | Workspace path | string | ~200 B | Stable per session |
| 14 | Runtime info | model name | ~200 B | Stable per session |

**Stable total: ~25–70 KB per session.** All of this is cached on Together vLLM (byte-prefix match) and on Anthropic (when explicit `cache_control` is set). After turn 1, the LLM provider re-uses the cached KV state for the stable prefix — we pay only for the volatile section.

**Verified stable:** none of the section builders read mutable per-turn state. The `appendDateTimeSection` and `buildConversationContextSection` are in the VOLATILE block, not stable. ✓

## VOLATILE prompt (rebuilt every turn; NOT cached)

| # | Section | Source | Typical bytes | Worst case |
|---|---|---|---|---|
| 1 | ConversationContext | channel + sender + idle_gap + last_interaction | 200–500 B | 1 KB |
| 2 | DateTime | UTC timestamp + day-of-week | ~200 B | 300 B |
| 3 | **Working Memory** (V1.13) | `agent/working_memory.zig::renderBlock` (top 10 of 15 slots) | 0.5–2.5 KB | 3 KB (10 slots × 250B max content per slot + framing) |
| 4 | `<memory_for_turn>` | `agent/memory_loader.zig::loadTurnMemorySlot` | 1–32 KB | **64 KB** (cap = `DEFAULT_MEMORY_CONTEXT_MAX_BYTES`) |

**Volatile total: 2–67 KB per turn.** This is the per-turn LLM input cost.

**The dominant cost is `<memory_for_turn>`.** Working Memory is small (≤3 KB) compared to the recall payload.

## DUPLICATIONS observed

### DUP-1: Identity rendered TWICE every turn

**Where it shows up:**
- `<working_memory>` slot 0 — pinned `slot_type=identity` (V1.13 Day 1)
- `<memory_for_turn>` `<active_identity>` block (legacy memory_loader, V1.8-9)

**Cost:** ~500 B–2 KB duplicated per turn. Not catastrophic, but real.

**Fix path (post-V1.13):**
- Working Memory takes ownership of identity rendering when populated.
- Drop `<active_identity>` injection from `memory_loader` when WM is non-empty.
- Fallback: when WM is empty (postgres degraded, fresh session), `<active_identity>` legacy path still fires.

This is a clean refactor; doesn't ship in V1.13 to avoid risk during merge prep.

### DUP-2: Workspace files rendered in Identity AND inline-quoted in agent's reasoning

`AGENTS.md`, `USER.md`, `MEMORY.md` content sits in the Identity section verbatim. The agent then sometimes calls `file_read AGENTS.md` and re-injects the bytes into history. Not a per-turn cost (only when the agent decides to read), but worth knowing.

**No fix needed; this is agent-decision.**

## Context window fill over a long session

Assume Kimi K2.6 (256K context window):

| Turn | Stable | Volatile | History (cumulative user+assistant msgs) | Total | % of 256K |
|---|---|---|---|---|---|
| 1 | 50 KB | 5 KB | 0 KB (just-arrived user msg) | 55 KB | 21% |
| 5 | 50 KB | 25 KB (memory recall warming) | 8 KB | 83 KB | 32% |
| 10 | 50 KB | 35 KB | 18 KB | 103 KB | 40% |
| 30 | 50 KB | 40 KB | 60 KB | 150 KB | 59% |
| 50 | 50 KB | 45 KB | 110 KB | 205 KB | 80% |
| 60 | 50 KB | 50 KB | 140 KB | 240 KB | 94% — **Pass A trigger at 70% (179 KB) already fired** |

**Pass A** (cheap dedup + placeholder substitution) fires at 70% pressure → ~179 KB. **Pass C** (LLM summarization) fires at 90% → ~230 KB. Both are configured in `compaction.zig`.

**With Pass A** firing around turn 35–45, history gets dedup'd and old tool outputs collapsed to placeholders. Effective fill plateaus around 60–70%.

**With Pass C** firing rarely (only on extreme pressure), it's the safety net. In normal Kimi K2.6 usage, Pass A handles most of it.

## What changes turn-to-turn (per-turn LLM input cost)

```
Per-turn input bytes uploaded:
  STABLE (cached, ~free after turn 1)              0 KB effective
  VOLATILE                                         2–67 KB (avg ~25 KB)
  CURRENT user message                             0.1–4 KB
  TOTAL per-turn input                             ~25–70 KB ≈ 6K–18K tokens
```

At Kimi K2.6's $1.20 / M input tokens: **~$0.007–$0.022 per turn** for prompt overhead. Manageable. If user sends rapid back-and-forth (5–10 turns/min), that's ~$1–2/hour for a heavy chat.

## Recommendations (by priority)

### NOW (during V1.13 merge)
- ✅ Done: this audit document committed.
- ✅ Done: Brain Architecture section landed in stable prompt — agent now KNOWS what's available.

### NEXT (post-V1.13 merge — small follow-ups)

1. **Resolve DUP-1.** Drop `<active_identity>` legacy injection when working_memory is populated. Saves ~500 B–2 KB per turn. Adds a fallback path so non-postgres deploys don't lose identity. ~30 min work.

2. **Profile Identity section size.** If `MEMORY.md` or `USER.md` grow to >10 KB each, the stable section bloats. Add a soft warning when stable exceeds 80 KB so we catch growth early.

3. **Add stable_prefix_byte_hash log.** Already documented in V1.11 plan but verify it actually emits. Lets us see in prod whether the cache is hitting.

4. **Consider tightening `<memory_for_turn>` cap.** Currently 64 KB. Real usage rarely needs more than 16 KB; the 64 KB ceiling is defensive. Drop to 32 KB and observe — saves 30 KB per turn worst-case.

### LATER (post-V1.13)

5. **Per-section byte logging.** Emit a log line per turn with the byte count of each volatile section so we can see in prod which is dominating.

6. **Token-accurate counting.** Today we estimate at 4 chars/token. Production tokenization on Kimi K2.6 may differ ±20%. Add per-provider tokenizer bindings post-V1 ship.

## Confidence

| Aspect | Confidence | Why |
|---|---|---|
| Stable section is genuinely stable (cache works) | **High** | Audited every section builder; no per-turn state read. |
| Volatile is bounded | **High** | Hard caps at 64 KB memory + ≤3 KB WM + small fixed sections. |
| Dup-1 observed but not catastrophic | **High** | 500 B–2 KB extra per turn. Real, fixable, not blocking. |
| Pass A compaction triggers correctly | **Medium** | Logic is correct; have not stress-tested at 70% on Kimi K2.6 in prod (sessions don't usually reach it). |
| Pass C compaction triggers correctly | **Medium** | Same — rare in production. |
| Per-turn cost ~$0.007–$0.022 | **High** | Math checks. |

Overall: **the context architecture is sound.** The duplications are minor and fixable. Caching works because stable is genuinely stable. Volatile is bounded. Compaction handles long sessions.

## Bottom line

After V1.13 merges:

- Per-turn LLM input cost is dominated by `<memory_for_turn>` (recall payload) + small overhead.
- Working Memory adds <3 KB per turn — negligible.
- Brain Architecture briefing adds ~3 KB to stable, ZERO per-turn cost (cached).
- One real duplication (DUP-1: identity rendered twice) deserves a 30-min cleanup commit post-merge.
- No structural problems. No runaway growth paths.
