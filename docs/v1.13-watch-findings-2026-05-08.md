**The cron sentinel is not strictly necessary.** The briefing alone is enough. Reduces V1.14 work.

## NEGATIVE — what's broken or weak

### N-01 (HIGH): Slot 0 identity content is wrong

`pinIdentityFromUserState` pulls via `listIdentityFacts`. The bundle includes:

> - Ahmed is working on the nullALIS project with Google. ← extraction error; Ahmed isn't the user
> - User works on nullALIS with the Neptune team in Riyadh. ← OK
> - **Assistant is an open-source AI framework called Agent Zero** ← WRONG! Stale/incorrect data
> - User prefers to call the assistant ZAKI ← OK

**Two of four facts are wrong.** ZAKI reads this as identity context. May believe it.

**Root cause:** `listIdentityFacts` filter is too lenient — pulls activity facts and old training data, not just identity. The pinned content quality depends on `listIdentityFacts` being clean.

**Fix:** Either (a) tighten `listIdentityFacts` to filter on canonical key prefixes (`user_identity*`, `boss_identity`, `user_persona*`), OR (b) compose a curated identity bundle on first session create instead of pulling raw rows.

### N-02 (HIGH): Together extraction LLM unreliable

Every entity_pipeline job in the watch session timed out:
```
extraction_queue.processed job_id=3 outcome=llm_failed elapsed_ms=40065
extraction_queue.processed job_id=3 outcome=llm_failed elapsed_ms=40080
extraction_queue.processed job_id=3 outcome=llm_failed elapsed_ms=...
```

10s timeout × 4 retry attempts = 40s before terminal failure. queue.failed=3 / queue.done=1 across the session. **Net new entities + edges from this session: ZERO.**

Together's reliability today is low. Our retry logic correctly degrades, but we get no new graph data when Together is slow.

**Mitigation paths:**
- Lower retry count (currently provider_reliable retries 2x + initial = 3 attempts × 10s = 30s base)
- Add a per-day failure rate alert
- Consider a sidecar provider (Groq) for entity extraction when Together is degraded
- Accept that some days the brain doesn't grow forward — session_end summarizer + manual memory_store still capture value

### N-03 (MEDIUM): No conversation_messages table

My V1.13 architecture audit doc claimed conversations live in `conversation_messages`. Actual table is `messages`. Documentation drift. Other docs may have similar errors.

**Fix:** audit pass on docs against actual schema. ~15 min.

### N-04 (MEDIUM): WM slot 0 dual-render risk persists

Two working_memory rows at slot 0 visible (different sessions). Each pins identity. The DUP-1 fix gates skip_legacy_identity on 4 conditions but doesn't gate on whether slot 0 ACTUALLY has identity-quality content. With N-01 (slot 0 has bad content), agent gets bad identity from one source instead of both sources. Net UX: same wrong information, half the bytes.

**Fix:** N-01 first; then re-evaluate N-04.

### N-05 (LOW): No cron entry installed for dream

The cron entry shape is documented in `dream.zig`:
```json
{"command": "dream", "schedule": "0 3 * * *", "kind": "agent"}
```
But no installer / onboarding scaffolds it. Each user must manually edit cron.json.

Given P-01 (dream fires from chat too), this is less urgent. But for production users we want 3 AM auto-fire without manual setup.

**Fix:** add to onboard scaffold. ~15 min.

## Verdict

**The architecture works.** The agent uses what V1.13 built — working memory, async queue (when Together cooperates), procedural capture, and most impressively the dream cycle (without even needing the cron sentinel — the briefing taught it what to do).

**The content quality has one urgent flaw** — N-01 slot 0 identity bundle is wrong. Otherwise the system behaves coherently across English/Arabic, handles filler vs substance correctly, and reflects honestly when it doesn't know things.

**Confidence after watch:** the architecture promises are real. The day-1 polish needs N-01 fixed.

If we fix N-01 + ship N-03/N-05, we have a defensible "first non-experimental V1." Mike-bait. Booth-ready.
