## Why these two together

**Todo tool** and **second-brain visualization** share an architecture: both are first-class views over a structured-memory layer that persists across turns and surfaces in the UI. Building them together amortizes the underlying "structured memory queries" surface.

If we ship them separately, the todo tool needs its own persistence + frontend rendering, and the second-brain needs its own. Together, they share the **memory graph + timeline + compose** primitives.

---

## Part 1 — Todo tool

### What it does

When the user asks the agent for multiple tasks (3+, comma-separated, numbered, or a "do X then Y then Z" prompt), the agent creates a todo list, shows it back to the user, and works through it with status tracking. Carries across turns. User and agent both see the list.

### Tool surface (3 actions)

```zig
// src/tools/todo.zig — new file
pub const TodoTool = struct {
    workspace_dir: []const u8,
    memory: ?Memory = null,
    session_id: ?[]const u8 = null,

    // Action: "create"
    //   args: { title: string, items: [{title, depends_on?: [int]}] }
    //   returns: { list_id: string, item_ids: [int] }
    //
    // Action: "update"
    //   args: { list_id, item_id, status: "pending"|"in_progress"|"completed"|"blocked", note?: string }
    //   returns: { item_id, new_status, list_summary: { total, completed, in_progress, blocked } }
    //
    // Action: "list"
    //   args: { list_id?: string }    // null = current session's active list
    //   returns: { list_id, title, items: [{id, title, status, depends_on, note}] }
};
```

### Persistence

Backed by the **memory** layer (postgres or sqlite). Each list = one memory entry with a structured payload. Each item = a sub-row keyed on (list_id, item_id). Schema piggybacks on existing `memories` table — no migration:

```
memory_kind = "todo_list"
memory_key  = "todo:<session_id>:<list_id>"
content     = JSON {title, items: [{id, title, status, depends_on, note}], created_at, updated_at}
```

Reuses the existing memory hot-path → memory_recall surfaces todos to the agent on context retrieval. No new tables.

### System prompt addition

Add to `prompt.zig::buildResponseProtocolSection`:

> **Multi-task prompts:** when the user requests 3+ distinct tasks (numbered, comma-separated, or "first X then Y then Z"), call `todo_create` BEFORE acting. Show the plan back to the user. Update item status as you progress. Reason aloud about dependencies and ordering. The user can edit the plan; respect their changes on the next turn.

### Frontend surface

Reuses existing **MemoryViewer** modal in zaki-prod. Filter `memory_kind = "todo_list"` to render as a checklist with:
- Title at top
- Each item: status badge (pending/in_progress/completed/blocked) + checkbox + note
- "Mark item done" button → frontend posts `todo_update` action via the agent (chat message: "mark item 3 as completed")
- Or operator can `PATCH /api/v1/users/:id/memory/:key` directly (existing endpoint)

### Effort estimate

- **Backend tool:** 1 day (new `tools/todo.zig`, action dispatch, JSON schema, tests)
- **Memory schema piggyback:** half day (key namespace + retrieval rules)
- **Prompt directive:** 30 min
- **Frontend MemoryViewer filter + render:** half day (zaki-prod, separate PR)

Total: **2 days backend + half day frontend**, parallelizable.

## Part 2 — Second-brain visualization

### What it does

A graph + timeline view of the user's accumulated knowledge: every memory entry (semantic memories, todos, tool-results, file-references) as a node, with edges representing relationships (same session, semantic similarity, explicit references). User can navigate, search, time-travel, and compose new memories from existing ones.

This is the V-infinity "differentiator" memory entry calls out: not a chat log, but a living graph of what the agent has learned with you.

### Backend surface (3 endpoints)

```
GET  /api/v1/users/:id/brain/graph
  ?since=<unix_ts>        // optional: incremental; default = full graph
  ?max_nodes=<int>        // optional: cap, default 500
  ?node_kinds=<csv>       // optional: filter by kind (semantic, todo_list, tool_result, etc.)

  Response: {
    nodes: [{ id, kind, title, created_at, last_accessed_at, importance, preview }],
    edges: [{ from, to, kind: "session"|"semantic"|"reference", weight }]
  }

GET  /api/v1/users/:id/brain/timeline
  ?from=<unix_ts>&to=<unix_ts>
  ?session_filter=<key>   // optional: just one session

  Response: {
    events: [{ timestamp, kind, memory_key, summary, session_key }]
  }

POST /api/v1/users/:id/brain/compose
  Body: { source_keys: [string], directive: string }

  Response: { new_memory_key: string, summary: string }
  // Agent runs a synthesis pass over the source memories + the user's directive
  // ("combine these notes into a research outline") and writes a NEW memory entry
  // referencing the sources. Operates as a normal memory_write under the hood.
```

### Backend implementation notes

- **`/brain/graph`** — read-only over the existing `memories` table + an in-memory edge cache. Edges:
  - **Session edges:** memories created in the same session_key
  - **Semantic edges:** cosine similarity between memory embeddings > 0.7 (only computed for the top N most-recent memories; lazy + cached)
  - **Reference edges:** when memory A's content cites memory B's key (parse `memory:<key>` patterns in content)
- **`/brain/timeline`** — straight DB query on memories + completion_events tables, sorted by timestamp.
- **`/brain/compose`** — agent calls a new `compose_memory` tool that:
  1. Retrieves all source memories
  2. Sends them to the LLM with the user's directive
  3. Parses the LLM's synthesis as a new memory_kind = "synthesis" entry
  4. Records `references` field linking back to source_keys

### Frontend surface (zaki-prod)

This is the real "wow" V1.5 surface. Suggest a new `/brain` route in zaki-prod:

- **Graph view** (default) — D3 or react-flow rendered, force-directed. Click node = open detail. Drag-select multiple = "compose" button appears.
- **Timeline view** — vertical scroll, grouped by day. Each item = one memory entry preview.
- **Search bar** — semantic search via existing `memory_recall` endpoint.
- **Compose modal** — select sources + write directive, fires POST `/brain/compose`, shows new memory in graph immediately.

### Effort estimate

- **Backend:** 2-3 days (3 endpoints, edge computation logic, compose_memory tool, tests)
- **Frontend:** 4-5 days (graph + timeline + search + compose modal)

Total: **3 days backend + 1 week frontend**.

## Locked design choices (Nova greenlight 2026-04-30)

1. **Todo persistence scope:** per-session. Todos are "this conversation's work." Cross-session todos can promote in a later iter if usage warrants.
2. **Graph edge weights:** static thresholds (cosine similarity > 0.7 default) for V1.5. **Plus traversal-event logging from day one** — when a user clicks an edge in the graph view, queries from one node to another, or references both nodes when composing, log an event with both node IDs + a timestamp. We don't use those events in V1.5; they bootstrap the V1.6 learned-weights system with 30+ days of real traversal data so we don't have to cold-start the learning.
3. **Graph node cap:** 500 default, user-toggleable in settings. Watch first-week telemetry: if users hit cap and search a lot, lower default + improve search; if they hit cap and don't search, raise default.
4. **Compose-memory authorship:** visible. The compose-memory entry carries `synthesized_by: "agent"` field + `references: [source_keys]` for full audit trail. User can hide synthesis entries via filter if it gets noisy.

Implementation starts 2026-05-01.
