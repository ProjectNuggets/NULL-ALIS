## 2. The 3 prompt changes that closed the gap

### A1.1 — NO FACTS guard
```
1. If the conversation contains no factual content (pure greetings,
   pleasantries, ack-only exchanges with nothing established), output
   EXACTLY the two characters: NO FACTS
```
**Effect:** `casual/conv_02` precision 0.00 → 1.00. Pure pleasantries no longer produce meta-narrative bullets.

### A1.2 — Direct-support rule
```
2. Every bullet must be DIRECTLY supported by user or assistant text
   in the conversation. Do NOT add inferences, suggestions you offered
   that were not adopted, or general commentary.
```
**Effect:** Reduced fabricated alternatives — `long_multi_topic` no longer treats assistant's offered alternative pricing as a fact.

### A1.3 — Attribution prefixes
```
3. When a fact reflects an assistant offer or suggestion (not a user
   statement or decision), prefix the bullet with "(assistant offered) ".
   When it reflects an unresolved consideration, prefix with "(undecided) ".
```
**Effect:** Provides a signal V1.6's `attributed_to` extraction can pick up. "(assistant offered) ..." can be tagged `attributed_to: assistant_offer` at parse time.

### A1.4 — Anti-meta-statement rule (iter2)
```
4. Do NOT include conversational meta-statements like "user thanked the
   assistant", "user mentioned X", "user asked about Y", "user requested Z".
   State the FACT or DECISION itself, never that the user articulated it.
   Bad: "User asked about pgvector indexing options".
   Good: "User chose ivfflat over HNSW for pgvector indexing".
```
**Effect:** Cuts the "User encountered X" / "User mentioned Y" narrations that were eating precision in iter1. Final +5pp precision.

### Plus the judge clarifications (A4 — iter1 + iter2)
- Added `accurate_supplementary` class to precision judge — true given conversation but not in narrow `facts[]` array. Counts as accurate.
- Iter2: bias judge toward `accurate_supplementary` on borderline. "Hallucinated should be RARE. If unsure, choose accurate_supplementary." Closes the judge over-strictness identified in iter1.

## 3. The summarizer deprecation decision

**Before V1.5.5:** spec assumed both compaction + summarizer would be extended for V1.6 atomic-fact extraction.

**After V1.5.5 measurement:** summarizer's structured `focus:`/`decisions:`/etc. format intrinsically loses recall on conversations with sparse signal. Iter1 + iter2 numbers (no summarizer prompt change, only judge update):

| Metric | Baseline | Iter2 | V1.6 status |
|---|---|---|---|
| Summarizer recall | 0.63 | 0.67 | not on V1.6 critical path |
| Summarizer precision | 0.30 | 0.89 | judge improvement only |

**Decision:** summarizer keeps existing role (`summary_latest`/`timeline_summary` prose for agent retrieval). It is NOT extended for V1.6 atomic-fact extraction. Compaction Pass C alone carries that load. This is a SIMPLIFICATION of the V1.6 plan — one prompt to evolve instead of two.

V1.7 candidate: revisit summarizer prompt strengthening if the brain page shows visible quality gaps from `summary_latest` rows. Today's evidence says no.

## 5. Latency note (Together vs Groq)

Iter1 + iter2 ran on Together AI (Llama-3.3-70B-Instruct-Turbo) due to Groq daily token quota exhaustion during the baseline run. Latency on Together p50 is 1-15s; p95 is ~27s.

**Production V1.6 targets Groq direct.** Baseline run on Groq showed:
- Compaction Pass C latency p95: 1.4s (under 3.0s gate)
- Summarizer latency p95: 1.0s

The Together latency is V1.5.5 measurement infrastructure, not a production characteristic. Documented as expected provider-speed difference. V1.6's `extraction.provider` config will default to Groq with Together fallback.

Compaction CAPTURES both old + new + supersession context. V1.6's contradiction LLM judge (`src/agent/edge_resolution.zig` per spec §4.4) will then RESOLVE — close out the NeoVim/macOS rows via bi-temporal `valid_to`. Calibration confirmed: compaction's job is to extract; V1.6's job is to resolve. No prompt change needed at this layer.
