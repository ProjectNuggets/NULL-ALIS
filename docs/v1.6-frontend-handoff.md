### M1 — Importance scoring (lands V1.6 commit ~4)

Every `/brain/graph` node will gain an `importance_score: number` (0.0 to 1.0). Formula:

```
importance = 0.4 * normalized(incoming_edge_count)
           + 0.3 * recency_decay(last_referenced_at, half_life=30d)
           + 0.2 * confidence_score
           + 0.1 * normalized(retrieval_hit_count)
```

**FE expectation:** node radius proportional to importance. Smallest visible at 0.0 (~6px), largest at 1.0 (~24px). Use a power scale (`sqrt(importance)` or `Math.pow(importance, 0.7)`) so mid-tier nodes are visually distinct without dominating.

**Today's stub:** until backend delivers, derive a synthetic importance from `(session_edges_count + reference_edges_count)` for each node. Replace with `node.importance_score` when shipped.

## Soft requirements (Obsidian floor, not ceiling)

These are gaps I noticed in your current work that would make the difference from "graph viewer" to "second brain":

### S1 — Hover-preview must feel instant
Obsidian hovers are zero-latency because everything's local. Our M3 is an HTTP call (~50-150ms). **Pre-fetch on idle** for visible nodes — when the user pauses cursor over an area, fire M3 calls for top-N visible nodes in background, cache for ~30s. Hover then renders from cache, feels instant. Use `requestIdleCallback` + a TanStack Query prefetch pattern.

### S2 — Stable graph layout across sessions
Obsidian's graph remembers node positions between visits. We can do the same: persist `(node.id, x, y)` tuples in `localStorage` keyed by user. On load, prefer cached positions when nodes match; only run physics simulation for new nodes. The user's mental map of "where things are" stays stable.

### S3 — Search-highlight in graph
M2 returns matching keys. When search is active:
- Matching nodes glow / pulse
- Non-matching nodes dim to 30% opacity
- Edges between two matching nodes stay full opacity
- Edges from matching → non-matching nodes dim slightly (10% opacity)

This is Obsidian's exact UX for graph search.

### S4 — Backlinks visible on hover
M3's `linked_memories` array has `edge_direction: "incoming" | "outgoing"`. Hover preview should show "X memories link TO this" prominently. Obsidian's "linked mentions" surface is one of its highest-value UX patterns.

### S5 — Timeline slider on /brain/timeline tab (V1.7 setup)
The timeline tab currently shows a chronological list. V1.7 adds `/brain/diff?date=<unix>` endpoint that returns "what your brain looked like as of date X." For V1.6, scaffold:
- A horizontal timeline at the top of the timeline tab
- Drag a slider → calls `/brain/timeline?to=<unix_at_slider>` (existing param)
- Visualization shows what was true at that moment

This is the killer "see your brain evolve" feature. Set it up now even if backend evolution support comes in V1.7.
