## 1. The headline (revised)

**Today our memory has S-tier substrate and B-tier extraction.** Reads are at parity with the references (RRF + MMR + temporal decay + optional LLM rerank, already shipped in `RetrievalEngine`). Writes are not ‚Äî no per-turn or compaction-derived structured extraction, no typed edges in storage, no conflict detection, and the bi-temporal model is one field deep.

**V1.5.5 measures whether compaction + summarization is a solid foundation** for extraction. If yes, V1.6 extends those existing LLM calls to emit structured atomic facts (free, no extra inference cost). If no, V1.6 falls back to per-turn extraction with a small dedicated LLM call.

**V1.6 closes the write-side gap to Mem0+Graphiti parity** PLUS adds the four "real second brain" surfaces (importance scoring, /brain/search, /brain/memory/{id} drilldown, source attribution) the prior spec missed.

**V1.7 adds depth** ‚Äî per-edge fact embeddings, LLM-judge entity dedupe, communities, profile materialization, /profile endpoint.

After all three land, nullalis is at full parity with open-source SOTA AND has the surfaces a real second brain UX needs.

## 3. V1.5.5 ‚Äî Compaction & Summarization Characterization

**Why this exists:** the entire V1.6 strategy (D2 compaction-derived) depends on compaction reliably preserving the facts a human would extract. We don't know if it does. Three days of measurement saves three weeks of debugging.

### 3.1 Build the conversation corpus

Source: real conversations from the dev DB (user_id=1, anonymized) + synthetic-but-realistic supplements where coverage gaps exist.

Coverage targets ‚Äî at least 2 conversations of each type:

| Type | Why |
|---|---|
| Short single-topic | Baseline behavior, easiest case |
| Long multi-topic (10+ turns, 3+ topic shifts) | Stress on context-bounded compaction |
| Code-heavy (function definitions, error traces) | Code blocks are notoriously dropped by summarizers |
| Mixed-language (English + Arabic) | Nova's actual usage; tests UTF-8 + multilingual fidelity |
| Emotional / sentiment-rich | Tests whether feeling-content survives or gets sterilized |
| Error-recovery (user corrects, agent retries) | Tests whether corrections supersede prior wrong info |
| Agentic (tool calls, results, multi-step) | Tool-result pruning interaction with summarizer |
| Casual conversational | Most common pattern, need to know it works |

Storage: `tests/compaction_corpus/<type>_<id>.jsonl` ‚Äî one message per line, JSON-shaped per the agent's history format. Anonymized: real names ‚Üí tokens (Nova ‚Üí Alex, etc.), real emails ‚Üí placeholders.

### 3.2 Build ground-truth fact extraction

For each corpus entry, write **by hand** the facts a human reader would extract.

Format: `tests/compaction_corpus/<type>_<id>.ground_truth.json`:
```json
{
  "facts": [
    {
      "subject": "Alex",
      "predicate": "prefers",
      "object": "Zig",
      "confidence": "high",
      "source_message_idx": 3
    },
    ...
  ],
  "narrative_themes": ["onboarding", "tech preferences"],
  "must_preserve": ["specific named entities mentioned at least twice"],
  "may_drop": ["throwaway pleasantries", "ack-only turns"]
}
```

This is the bar. If compaction loses a `must_preserve` fact, that's a recall failure.

### 3.3 Run corpus through compaction

For each conversation:
1. Replay the messages through `autoCompactHistory` at varying triggers (50% / 70% / 90% context fill ‚Äî simulate by tightening `token_limit`)
2. Capture Pass C output (the LLM-generated `compaction_summary` content)
3. Capture summarizer outputs (`summary_latest`, `timeline_summary`) at their natural cadence
4. Persist all outputs in `tests/compaction_corpus/<type>_<id>.run_<trigger>.json`

### 3.4 Score

For each (conversation, trigger) pair:

**Recall:** for each ground-truth fact, can it be reconstructed from the compaction output?
- Manual scoring or LLM-assisted (a separate, larger model judging the summary against ground truth)
- Score 0-1 per fact: 1 = explicit, 0.5 = implicit/derivable, 0 = missing

**Precision:** for each statement in the compaction output, is it accurate vs ground truth?
- Score 0-1 per statement: 1 = accurate, 0.5 = vague but correct, 0 = hallucinated/wrong

**Coverage by fact type:** what categories of facts reliably survive vs disappear?
- Named entities, temporal references, preferences, decisions, errors, code references, etc.

**Latency:** wall-clock time per Pass C invocation under realistic conditions.

**Failure modes:** for each conversation, inject Pass C LLM call failure (mocked) ‚Äî does the system recover gracefully? Are there silent loss paths?

### 3.5 Acceptance gates (V1.6 starts only if all pass)

- **Recall ‚â• 85%** on `must_preserve` facts across all conversation types
- **Precision ‚â• 95%** (hallucination rate ‚â§ 5%) ‚Äî extraction quality requires low BS
- **No silent loss** ‚Äî every failure mode either recovers or emits a structured `metric=compaction.failed` log line
- **Latency p95 ‚â§ 3 seconds** for Pass C under realistic load (we'll be running this at every compaction trigger; can't be slow)
- **Coverage parity across conversation types** ‚Äî no type can drop below 70% recall (otherwise compaction is biased and extraction inherits the bias)

### 3.6 If gates fail ‚Äî the fallback plan

Two paths depending on what V1.5.5 finds:

**Path A ‚Äî Compaction is mostly solid but has specific weaknesses (likely outcome):**
- Identify the specific failure modes (e.g., "code blocks lose 40% recall")
- Strengthen the compaction prompt with targeted instructions
- Re-run V1.5.5 ‚Üí re-score ‚Üí iterate until gates pass
- Then proceed to V1.6 with confidence the foundation works

**Path B ‚Äî Compaction is fundamentally weak (the LLM call quality is too low):**
- Revert to per-turn extraction (the original D2 default)
- Use Groq + Llama-3.3-70B for the extraction call (~$0.0006/turn, ~200ms)
- Compaction stays as-is for prose summary; doesn't carry extraction load
- V1.6 acceptance gates become the per-turn extraction quality bar (same recall/precision targets, different LLM call)

The choice is data-driven, not preference-driven.

### 4.3 M1 ‚Äî Importance scoring

**New file:** `src/memory/importance.zig`. Public surface:

```zig
pub const ImportanceInputs = struct {
    incoming_edge_count: usize,
    last_referenced_at: i64,
    confidence_score: f64,
    retrieval_hit_count: usize,
};

pub fn computeImportance(inputs: ImportanceInputs) f64 {
    // Formula:
    //   0.4 * normalized(incoming_edge_count)
    // + 0.3 * recency_decay(last_referenced_at)
    // + 0.2 * confidence_score
    // + 0.1 * normalized(retrieval_hit_count)
    //
    // Normalization: sigmoid(x / scale_const) where scale_const tunes the
    // saturation point. Recency decay: exp(-(now - ts) / half_life_seconds)
    // with half_life = 30 days.
}
```

**Wire into:**
- `state_mgr.upsertMemory` and `upsertMemoryWithMetadata` ‚Äî call `computeImportance` before write
- New trigger or scheduled job: refresh `importance_score` when `incoming_edge_count` changes (a memory gets new linked_memory_ids)
- `/brain/graph` handler: include `importance_score` on every node payload

**Acceptance:** PG smoke test seeds a graph with varying edge density, asserts importance ranking matches expected order.

**Estimated:** 1 day. Atomic commit.

### 4.4 Compaction-derived extraction (D2 primary path)

**Conditional on V1.5.5 Path A.** If V1.5.5 chose Path B (per-turn fallback), this section is replaced with the per-turn extraction implementation from spec v1 ¬ß3.3.

**Approach:** extend the existing compaction + summarizer prompts to emit structured atomic facts as JSON alongside the existing prose summary.

**Modified compaction Pass C prompt** ‚Äî output schema becomes:
```json
{
  "summary": "<existing prose summary, unchanged behavior>",
  "extracted_memories": [
    {
      "text": "<atomic fact, 15-80 words, self-contained>",
      "subject": "<entity name>",
      "predicate": "<RELATION_TYPE_SCREAMING_SNAKE>",
      "object_key": "<other entity name OR another memory.key>",
      "confidence": <0.0-1.0>,
      "attributed_to": "user" | "assistant",
      "source_message_idx": <int>,
      "valid_at": <unix epoch or null>,
      "linked_memory_ids": ["<existing memory.key>", ...]
    }
  ]
}
```

The prose `summary` field continues feeding `compaction_summary/{session}/{ts}` continuity artifact (zero behavior change). The new `extracted_memories[]` array goes through the V1.6 persist path.

**Persistence:** new function `src/agent/extraction_persist.zig::persistExtractedMemories(allocator, state_mgr, user_id, session_id, extracted)`:
1. MD5 content_hash pre-filter (gap #13)
2. Cosine dedup vs recent rows (D4 mitigation, threshold 0.92)
3. Entity coreference (M1 - use memory_entities cosine 0.95 threshold)
4. Bi-temporal close-out check (Graphiti contradiction judge, ¬ß4.5)
5. Write surviving facts via `state_mgr.upsertMemoryWithMetadata` with full V1.6 metadata
6. Update `memory_entities.linked_memory_ids` arrays
7. Emit `memory_events` row with `event_type='extraction'`

**Same for summarizer prompt** ‚Äî `summary_latest` and `timeline_summary` outputs gain the same `extracted_memories[]` JSON sidebar.
