# V1.8-4 Decision — drop the deterministic cheap pass

**Date:** 2026-05-05 (mid-build, post V1.8-1/2/3/5)
**Author:** Claude (with Nova's autonomy directive: "go on as you think you are the best father to your son")
**Decision:** Subtract V1.8-4 from the V1.8 sprint. No regex extractor. No every-turn LLM extraction either.

---

## What V1.8-4 was

Per `docs/v1.8-build-plans.md` §V1.8-4 (sha `a4152f2`):

- New file `src/agent/cheap_extraction.zig`, ~205 LOC
- 5 declarative regex shapes ("I prefer X", "My name is X", "I work at X", "deadline is DATE", "X is my Y")
- Plus entity regex + date regex
- Runs every captured user message
- Writes triples with `attribution=deterministic`, `confidence=0.6`
- Routes through existing `extraction_persist.persistExtracted`

Stated intent: "deterministic floor — catches identity/preference shapes when LLM extraction silently misses."

## Why it was queued (state at Phase 6 plan)

When V1.8-4 was specified, none of V1.8-1/2/3/5 had shipped. The world was:
- Judge `judge_provider=null` at gateway.zig:1562 → contradictions never applied (G-B)
- LLM compaction prompt didn't always emit `===EXTRACTED===` tail → silent extraction misses (G-A part 1)
- Pass C unreachable on big-context models → no extraction triggered for long-content low-message-count histories (G-A part 2)
- Vector bypass in persistExtracted (G-C)
- Forget didn't cascade to edges (G-NEW)

Given that landscape, "regex floor on every turn" had defensive value: it was the only thing that would catch a fact when literally everything else was broken.

## Why the world changed under the plan

V1.8-1 / V1.8-2 / V1.8-3 / V1.8-5 closed the actual bugs:

| Pre-V1.8 failure | Fix | Now |
|---|---|---|
| Judge null → contradictions logged then dropped | V1.8-1 wired `judge_provider` + reordered `applyContradictions` before `is_duplicate` short-circuit | Contradictions actually apply. preference_changes F1 0.571 → 1.000. |
| Compaction prompt sometimes omits JSON tail | V1.8-5 prompt: "ALWAYS append the EXTRACTED block. If no triples, emit `===EXTRACTED===\n[]`" | Tail always present. Parser never starves. |
| Pass C unreachable on big-context models | V1.8-5 `keep_recent` cap forces compaction even on long-content low-message-count histories | Pass C fires at 90% token pressure regardless of message count. |
| Extracted memories not vectorized | V1.8-2 threaded `mem_rt` through `persistExtracted` | 79% recent embedding coverage (was 53%). |
| Forget left orphan edges | V1.8-3 bi-temporal cascade in transaction | edges_active 1→0, edges_closed 0→1 verified. |

The "LLM silently misses identity/preference shapes" failure mode is gone. When the LLM extractor runs, it now (a) actually emits structured output, (b) gets its judge applied, (c) writes vectorized memories, and (d) handles dedup/contradict.

## Triggers for extraction post-V1.8-1/2/3/5

Three independent paths now call `persistExtracted`:

1. **Compaction Pass C** at 90% token pressure (per Nova: "every turn is too much, we compact at 90%"). Source: `src/agent/compaction.zig::autoCompactHistory`. V1.8-5 prompt + `keep_recent` cap guarantee emission and reachability.
2. **Session-end summarizer** at `src/agent/commands.zig:1482` (`persistSessionCheckpoint` → `persistSessionSemanticSummary`). Same `summarizer.zig` prompt scaffold, same `persistExtracted` write path. Fires on `clearSessionState`.
3. **LLM voluntary tool calls** to `memory_store` via `src/tools/memory_store.zig:171`. Same `persistExtracted`, with V1.8-1's judge wired.

A fact mentioned in a conversation will get extracted at:
- T+0 if the LLM proactively calls `memory_store` (rare, but possible)
- T+(first compaction) at 90% token pressure
- T+(session end) at the latest

The phantom failure mode V1.8-4 was protecting against: "user mentions a fact, conversation never reaches 90%, never ends, fact never extracted." That window:
- Closes on session-end (every session ends eventually — `clearSessionState` fires on /clear or new session)
- Doesn't appear in eval corpora that pass at F1 ≥ 0.8 post-V1.8-1
- Costs 205 LOC of permanent regex maintenance to "guarantee" what already works

## Why regex specifically is wrong

Setting aside whether a floor is needed, regex is the wrong implementation:

- **Brittle on natural language.** "I prefer Helix" matches; "i actually prefer helix tbh" doesn't. "I prefer not to answer" false-matches. Every shape needs N variant patterns; maintenance burden grows with vocabulary, not value.
- **Two sources of truth → reconciliation.** Regex writes `confidence=0.6`, LLM writes `confidence=0.85`. The judge has to arbitrate. Add a new shape and you've created a drift surface.
- **Doesn't generalize.** English-only, present-tense, first-person declaratives. The next-gen agent serves any task in any language.
- **Doesn't fix the failure modes worth caring about.** If the extraction LLM call errored, network is down, or schema was malformed → regex on the user message doesn't help. The system is already in failure mode.

The 1990s-NLP move bolted onto a 2026 LLM agent. `[[feedback_next_generation]]` says no.

## Alternatives considered

1. **Cheap = small fast LLM on every turn.** Same `persistExtracted` path, route to a cheap/fast model (DeepSeek-Lite class) on every turn. ~30 LOC of routing.
   - **Rejected by Nova:** "every turn is too much, we compact at 90% token pressure."
   - On reflection: every-turn LLM extraction also adds latency + cost without measurable F1 gain over the existing 3-trigger setup. Belt-and-suspenders for a problem the eval didn't show.

2. **Streaming extraction in parallel with the turn.** Async pipeline that resolves by the time the agent finishes its main response.
   - Bigger refactor. Defer to V1.9 if eval surfaces a real timing gap.

3. **Drop entirely.** Trust the three triggers (compaction / session-end / voluntary tool).
   - **Selected.** Re-run the eval suite as truth-check. If F1 holds, the floor was unnecessary.

## Conditions to revisit

This decision should be re-opened if:

- A future eval corpus targeting "short-conversation identity facts" shows F1 < 0.75
- Session-end isn't reliably hit in production telemetry (e.g., processes crash without graceful shutdown)
- A real user reports "I told it my name and it forgot the same session"

In those cases the right fix is **option 1 (cheap small LLM on a cadence)**, not regex. The cadence to evaluate would be: every N turns where N=3-5, or only when message count > 4 and no extraction has run yet.

## Locked binding

Per [[feedback_next_generation]]: every commit asks *"what would the next generation build, knowing what we know in 2026?"*

The next generation does not write 205 LOC of English-regex extractors when it has frontier LLMs and a working judge. It writes the smallest amount of routing code needed to point existing extraction at existing triggers, then measures.

V1.8-4 fails that test. Subtraction is the next-gen move.
