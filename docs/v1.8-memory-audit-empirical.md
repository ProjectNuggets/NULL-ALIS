## Executive summary

The audit fundamentally rewrites my prior diagnosis. The 0-edges DB-wide finding from V1.7 was NOT due to the extraction pipeline being broken — **the pipeline works perfectly when the agent calls `memory_store` with triple args**. The actual issues are 4 distinct gaps, not 1:

1. **G-A — Prose extraction never fires.** Lifecycle summarizer creates `summary_latest/`, `timeline_summary/`, `session_checkpoint_*` rows but emits ZERO structured triples in their content. Compaction Pass C did NOT fire during this audit (token pressure stayed under 10.8K). Prose-to-edges extraction is dormant in practice.

2. **G-B — Contradiction judge is OFF.** When the agent stores "I prefer NeoVim" after previously storing "I prefer Helix," the old edge stays `is_latest=true`. Memory accumulates STALE TRUTH. Cross-session recall in T11 returned "Helix" (T2 fact) when "NeoVim" (T4 fact) is the latest. **User-impacting bug.**

3. **G-C — `memory_recall` semantic miss.** Cross-session prompt "What's my preferred editor?" returned "No memories found" despite an extracted fact `extracted_aa601076031a93d1` containing "Nova prefers Helix over NeoVim..." Either embeddings aren't created on store, or retrieval doesn't surface the right candidate. Agent fell back to reading `MEMORY.md` (markdown mirror) — clever but not the design intent.

4. **G-D — Coreference produces hash keys, not entity_<name>.** Every typed edge target is a 32-char hex hash (`26c12d732c46b337059fbf0dab89f836`), never `entity_helix` / `entity_hrs` / `entity_alex`. V1.6 cmt8 entity coref didn't fire (or fired and found no embedding match — first-mention every time). Graph is hash-soup, not human-navigable.

The good news: **architecture is right, schema is right, the typed-edge surface populates correctly when triggered.** All 4 gaps are tractable.

## Phase 3 design implications (preview — formal doc next)

Based on audit + research, the V1.8 sprint shape sharpens:

1. **G-A fix (HIGHEST priority)**: make prose-to-edges extraction actually fire. Two paths:
   - **Lifecycle summarizer enhancement**: tighten the JSON-tail prompt + lower its bar for emitting triples
   - **NEW deterministic cheap pass**: extract entities/dates/preferences from EVERY message via regex/NER (Mem0g pattern). Floor of edge density.

2. **G-B fix**: turn judge ON in production. Either set `judge_provider`/`judge_model` config defaults, or pass through the active provider as fallback (which the code at `compaction.zig:494` already does — verify if disabled by some other gate).

3. **G-C diagnose**: verify pgvector index + embeddings are created on `memory_store`. May need to add a probe.

4. **G-D fix**: improve coref by ensuring entities get added to `memory_entities` table on first mention with their string content embedded — so subsequent mentions can match at 0.95.

5. **Adopt research S-tier**: ASSUME INTERRUPTION framing (one-line prompt change), sem/epi/proc taxonomy in prompt, sleep-time worker generalization.

## Phase 1b-1 — Stress test (force Pass C)

Sent T13_stress through T18_stress (6 long prompts ~3K chars each about V-infinity pillars / project structure / V1.7-ship train). Goal: push token pressure past 90% to fire compaction Pass C.

**Result**: Memories grew steadily (47→49→51→53→55→57) but ALL via autosaves (`autosave_user_*`, `autosave_assistant_*`). **Zero new typed edges.** Pass C DID NOT FIRE.

**Why** — verified via `model_capabilities.zig:269 resolveContextTokens`:

```zig
pub fn resolveContextTokens(override: ?u64, model_ref: []const u8) u64 {
    if (override) |v| return v;
    if (lookupCapabilities(model_ref)) |c| return c.context_window;
    return DEFAULT_CONTEXT_WINDOW;
}
```

And `agent/root.zig:684`:
```zig
const token_limit_override = if (cfg.agent.token_limit_explicit) cfg.agent.token_limit else null;
```

Without `cfg.agent.token_limit_explicit=true` in config, override is null. Falls through to `lookupCapabilities("deepseek-ai/DeepSeek-V4-Pro")` → 512K context window. **Pass C trigger = 460,800 tokens.**

`/context` slash command reports `token_estimate=33771`. We're at **7.3% pressure** on a 38-message conversation. To trigger Pass C compaction, a user would need ~1800 turns of similar length.

**G-A is now fully confirmed and quantified**: Pass C compaction is architecturally dead for any modern long-context model. The 70/90% threshold gates were designed for 4K-8K-era models. With 256K-1M-context models in production, Pass C never fires in normal use.

**This is the V1.8 anchor finding.** The lifecycle summarizer is the ONLY active prose-extraction pipeline today, and even it doesn't emit typed edges from prose.
